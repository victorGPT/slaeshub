# 客户详情页 — 内容设计文档

## 1. 这个页面回答什么问题

BD 点进一个客户详情，脑子里只有四个问题：

1. **这个人是谁，多重要？**
2. **他的钱最近怎么了？**
3. **为什么会这样，我该做什么？**
4. **之前跟他聊过什么，现在什么状态？**

页面的所有内容都围绕这四个问题组织，不回答这四个问题的信息不放。

---

## 2. 区块设计

### 区块 A：客户身份（这个人是谁）

**回答的问题：** 我现在看的是谁，他在我的客户盘里算什么级别。

| 内容项 | 说明 | 必要性 |
|--------|------|--------|
| 客户名 | 中文名 | 必须 |
| UID | 平台唯一 ID | 必须 |
| 头像 | 首字母头像或上传照片 | 有则展示 |
| 状态标签 | 流失 / 增长 / 稳定 | 必须 |
| 系统标签 | 如"流失风险"、"资沉增长中"，系统自动打，BD 不可编辑 | 必须 |
| BD 标签 | BD 自己打的标签，如"偏好短线"、"英文沟通"，可增删 | 必须 |
| BD 归属 | 这个客户归哪个 BD 管，接入时间 | MVP 可缺省 |

**设计要点：**
- 状态标签是最重要的视觉锚点（红/绿/灰），一眼能看出这个客户现在是什么状态
- BD 标签是销售自己的"备忘贴纸"，不要限制数量，但 UI 上折叠到第二行

---

### 区块 B：资沉核心指标（他的钱最近怎么了）

**回答的问题：** 盘子多大，是在涨还是在掉，掉/涨了多少。

| 内容项 | 说明 | 展示方式 |
|--------|------|---------|
| 当前资沉 | 8 周均值（USDT 折算） | **最大字号**，页面视觉中心 |
| 本周均值 | 最近一周日均资沉 | 与 8 周均值对比 |
| 趋势 | AdjustedTrendRate（线性回归斜率，已去市场） | 百分比/周，正绿负红 |
| 趋势稳定度 | R²（线性回归可信度） | R²≥0.7 "趋势明确"，<0.7 "波动较大" |
| 绝对变化 | w8 - w1 的绝对值 | 万/亿格式，带 ↑↓ 箭头 |

**设计要点：**
- 当前资沉是**整个页面最大的数字**，BD 打开页面第一眼看到的就是"这个客户有多少钱"
- 趋势用 8 周线性回归，不是简单两点对比。能区分"持续流失"和"单周波动"
- 趋势稳定度告诉 BD"这个趋势靠不靠谱"：实心箭头 = 明确，虚线箭头 = 不确定
- **不需要额外显示市场基准**（趋势已去掉市场噪音）

**展示示例：**
```
↘ -2.5%/周 · 趋势明确      ← BD 必须关注
↗ +1.8%/周 · 波动较大      ← BD 先观察
→  0.1%/周 · 稳定          ← 常规维护
```

**这里不放的东西：**
- ~~N90（净贡献）~~：新框架下资沉就是价值，不需要另算贡献
- ~~E90（变现效率）~~：同上
- ~~交易量~~：是过程量，不是结果量
- ~~市场整体趋势~~：已在回归中去掉，不需要展示

---

### 区块 C：原因 + 动作（为什么会这样，我该做什么）

**回答的问题：** 系统认为这个客户为什么值得关注，以及建议我现在做什么。

| 内容项 | 说明 | 展示方式 |
|--------|------|---------|
| 原因标签 | 为什么进入当前视图（如"理财到期/赎回"、"近期持续入金"） | 文本，卡片内 |
| 建议动作 | 当前最值得做的一件事（如"安排销售回访"） | 金色文字，突出 |

**设计要点：**
- 原因和动作放在同一张卡片里，用分隔线隔开
- **只给一个动作**，不给选择题（MVP 原则：减少决策负担）
- 动作来自白名单：销售回访 / 续存提醒+权益说明 / 资沉补贴券 / U卡积分活动 / 场景绑定说明

---

### 区块 D：业务参与情况（这个客户做了哪些业务）

**回答的问题：** 客户在平台上做了几条业务线，哪些做了哪些没做。

| 业务线 | 说明 |
|--------|------|
| 合约 | 双仓合约交易 |
| 杠杆 | 超级杠杆交易 |
| 理财 | 理财产品持仓 |
| 金融卡 | U卡/金融卡使用 |
| 云算力 | 云算力服务 |
| Mini合约 | Mini合约交易 |

**设计要点：**
- 6 条业务线全列出来，参与的用金色条 + "已参与"，未参与的用灰色条 + "未参与"
- 右上角显示"X/6 条业务"作为概览
- **这个区块的核心价值是帮 BD 发现交叉销售机会**：如果客户有高资沉但只做了 2 条业务，说明还有空间

**MVP 阶段不做的：**
- ~~每条业务线的具体金额/占比~~：数据维度太多，MVP 先不放
- ~~业务线贡献条形图~~：同上

---

### 区块 E：最近事件（有什么需要我处理的）

**回答的问题：** 系统最近触发了哪些跟这个客户相关的事件。

| 内容项 | 说明 |
|--------|------|
| 事件类型 | 资沉流失预警（红）/ 理财到期提醒（金）/ 资沉增长通知（绿）/ 定期回访（灰） |
| 事件摘要 | 一句话描述 |
| 触发时间 | 相对时间（如"2天前"） |

**设计要点：**
- 最多显示 2-3 条最近的事件
- 每条事件左侧有颜色圆点，一眼区分紧急程度
- 点击可跳转到事件线程页
- 事件内容与客户状态相关：流失客户看到出金预警，增长客户看到入金通知

**MVP 阶段限制：**
- 当前事件数据是静态模板（按客户状态生成），不是真实触发
- 后续接入真实事件系统后替换

---

### 区块 F：BD 备注（我之前记了什么）

**回答的问题：** 我自己对这个客户的自由备注。

| 内容项 | 说明 |
|--------|------|
| 自由文本 | BD 自己写的备注，没有格式限制 |

**设计要点：**
- 一个 textarea，输入即保存（localStorage）
- 不需要格式化、不需要 @ 提及、不需要模板
- 这是 BD 最私人的工具，保持极简

---

### 区块 G：跟进记录（之前跟他聊过什么）

**回答的问题：** 这个客户的历史互动记录。

| 内容项 | 说明 |
|--------|------|
| 时间 | 跟进发生的时间 |
| 执行人 | 哪个 BD 做的 / 系统自动触发 |
| 内容 | 做了什么，客户什么反应 |

**设计要点：**
- 时间线样式（左侧竖线 + 节点）
- 最新的在最上面
- 区分"人工记录"和"系统记录"（如"系统：触发资沉流失预警"）
- 顶部有"+ 添加记录"按钮

**MVP 阶段限制：**
- 当前是静态模板数据
- 后续接入真实 CRM 后替换

---

## 3. 区块排列优先级

从上到下，按 BD 的信息消费顺序排列：

```
A. 客户身份         → 先确认"我在看谁"
B. 资沉核心指标     → 再看"他的钱怎么了"
C. 原因 + 动作      → 然后看"我该做什么"
D. 业务参与情况     → 补充上下文"他做了哪些业务"
E. 最近事件         → 有没有紧急的事要处理
F. BD 备注          → 我之前记了什么
G. 跟进记录         → 历史互动
```

**前三块（A/B/C）是核心**，BD 打开页面 3 秒内能看完。
**后四块（D/E/F/G）是辅助**，BD 有需要时往下滚。

---

## 4. 明确不放的内容

| 不放的内容 | 原因 |
|-----------|------|
| N90（集团净贡献） | 新框架下资沉就是价值，不需要另算 |
| E90（资金变现效率） | 同上 |
| 交易量/交易额 | 过程量，不是结果量 |
| 客户画像/AI 标签 | MVP 不做复杂画像 |
| 多业务线贡献条形图 | 数据维度过多，Phase 2 再加 |
| 市场整体趋势 | 趋势已经去掉市场噪音，不需要单独展示 |
| 竞品对比 | 无数据来源 |
| 客户净值分层 | MVP 不做 |
| 推荐多个动作 | 只给一个，减少决策负担 |

---

## 5. 后续迭代方向

| 优先级 | 迭代项 | 条件 |
|--------|--------|------|
| P1 | 跟进记录接入真实 CRM 数据 | CRM 系统打通 |
| P1 | 事件卡片接入真实触发系统 | 事件引擎上线 |
| P2 | 业务线具体金额和占比 | 业务级数据可用 |
| P2 | BD 归属 + 接入时间 | 客户分配系统确认 |
| P3 | 资沉趋势折线图（30 天） | 有每日快照数据 |
| P3 | 动作留痕（点击"执行动作"后自动记录） | 动作系统上线 |
