# Sales Hub 页面设计说明 — 从第一性原理出发

## 0. 第一性原理

**资金沉淀就是价值。**

不是"待转化的潜力"，不是"手续费的前置条件"，是价值本身。

原因：资金沉淀带来的收益（平台用资金做市、借贷、资金池等）远大于客户交易产生的手续费。一个客户把 1000 万放在平台上什么都不做，他给平台创造的价值可能比一个每天交易 100 万但不留钱的客户更大。

由此推导出销售工作的本质：

```
销售的工作 = 让更多的钱留在平台上，更久。
```

拆开就是三件事：

1. **守住大盘子** — 已有的大额沉淀不能丢
2. **止住流失** — 正在流出的钱要拦回来
3. **放大增长** — 正在流入的钱要顺势做大

所有页面设计都从这三件事出发。任何不服务于这三件事的元素，不放。

## 1. 首页（home.html）

### 这个页面回答什么

> BD 每天早上打开 App 的第一个问题：**今天先看谁？**

### 设计决策与理由

#### 三个 Tab，不是两个榜

| 设计决策 | 理由 |
|---------|------|
| 用"资沉总量 / 资沉流失 / 资沉增长"三个视角，不用"高价值 / 高潜力"两个分类 | 分类回答的是"这个人是什么类型"，但销售不关心类型，销售关心的是"今天先动谁"。三个 Tab 直接对应三种动作意图：看盘、止损、放大。 |
| 默认 Tab 是"资沉总量"，不是"流失"或"增长" | 因为 BD 的第一需求是**建立盘感**——先知道自己管着多大的盘子，哪些人最重要。有了盘感之后，才看变化。如果默认就是流失列表，BD 会焦虑但没有上下文。 |
| 流失 Tab 有红色数字 badge，增长 Tab 有绿色 badge | 让 BD 不切 Tab 就能知道"今天有几个人在掉、几个人在涨"。这是**被动感知**——信息主动找人，而不是人去找信息。 |

#### 排序：为什么资沉总量 Tab 只按钱多少排，不掺别的

```
TotalRank = mean(w1...w8)
```

**理由：** 这一层的目标只有一个——让 BD 知道谁最重要。如果掺杂活跃度、动作匹配度、趋势等因子，排序就变成黑箱，BD 不知道"为什么这个人排第三"。

资沉总量是**最透明、最不需要解释的排序标准**。BD 看到列表，脑子里只有一个认知："排越前面的钱越多。" 这是信任的基础。

#### 排序：为什么流失/增长榜用 log 压缩

```
ChurnScore = log(1 + F_mean / T_F) × max(0, -AdjustedTrendRate - δ) × Conf
```

**理由：** 如果直接用 `F_mean × |TrendRate|`，5000 万的客户微跌 1% 就会排在 500 万客户暴跌 10% 前面。这不符合销售直觉——500 万暴跌可能更紧急。

log 压缩让盘子大小的影响变成"对数级"而不是"线性级"。大客户仍然排前面（log 单调递增），但不会因为绝对值大就永远霸榜。

**这怎么服务第一性原理：** log 压缩确保"最危险的流失"排最前面，而不是"最大客户的微波动"排最前面。销售的时间花在真正需要救的客户上。

#### 卡片：只展示四项信息

```
① 资沉总量  ② 状态 + 趋势 + 稳定度  ③ 原因标签  ④ 建议动作
```

**理由：** 首页不是分析页，是**分诊台**。BD 需要的是"3 秒判断要不要点进去看"。四项信息恰好回答：

1. **这人多重要**（资沉总量）
2. **他怎么了**（状态 + 趋势）
3. **为什么**（原因标签）
4. **我做什么**（建议动作）

多一项就是噪音，少一项就缺决策依据。

#### 趋势展示：%/周 + "趋势明确"/"波动较大"

**理由：** 百分比告诉 BD 方向和速度，稳定度告诉 BD "这个方向靠不靠谱"。

- `↘ -3.2%/周 · 趋势明确` → BD 立刻行动
- `↗ +1.8%/周 · 波动较大` → BD 先观察

如果只有百分比没有稳定度，BD 会对每一个负数都紧张。加了稳定度，BD 能区分"真的在跌"和"只是上下晃"。

**这怎么服务第一性原理：** 让销售的注意力集中在**确定性高的资沉变化**上，而不是被噪音分散。

## 2. 客户详情页（client-detail.html）

### 这个页面回答什么

> BD 点进一个客户后的四个问题：
> 1. 这是谁，多重要？
> 2. 他的钱最近怎么了？
> 3. 为什么会这样，我该做什么？
> 4. 之前跟他聊过什么？

### 设计决策与理由

#### 区块 A：客户身份（头像 + 名字 + 标签）

| 元素 | 为什么放 |
|------|---------|
| 头像 + 名字 | 确认"我在看谁"。人类处理面孔和名字比处理 UID 快。 |
| 状态 badge（流失/增长/稳定） | 首页已经看过一次，详情页再确认一次。状态是整个页面的**色调锚点**——流失客户页面偏红，增长客户偏绿。 |
| 系统标签（🔒 重点客户 / 流失风险） | 系统自动打的标签，BD 不可编辑。代表"平台认为这个客户是什么"。 |
| BD 标签（可编辑） | BD 自己打的标签，如"偏好短线"、"英文沟通"。代表"我对这个客户的理解"。这是 BD 的**私人记忆外挂**。 |

**这怎么服务第一性原理：** 标签系统让 BD 能快速回忆"我跟这个客户的关系上下文"，不需要重新翻聊天记录。关系上下文越清晰，BD 的沟通越精准，客户留存越好。

#### 区块 B：资沉核心指标（页面视觉中心）

```
当前资沉（最大字号）
本周均值    近期趋势    趋势稳定度
```

**当前资沉是整个页面最大的数字。**

**理由：** 第一性原理说"资沉就是价值"。那页面上最大的数字就应该是资沉。不是净贡献，不是交易量，不是效率——是资沉。

| 指标 | 为什么放 | 为什么这样展示 |
|------|---------|--------------|
| 当前资沉 (8 周均) | 回答"这个客户有多重要" | 最大字号，28px，是页面的视觉锚点 |
| 本周均值 | 回答"最近一周跟 8 周均值有没有偏离" | 与当前资沉同行对比，BD 一眼看出差异 |
| 近期趋势 (%/周) | 回答"在涨还是在跌，多快" | 用 8 周线性回归算出，不是简单两点对比。能区分"持续流失"和"单周波动" |
| 趋势稳定度 | 回答"这个趋势靠不靠谱" | R² ≥ 0.7 = "趋势明确"，< 0.7 = "波动较大" |

**为什么不放这些：**

| 不放的指标 | 理由 |
|-----------|------|
| N90（净贡献） | 新框架下资沉本身就是价值，净贡献是旧世界的产物 |
| E90（变现效率） | 同上。效率低不再是负面信号——客户不交易但钱放着，平台照样赚 |
| 交易量 | 过程量，不是结果量。交易多不代表资沉多 |
| 市场整体趋势 | 趋势数字已经去掉了市场噪音，不需要单独展示。展示了反而让 BD 困惑"这个数字是干嘛的" |

**这怎么服务第一性原理：** 整个指标区只围绕一个量——资金沉淀。它的绝对值、短期变化、变化方向、方向可信度。四个维度，一个量。BD 看完这个区块，对"这个客户的钱怎么了"有完整认知。

#### 区块 C：原因 + 建议动作

**合并在一张卡片里，不分两个区块。**

**理由：** 原因和动作是因果关系——"因为理财到期了，所以建议你去回访"。分开展示会断裂这个因果链。放在一起，BD 的认知流是连续的："哦，理财到期了 → 那我去提醒续存。"

**只给一个动作，不给多个。**

**理由：** 给三个选择 = 给零个选择。销售的时间是最稀缺的资源，他需要的是"告诉我做什么"，不是"给我一道选择题"。

**这怎么服务第一性原理：** 原因解释"资沉为什么在变"，动作回答"怎么让资沉留住/变多"。每个动作都指向同一个目标——保住或增加资金沉淀。

#### 区块 D：业务参与情况（6 条业务线）

```
合约 ████████████ 已参与
杠杆 ████████████ 已参与
理财 ░░░░░░░░░░░░ 未参与
金融卡 ░░░░░░░░░░░░ 未参与
云算力 ████████████ 已参与
Mini合约 ░░░░░░░░░░░░ 未参与
```

**理由：** 业务线参与度不直接影响排序（MVP 不做），但它回答一个重要的**上下文问题**：这个客户的钱散布在哪些业务里？

一个只做合约的客户和一个做了 5 条业务线的客户，即使资沉相同，流失风险也不同——业务线越多，迁移成本越高，关系越粘。

**这怎么服务第一性原理：** 帮 BD 发现**关系加深的机会**。如果客户资沉很高但只做了 2 条业务，说明还有空间通过交叉销售增加粘性，从而**保护资沉**。

#### 区块 E：最近事件

**只展示 2-3 条最近的事件，不展示全部。**

**理由：** 详情页不是事件中心。事件的作用是给 BD 一个快速的"有没有紧急的事"信号。如果有，点进去看详情；如果没有，继续往下。

事件颜色跟客户状态联动：
- 流失客户 → 红色事件（出金预警、到期提醒）
- 增长客户 → 绿色事件（入金通知、新业务开通）
- 稳定客户 → 灰色事件（常规回访）

**这怎么服务第一性原理：** 事件的本质是"资沉发生了变化的信号"。出金预警 = 资沉在减少，入金通知 = 资沉在增加，到期提醒 = 资沉即将减少。每个事件都是对资沉变化的预警或确认。

#### 区块 F & G：BD 备注 + 跟进记录

**备注是自由文本，跟进记录是时间线。**

**理由：** 备注回答"我对这个客户的私人理解"，跟进记录回答"之前发生了什么"。两者合在一起构成**关系记忆**。

**为什么备注用自由文本，不用结构化表单：** 因为 BD 的思维是非结构化的。"这个人不喜欢打电话，偏好微信"、"上次聊天提到他考虑移民"——这些信息无法被任何表单捕捉。自由文本是成本最低、信息密度最高的记忆方式。

**这怎么服务第一性原理：** 关系记忆让 BD 每次联系客户都不像"第一次见面"。关系质量越高，客户对平台的信任越深，资沉留存越稳。

## 3. 全部客户列表（clients.html）

### 这个页面回答什么

> BD 想找一个特定客户，或者想按条件筛选客户。

### 设计决策与理由

#### 默认按资沉降序，不按字母或最近活跃

**理由：** 与第一性原理一致——最重要的客户（资沉最大）排最前面。BD 下拉列表时，脑子里建立的就是"从最重要到最不重要"的序列。

#### 搜索：支持按名字和 UID

**理由：** BD 有两种找人方式——记得名字的直接搜名字，从系统通知跳过来的用 UID。覆盖这两种场景就够了。

#### 状态筛选 pill：全部 / 流失 / 增长 / 稳定

**理由：** BD 有时候想"只看我的流失客户"或"只看增长中的客户"。状态筛选是最轻量的过滤方式——不需要复杂的筛选器，四个按钮点一下就行。

**为什么没有更多筛选维度（如按业务线、按资沉区间）：** MVP 不做。100 个重点客户的列表，靠搜索 + 状态筛选已经够用。加更多筛选器只会增加认知负担。

#### 卡片格式与首页完全一致

**理由：** 一致性。BD 在首页和列表页看到的卡片是同一个样子，不需要重新学习"这个数字是什么意思"。

**这怎么服务第一性原理：** 列表页是首页的延伸。首页是"系统帮你挑了最该看的人"，列表页是"你自己去找你想看的人"。两个入口，同一个数据，同一个排序逻辑，同一个视觉语言。

## 4. 事件中心（events.html）

### 这个页面回答什么

> 最近有什么跟资沉变化相关的事件需要我处理？

### 设计决策与理由

#### 事件类型围绕资沉变化

| 事件类型 | 颜色 | 本质 |
|---------|------|------|
| 资沉流失预警 | 红 | 资沉在减少 |
| 理财到期提醒 | 金 | 资沉即将减少（除非续存） |
| 资沉增长通知 | 绿 | 资沉在增加 |

**旧版事件是围绕"交易行为"设计的**（爆仓、盈利、充值）。新版事件围绕"资沉变化"设计。

**理由：** 交易行为是过程，资沉变化是结果。BD 不需要知道客户每笔交易的细节，BD 需要知道的是"客户的钱是在增加还是在减少，为什么"。

#### 事件列表按紧急程度排序，不按时间

**理由：** 时间排序回答"最近发生了什么"，但 BD 更需要回答"现在最该处理什么"。一个 3 天前的大额流失预警，比 1 小时前的小额入金通知更紧急。

**这怎么服务第一性原理：** 每个事件都是资沉变化的信号。事件中心的职责是**把最重要的资沉变化信号推到 BD 面前**。

## 5. 事件线程（event-thread.html）

### 这个页面回答什么

> 这个事件到底怎么回事，我该怎么处理，进展到哪了？

### 设计决策与理由

#### 结构：事件头 → AI 分析 → 建议动作 → 任务状态

**理由：** 遵循 BD 的认知流——先看"发生了什么"（事件头），再看"为什么"（AI 分析），再看"做什么"（建议），最后标记"做到哪了"（状态）。

#### 事件头的核心指标是资沉变化金额

```
资沉流失预警 · 张伟
-$150 万
```

**理由：** 不是交易额，不是盈亏，是**资沉变化金额**。这个数字直接告诉 BD"我们正在损失多少钱"。

#### AI 分析用自然语言，不用数据表格

```
该客户近7天持续出金，30日均资沉从 $1,350万 降至 $1,200万。
理财产品已到期未续存。
```

**理由：** BD 不是数据分析师，不会看表格里的数字做推理。自然语言直接告诉 BD 结论和因果关系，零认知成本。

#### 四个任务状态：未开始 / 跟进中 / 已完成 / 已忽略

**理由：** 这是最简化的任务管理。BD 只需要标记"我还没开始"、"我在跟"、"搞定了"或"不管了"。不需要更复杂的工作流。

"已忽略"很重要——它给了 BD 一个**合法的关闭理由**。不是所有事件都值得跟进，BD 需要能说"这个我知道了但不打算处理"。

**这怎么服务第一性原理：** 事件线程是"发现资沉变化 → 理解原因 → 采取行动 → 确认结果"这个闭环的载体。没有这个闭环，事件就只是通知——看完就忘了。

## 6. 个人页面（profile.html）

### 这个页面回答什么

> 我自己的基本信息和设置。

### 设计决策与理由

**MVP 阶段这个页面是占位。** 因为它不直接服务于"保护和增长资沉"这个核心目标。

未来可能放的内容：
- BD 的客户盘总资沉概览
- 个人 KPI（管理的总资沉、流失率、增长率）
- 设置（通知偏好、语言）

但 MVP 不做——先把前五个页面做好。

## 7. 设计原则总结

### 原则 1：资沉是唯一的北极星

每个页面的核心数字、排序标准、事件触发条件、动作建议，都指向同一个量：资金沉淀。不混入交易量、净贡献、效率等旧世界的指标。

### 原则 2：先看盘，再看变化，再行动

页面的信息层次严格遵循这个顺序。首页先看"盘子多大"（总量 Tab），再看"在变吗"（流失/增长 Tab）。详情页先看资沉数字，再看趋势，再看原因和动作。

### 原则 3：给结论，不给数据

BD 不需要知道"R² = 0.82"，BD 需要知道"趋势明确"。BD 不需要知道"AdjustedTrendRate = -0.032"，BD 需要知道"每周跌 3.2%"。所有后台算法对前台不可见，前台只展示人话。

### 原则 4：一个动作，不给选择

每个客户只给一个建议动作，每个事件只给一套建议方案。减少决策负担，增加执行速度。

### 原则 5：宁可多看一个没事的人，不要漏掉一个在跑的人

趋势可信度（Conf）的下限设为 0.5，意味着即使趋势不确定的客户也不会被完全过滤掉。对销售来说，false negative（漏掉真正在流失的客户）的成本远大于 false positive（多看了一个没事的客户）。

### 原则 6：透明优先于精确

资沉总量榜只按金额排序，不掺任何因子。BD 一看就懂"为什么这个人排第一"。黑箱排序在数学上可能更优，但在销售信任上一定更差。
