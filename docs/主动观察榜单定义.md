# A 交易所销售工具｜主动观察榜单定义（Markdown 版）

## 1. 文档目的

本文档用于定义 A 交易所销售工具中的“主动观察”模块，明确榜单的本质、边界、命名、入榜规则、排序逻辑与展示结构，确保该模块始终服务于一个核心目标：

**帮助销售更快发现值得优先维护、挖潜和深耕的客户。**

这意味着，前台给销售看的不应该是原始数据本身，而应该是：

- 客户机会
- 客户画像
- 入榜理由
- 主价值方向
- 建议关注点

原始数据只在后台作为证据和计算基础存在，不直接成为销售的主界面。

---

## 2. 第一性原理定义

### 2.1 模块本质

“主动观察”不是一个数据看板，也不是一个事件中心，更不是一个传统 BI 排行榜。

它的本质是：

> **把后台数据分析压缩成可被销售理解和消费的客户机会榜单。**

榜单表达的不是“谁的数据更大”，而是：

> **谁更值得销售优先投入时间，以及为什么值得。**

### 2.2 销售真正需要解决的问题

销售主动打开该模块时，本质上只在做一件事：

> **做注意力分配。**

因此，榜单存在的唯一理由，是帮助销售更快回答这句话：

> **“我现在先看谁，为什么看他？”**

### 2.3 主动观察 vs 被动通知

为了避免边界模糊，建议内部统一用语：

| 模块 | 本质 | 触发方式 | 前台定位 |
|---|---|---|---|
| 主动观察 | 客户机会识别与解释 | 销售主动打开查看 | 拉模式 |
| 被动通知 | 事件触发与提醒 | 系统主动推送给销售 | 推模式 |

因此，诸如“大额充值”“大额申购”“高额消费”等事实，在本模块中不应被称为“通知事件”，而应被理解为：

> **主动观察理由 / 入榜成因**

---

## 3. 一级榜单设计结论

基于第一性原理，一级主榜单建议只有 **4 个**，外加 **1 个补充榜**。

原因很明确：

- 主榜单必须围绕销售的经营任务来命名
- 不能围绕业务线或原始数据字段命名
- 短期事实不能和长期经营优先级抢主入口

### 3.1 榜单清单

| 类型 | 主标题 | 副标题 | 作用 |
|---|---|---|---|
| 主榜单 1 | 稳住价值 | 重点客户·价值观察 | 识别并维护已知大客户 |
| 主榜单 2 | 释放资产 | 高潜客户·资产观察 | 挖掘资产底盘强但价值未释放的客户 |
| 主榜单 3 | 做深强项 | 单业务强客户·深度观察 | 从客户强项业务切入，做深平台关系 |
| 主榜单 4 | 扩展关系 | 业务空间客户·渗透观察 | 找到已有基础但仍有明显业务空白的客户 |
| 补充榜单 | 近期待看 | 近期事实·主动观察 | 承接近期值得主动翻看的事实，不抢主入口 |

### 3.2 为什么不能直接做“现货榜 / 期货榜 / 理财榜 / U 卡榜 / 云矿机榜”

因为这种划分方式站在业务部门视角，而不是销售视角。

这类榜单表达的是：

> 哪条业务的数据更大

而销售真正需要的是：

> 哪个客户更值得我现在花时间，以及我该从什么角度理解他

因此，业务榜并不消失，但必须降级为二级视角或证据维度，而不能成为一级主榜单入口。

## 4. 榜单共通规则

所有榜单必须共享以下底层规则。

### 4.1 客户实体归并

榜单看到的对象必须是“客户实体”，而不是单个 UID。

必须先完成：

- 主子账户归并
- 机构主体归并
- 明确的关联账户归并

否则销售看到的不是客户机会，而是数据碎片。

### 4.2 经营池筛选

以下对象不应进入主动观察池：

- 内部账户
- 测试账户
- 做市账户
- 代理结算类账户
- 不可经营账户
- 风控强限制账户（视业务规则可排除或单列）

### 4.3 门槛入池 + 榜内排序

所有榜单不采用纯 Top N 裸排，而采用两层机制：

1. 先判断有没有资格进入某个榜单
2. 再判断在该榜单内谁更值得排前

这样可以避免把榜单误做成纯排名游戏。

### 4.4 时间口径统一

建议所有主榜单统一采用三层时间口径：

- 近 90 天：看稳定价值和长期关系
- 近 30 天：看近期状态
- 近 7 天：只用于“近期待看”补充榜

### 4.5 卡片表达结构统一

所有榜单前台统一使用“机会卡”而不是宽表。每张卡只展示以下内容：

- 客户名称或标识
- 机会类型
- 入榜理由
- 主价值方向
- 建议关注点
- 两个数值锚点

---

## 5. 榜单可执行规则表

## 5.1 稳住价值｜重点客户·价值观察

### 定义

该榜单服务的是已知大客户。它解决的问题不是“谁最近有动作”，而是：

> **哪些客户已经形成稳定价值，必须持续维护。**

### 规则表

| 项目 | 规则 |
|---|---|
| 服务对象 | 已知大客户、确定性高价值客户、多业务稳定贡献客户 |
| 核心问题 | 哪些客户已经形成稳定价值，必须优先维护 |
| 入榜条件 | 满足以下任一条件即可进入候选池：近 90 天综合净贡献进入重点人群；近 90 天日均资产进入重点人群；至少两条核心业务同时形成显著贡献且 90 天稳定 |
| 排序逻辑 | 先看近 90 天综合净贡献，再看近 90 天日均资产，再看多业务关系深度，最后看近 30 天状态 |
| 主价值方向 | 综合价值、期货价值、现货价值、理财价值、多业务综合关系 |
| 建议关注点 | 稳住现有关系，维持核心资产与交易关系，避免被竞争平台撬走 |
| 卡片数值锚点 | 近 90 天综合净贡献 + 近 90 天日均资产；或主业务贡献 + 近 90 天日均资产 |
| 二级视角 | 综合贡献、资产稳定、多业务关系 |
| 典型理由模板 | “近 90 天综合贡献进入重点人群，当前主要价值在期货。” / “平台资产长期处于重点人群，且多业务关系稳定。” / “现货与理财贡献同时显著，属于持续维护客户。” |

### 说明

这个榜单必须拒绝“短期热闹”干扰。一个客户最近一周有大动作，但如果在 90 天维度上不具备稳定价值，就不应该压过真正的大客户。

---

## 5.2 释放资产｜高潜客户·资产观察

### 定义

该榜单服务的是资产型潜力客户。它解决的问题是：

> **哪些客户的钱已经在平台上，但价值还没有被充分转化成更深的平台关系。**

### 规则表

| 项目 | 规则 |
|---|---|
| 服务对象 | 资产底盘强但价值释放不足的客户，高资金沉淀但业务渗透浅的客户 |
| 核心问题 | 哪些客户有明显资金基础，但平台关系还没做透 |
| 入榜条件 | 满足以下组合条件之一：近 90 天日均资产进入重点人群且综合净贡献与资产规模不匹配；总资产高但业务覆盖低；稳定币或闲置资产占比高且理财/交易/U 卡渗透浅 |
| 排序逻辑 | 先看资产底盘质量和稳定性，再看未释放程度，再看切入口清晰度，最后看近 30 天状态 |
| 主价值方向 | 资产沉淀、稳定币沉淀、理财转化空间、交易转化空间 |
| 建议关注点 | 释放资产价值，把沉淀资金转化为理财、交易、U 卡或更深的平台关系 |
| 卡片数值锚点 | 近 90 天日均资产 + 业务覆盖度；或近 90 天日均资产 + 闲置资产比例 |
| 二级视角 | 高资产低交易、高资产低理财、高资产低覆盖 |
| 典型理由模板 | “平台资产沉淀进入重点人群，但理财渗透较低。” / “近 90 天日均资产较高，综合关系仍偏单一。” / “稳定币沉淀显著，但交易和理财价值尚未释放。” |

### 说明

这个榜单不是“谁最有钱”，而是“谁最值得把钱转化出来”。  
只有“有底盘”但又“未释放”的客户，才应进入该榜单。

---

## 5.3 做深强项｜单业务强客户·深度观察

### 定义

这是最接近业务排行需求的榜单，但其表达必须从“业务榜”翻译成“经营榜”。

它解决的问题是：

> **哪些客户已经在某一条业务线上做深，值得从强项业务切入做深平台关系。**

### 规则表

| 项目 | 规则 |
|---|---|
| 服务对象 | 在现货、期货、理财、U 卡、云矿机某一条业务线上显著领先的客户 |
| 核心问题 | 哪些客户已经在某条业务线上做深，值得从强项切入做深关系 |
| 入榜条件 | 某单一业务深度进入重点人群，且业务覆盖不完整或综合关系集中在单一业务 |
| 排序逻辑 | 先看主业务深度，再看该业务净价值，再看从强项延展出去的潜力，最后看资产底盘 |
| 主价值方向 | 当前主要价值在现货 / 期货 / 理财 / U 卡 / 云矿机 |
| 建议关注点 | 从强项切入，扩大平台关系，不要只停留在单业务关系 |
| 卡片数值锚点 | 主业务深度值 + 综合资产或综合净贡献 |
| 二级视角 | 现货强项、期货强项、理财强项、U 卡强项、云矿机强项 |
| 典型理由模板 | “期货深度进入重点人群，但平台关系集中在单一业务。” / “现货贡献显著，理财与 U 卡仍有扩展空间。” / “U 卡使用深度较高，但平台资产沉淀仍偏浅。” / “云矿机投入规模较高，综合关系尚未形成。” |

### 说明

你最初提出的现货、期货、理财、U 卡、云矿机各类排行，在这里全部被保留，但身份发生了变化：

> 它们是“做深强项”的二级视角，而不是首页一级入口。

---

## 5.4 扩展关系｜业务空间客户·渗透观察

### 定义

该榜单服务的是关系不够宽、不够完整，但已经有基础信号的客户。

它解决的问题是：

> **哪些客户已经有基础关系，但平台业务还没做透。**

### 规则表

| 项目 | 规则 |
|---|---|
| 服务对象 | 已有资产、已有单业务优势或已有贡献，但平台关系覆盖窄的客户 |
| 核心问题 | 哪些客户已经有基础关系，但平台业务还没做透 |
| 入榜条件 | 至少具备一类强信号（价值、资产或单业务深度），同时存在一个或多个关键业务空白，且这个空白与现有关系之间有自然延展性 |
| 排序逻辑 | 先看已有关系基础，再看空间明确度，再看转化可行性，最后看客户稳定性 |
| 主价值方向 | 当前主要价值在资产 / 现货 / 期货 / 理财 / U 卡 / 云矿机中的已有强项 |
| 建议关注点 | 补齐关键业务空白，提升综合关系深度 |
| 卡片数值锚点 | 已有基础值 + 业务覆盖度 |
| 二级视角 | 交易转理财、资产转交易、U 卡转留存、云矿机转综合关系 |
| 典型理由模板 | “平台资产较高，但理财关系仍为空白。” / “期货业务显著，现货与资产沉淀仍有空间。” / “U 卡使用活跃，但平台留存关系较浅。” / “已形成单业务优势，综合关系覆盖偏窄。” |

### 说明

这个榜单和“做深强项”的区别必须清楚：

- “做深强项”强调：某条业务已经很强
- “扩展关系”强调：客户已经有基础，但平台关系仍有明显空白

一个客户可以同时进入两个榜单，但首页只能保留一个主机会类型。

---

## 5.5 近期待看｜近期事实·主动观察（补充榜）

### 定义

这个榜单承接的是“值得主动翻看的近期事实”，但它不应该和前四个主榜单抢主入口。

它解决的问题是：

> **最近有哪些事实值得销售主动翻一下，但不一定要立刻推送通知。**

### 规则表

| 项目 | 规则 |
|---|---|
| 服务对象 | 近期出现显著事实、值得销售主动查看的客户 |
| 核心问题 | 最近有哪些事实值得销售主动翻看 |
| 入榜条件 | 近 7 天发生显著事实，如大额充值、大额理财申购、高额 U 卡消费、大额云矿机购买、资产台阶式上升；同时客户满足基础价值底线，或事实本身强度极高 |
| 排序逻辑 | 先看事实强度，再看客户基础价值，再看事实与现有关系的关联性 |
| 主价值方向 | 延续该客户原有主价值方向，不单独发明新方向 |
| 建议关注点 | 确认近期事实背后的价值释放方向 |
| 卡片数值锚点 | 近期事实值 + 一个客户基础值 |
| 二级视角 | 大额入金、大额申购、高额消费、大额购买、新晋高值 |
| 典型理由模板 | “近 7 天出现大额充值，且平台资产基础较强。” / “近 7 天理财申购显著，值得确认资产转化趋势。” / “近期 U 卡消费台阶式上升，建议主动查看平台关系变化。” |

### 说明

这个榜单不能被做成流水消息流。  
如果最后只是在播报“谁刚刚充值了”“谁刚刚消费了”，那它就不是主动观察，而是延迟版通知流。

## 6. 首页去重与主机会类型优先级

首页不应该是五个榜单的简单拼接，而应该是一个：

> **去重后的客户机会流**

同一个客户在首页只能保留 **一个主机会类型 + 一条主理由**。

建议首页主机会类型优先级固定为：

1. 稳住价值
2. 释放资产
3. 做深强项
4. 扩展关系
5. 近期待看（仅作补充标记，不抢主位）

这个优先级本质上就是销售时间的优先级：

- 先稳住已经重要的人
- 再挖掘资产底盘强的人
- 再深耕业务强项客户
- 再补齐关系空白
- 最后再看近期事实

进入具体榜单后，允许同一客户重复出现；首页必须去重。

---

## 7. 榜单与业务排行的关系

为了避免误解，这里明确写清楚：

现货、期货、理财、U 卡、云矿机等业务维度排行并不消失，但它们的定位被调整为：

> **主榜单内的二级视角 / 客户详情内的证据维度**

它们不应作为一级主入口存在。

因此：

- 现货交易量头部客户 → 归入“做深强项”
- 期货交易量头部客户 → 归入“做深强项”
- 理财持有量头部客户 → 归入“做深强项”或“稳住价值”
- U 卡使用头部客户 → 归入“做深强项”
- 云矿机投入头部客户 → 归入“做深强项”或“释放资产”

---

## 8. 为什么这套榜单符合本质需求

### 8.1 它服务的是销售，而不是数据部门

榜单按“经营任务”组织，而不是按“数据字段”组织。  
销售看到的是客户机会，而不是指标平铺。

### 8.2 它既覆盖已知大客户，也覆盖潜在大客户

- “稳住价值”覆盖已知大客户
- “释放资产 / 做深强项 / 扩展关系”覆盖不同类型的潜在大客户

### 8.3 它既保留业务颗粒度，又不让业务颗粒度绑架首页

业务排行被正确保留下来，但降级为二级视角。  
这既不损失业务判断力，也不让首页退化成 BI 报表。

### 8.4 它把近期事实放在正确的位置

近期事实有价值，但不是主入口。  
它被放进补充榜，不会盖过长期经营优先级。

### 8.5 它天然支持“数据 → 画像 → 理由 → 策略”的链路

每个榜单都不只是一个列表，而是一个经营语义容器，天然适合承接后续的数据转译逻辑和推荐理由模板。

## 9. 下一步建议

本轮已经完成“榜单层”的定义。  
下一步建议继续固化两张规则表：

1. **数据支撑白名单表**  
   明确每个榜单背后允许使用哪些字段和口径。

2. **数据转理由规则表**  
   明确如何从原始数据计算出机会类型、入榜理由、主价值方向与建议关注点。

到这一步，产品、数据、策略三方就可以进入真正的内部对齐。

---

## 10. 一句话总结

> **主动观察不是给销售看客户数据，而是把后台数据压缩成四类主榜单和一个补充榜，让销售更快发现谁值得优先维护、挖潜和深耕。**
