# 用户贡献收益查询系统 — 业务逻辑梳理

> 基于项目代码分析整理，最后更新：2026-03-28

---

## 一、系统目标

**核心问题**：计算大客户在平台上贡献的净利润（即平台从该用户身上赚到的钱）。

**计算公式**：

```
用户净利润贡献 = 5个交易模块收入 − 理财利息支出 − 返佣支出
```

- 5 个交易模块产生**平台正向收入**（手续费 + 用户亏损）
- 理财和返佣是**平台成本**（利息支出 + 渠道返佣）

---

## 二、7 大业务模块总览

| # | 模块 | 性质 | 一句话说明 |
|---|------|------|-----------|
| 1 | 双仓合约 | 收入 | 用户做合约交易，平台赚手续费和对赌盈亏 |
| 2 | 超级杠杆 | 收入 | 用户做杠杆交易，平台赚手续费和对赌盈亏 |
| 3 | 理财 | **支出** | 平台给用户提供利息，是平台的成本 |
| 4 | 金融卡 | 收入 | 用户开卡和刷卡消费产生的手续费 |
| 5 | 云算力 | 收入 | 用户购买算力和电费，平台直接赚取 |
| 6 | Mini合约 | 收入 | 简化版合约交易，手续费 + 对赌盈亏 |
| 7 | 反佣 | **支出** | 平台付给渠道代理的返佣，是成本项 |

## 三、各模块详细拆解

### 3.1 双仓合约（Futures）

**业务场景**：用户在双仓合约市场做期货交易（如做多/做空 BTC）。

**UID 映射**：合约系统有独立的用户体系，一个主站用户可能对应多个合约账户，需要先做 UID 映射才能查到数据。

**平台怎么赚钱**：取决于订单是否对冲——

| 订单类型 | 平台角色 | 平台收益 |
|---------|---------|---------|
| 对冲单 | 中间商，把订单转到外部交易所 | 只赚**手续费** |
| 非对冲单 | 当庄家，和用户对赌 | **手续费 + 用户亏损**（用户赚则平台亏） |

**总结**：对冲单稳赚手续费，非对冲单赌用户亏损，高风险高收益。

---

### 3.2 超级杠杆（Margin）

**业务场景**：用户用杠杆做现货交易（如 10x 杠杆做多 BTC），只计算**已平仓**的订单。

**平台怎么赚钱**：逻辑和双仓合约一致——

| 订单类型 | 平台收益 |
|---------|---------|
| 对冲单 | 开仓手续费 + 平仓手续费 |
| 非对冲单 | 开仓手续费 + 平仓手续费 − 用户盈亏 |

**总结**：和双仓合约本质相同，都是"对冲赚手续费，非对冲当庄家"。

---

### 3.3 理财（Finance）

**业务场景**：用户购买平台理财产品（如 USDT 活期/定期），平台给用户发放利息。

**平台怎么"赚钱"**：**不赚钱，这是纯支出**。
- 用户赚到的利息就是平台付出去的钱
- 用户理财规模越大、利率越高 → 平台成本越大
- 在净利润计算中，这个模块始终是**负值**，拉低总利润

**只统计已发放的利息**，未发放的不计入。

---

### 3.4 金融卡（Card）

**业务场景**：用户申请加密货币金融卡（借记卡），用于日常消费。

**平台怎么赚钱**：纯手续费收入，两个来源——
1. **开卡费**：用户申请新卡时支付的一次性费用
2. **消费手续费**：用户每次刷卡消费时平台抽取的手续费

**总结**：没有对赌风险，是稳定的手续费收入。

---

### 3.5 云算力（Cloud Mining）

**业务场景**：用户在平台购买云算力产品（挖矿），平台代管矿机。

**平台怎么赚钱**：用户支付的费用就是平台收入——
1. **算力费用**：用户购买算力（hashrate）支付的金额
2. **电费**：用户承担的矿机电力成本

对于主站用户，平台收多少就赚多少，不需要额外换算。

---

### 3.6 Mini合约（Mini Futures）

**业务场景**：简化版合约交易，门槛更低，适合小额用户。

**平台怎么赚钱**：和双仓合约类似，但**不区分对冲/非对冲**，所有订单统一计算——

平台收入 = 开仓手续费 + 平仓手续费 + 资金费率 − 用户盈亏

即：**所有手续费 − 用户赚的钱**。用户亏损时平台额外赚钱，用户盈利时从手续费中扣除。

---

### 3.7 反佣（Commission / Rebate）⭐ 重点

**业务场景**：平台的代理/渠道推广体系。

**反佣机制**：
1. 代理 A 推荐用户 B 注册平台
2. 用户 B 在平台交易（合约、杠杆等），产生手续费
3. 平台从手续费中按约定比例**返还给代理 A**，这就是返佣
4. 返佣在交易结算后自动计算和分发，只统计**已结算**的

**平台怎么"赚钱"**：**不赚钱，这是纯支出**。
- 返佣是平台获客成本，付给渠道代理的回扣
- 返佣越高 → 平台净利润越低

**当前系统的查询维度**：
- 查的是"这个用户**作为代理**收到了多少返佣"
- **不是**"这个用户**作为交易者**，他的交易让平台给他的上级代理付了多少返佣"
- 所以：如果用户本身不是代理，返佣为 0

## 四、关键概念

### 4.1 对冲 vs 非对冲

这是理解双仓合约和超级杠杆利润的核心概念。

**对冲（Hedge）**：平台把用户订单转发到外部交易所执行。风险在外部，平台只赚固定手续费。类似"中间商"。

**非对冲（Non-Hedge）**：平台自己做对手方，和用户对赌。用户亏损 = 平台盈利，用户盈利 = 平台亏损。平台同时还收手续费。类似"庄家"。

> 举例：用户做多 BTC，亏了 100 USDT，手续费 2 USDT
> - 如果是对冲单：平台赚 2 USDT（只赚手续费）
> - 如果是非对冲单：平台赚 102 USDT（手续费 + 用户亏损）

### 4.2 渠道限制

所有查询都只看**主站（channel_id=101）**用户数据，不支持其他站点或白标。

### 4.3 精度处理

所有金额截断到 8 位小数（不是四舍五入），因为加密货币交易精度到聪（satoshi，10^-8）。API 响应中金额用字符串传输，防止精度丢失。

## 五、净利润计算全景

### 单用户实时查询

输入用户 UID + 时间范围，系统**并行查询** 7 个模块，然后汇总：

```
净利润 = 双仓合约 + 超级杠杆 + 金融卡 + 云算力 + Mini合约
         − |理财利息|
         − |返佣|
```

### 排行榜（预计算）

为了支持 TOP N 排行，系统会每小时预计算 TOP 1000 用户（按资产排名）的每日数据，提供：
- 当前周期各模块收益 + 总净利润
- 周环比：与上一周同期对比变化
- 8 周均值对比：与过去 8 周平均值对比
- 用户资产（资沉）数据

## 六、BD 客户分层框架（活水模型）

> 框架名称：**活水模型**
> 取意"问渠那得清如许，为有源头活水来"。高潜客户的资金沉淀是死水，BD 的工作是让它流动起来，转化为贡献。

---

### 核心三量

**N90 — 综合净贡献**
```
N90 = 近90天，这个客户给集团赚到的总净利润
    = 双仓合约 + 超级杠杆 + Mini合约 + 金融卡 + 云算力
      − 理财利息支出 − 返佣支出
```
> 回答的问题：这个客户现在值多少钱？

**F90 — 日均资沉**
```
F90 = 近90天，这个客户每天在平台上的平均资金总量
    = 每天全产品资金快照求均值
      （双仓、现货、衍生品、股票、跟单、趣投、理财、借贷、DeFi、超杠均计入）
```
> 回答的问题：这段关系的资金底盘有多厚？用日均而非快照，避免短期大额进出造成误判。

**E90 — 资金效率**
```
E90 = N90 ÷ F90
    = 每 1 元日均资沉，在近90天产生了多少净贡献
```
> 回答的问题：这笔钱有没有被充分变现？

---

### 关键阈值

**T_N — 高贡献阈值**
```
T_N = 同层客户 N90 的第90百分位（P90）
    = 贡献排在前10%的最低门槛
```

**T_F — 高资沉阈值**
```
T_F = 同层客户 F90 的第80百分位（P80）
    = 资沉排在前20%的最低门槛
```

**T_E — 变现均线**
```
T_E = 筛出 F90 ≥ T_F 的高资沉客户
      再取这批人 E90 的中位数（P50）
    = 同等资沉规模下，效率的中间水平
```
> 低于变现均线，说明这个客户的钱在平台上，但集团还没从这笔钱上赚到应有的利润。
> T_E 从内部数据自动计算，无需人工写死，每季度重算一次随业务自动调整。

---

### 分类决策树

```
输入：N90（实际贡献）、F90（日均资沉）、E90（资金效率）

if N90 ≥ T_N（高贡献阈值）:
    → 高价值客户（贡献达标型）
      集团已从该客户身上赚到足够多的钱

elif F90 ≥ T_F（高资沉阈值）且 E90 ≥ T_E（变现均线）:
    → 高价值客户（沉淀优质型）
      钱多且变现效率健康，即使绝对贡献暂未进前10%

elif F90 ≥ T_F（高资沉阈值）且 E90 < T_E（变现均线）:
    → 高潜客户（资产待释放型）
      钱已经在平台上，但集团还没充分赚到这笔钱

else:
    → 非主榜客户（不进入BD主动观察名单）
```

---

### 榜单排序规则

**高价值榜 — 按 N90 降序**
```
排序依据 = N90（综合净贡献）
谁给集团赚得最多，谁排第一
```

**高潜榜 — 按机会缺口降序**
```
理论贡献 = F90 × T_E
         = 按变现均线，这个客户本应给集团贡献多少

机会缺口 = 理论贡献 − N90（实际贡献）
         = F90 × T_E − N90
         = F90 × （T_E − E90）    ← 三种写法等价

机会缺口越大，说明"可挖潜的空间越大"，BD 应优先跟进
```

> **示例**：吴建国 F90 = $20M，变现均线 T_E = 0.30%
> - 理论贡献 = $20,000,000 × 0.30% = **$60,000**
> - 实际贡献 N90 = **$18,000**
> - 机会缺口 = $60,000 − $18,000 = **$42,000**
> 意味着：如果 BD 能把吴建国的效率提升到均线水平，集团每90天可多赚 $42,000。

---

### 辅助标签（仅高价值客户）

```
沉淀强：F90 ≥ T_F（高资沉阈值）
      → 关系底盘稳定，贡献有保障

沉淀弱：F90 < T_F（高资沉阈值）
      → 贡献虽高但资金底盘薄，关系偏脆
        典型：高频合约手，赚钱多但随时可能转平台
        BD 动作：提升资沉，加深关系绑定
```

---

### 分层逻辑说明

- **高价值与高潜互斥**：高潜的前提是 N90 未达 T_N，已达到 T_N 的一律归高价值
- **变现均线是相对值**：由真实客户数据决定，不是拍脑袋的固定数字
- **`沉淀弱`是风险信号**：这类客户是流失风险最高的群体，需 BD 重点跟进
- **高潜客户的 BD 动作核心**：提升 E90，让已有资沉转化为更多贡献

## 七、指标观测框架（三层）

客户详情页的三个核心指标，分别对应三种不同的观测目的，采用不同的对比逻辑：

### 第一层：主分类（绝对水平）

```
N90  综合净贡献  →  对比上期90天（季度环比）
F90  日均资沉    →  用于后台分类，不做短期对比
```

- 回答：这个客户**是谁**，属于高价值还是高潜
- 用长窗口（90天）过滤短期波动噪音，N90 本身就波动大，用季度对比才能看清结构性趋势

### 第二层：趋势标签（相对历史常态）

```
近7日日均资沉  vs  前8周均值

前8周均值 = (W₋₁ + W₋₂ + ... + W₋₈) / 8   ← 本周不参与计算，避免自我稀释

趋势变化 = (本周值 − 前8周均值) / 前8周均值
```

- 回答：这个客户**最近有没有异动**，钱是在增加还是在离开
- 用短窗口（7天）捕捉信号，用中期窗口（8周）作为基准，平衡灵敏度与稳定性
- **注意**：基准必须为前8周（不含本周），本周纳入均值会稀释异动幅度，导致预警失灵
- 与事件系统（出金预警、资沉异动事件）使用同一口径

### 第三层：结构健康度（横向对标）

前端展示使用**机会缺口**和**机会缺口比率**，而非直接展示 E 值和 T_E，降低 BD 理解成本。

**机会缺口（绝对值）**
```
机会缺口 = F90 × (T_E − E)

含义：按集团中位效率标准，这个客户近期还有多少净贡献未被释放
```

**机会缺口比率（相对值）**
```
机会缺口比率 = (T_E − E) / T_E

含义：该客户的效率距离集团中位数还差多少
```

**分级标准**

| 比率 | 等级 | 前端展示 | BD 行动方向 |
|------|------|---------|------------|
| > 60% | 高 | 红色 [高] | 优先开发，潜力巨大 |
| 30–60% | 中 | 黄色 [中] | 积极跟进，有明显空间 |
| < 30% | 低 | 灰色 [低] | 维护为主，接近均线 |
| E ≥ T_E | — | 绿色「贡献健康·重点维护」 | 高价值客户，维护关系 |

**注意事项：**
- E 的时间窗口（90天 / 70天等）待真实数据验证后确定，原型中统一用 E 表示
- 机会缺口仅对高潜客户有意义；高价值客户（E ≥ T_E）显示"超均线 ×N"
- BD 的目标：**至少将客户效率拉到 T_E（中位数）**，机会缺口归零时客户从高潜升级为高价值

**三层指标的前端展示映射**

| 格子 | 展示内容 | 对比基准 | 回答的问题 |
|------|---------|---------|-----------|
| ① 综合净贡献 | N90 + 季度环比 | 上期90天 | 这个客户值多少？ |
| ② 近7日资沉 | F7 + 趋势 | 前8周均值 | 钱最近有没有异动？ |
| ③ 机会缺口 | 缺口金额 + 比率等级 | 变现均线 T_E | 还有多少潜力没释放？ |

## 八、BD 行动效果衡量

### 礼金激活率

```
礼金激活率 = 客户已使用礼金次数 ÷ BD 已发送礼金总次数
```

**说明：**
- 礼金包括代金券、手续费抵扣券、专属活动邀约等所有形式的激励
- 激活 = 客户实际使用（兑换/参与），而非仅接收
- 以客户为单位统计，反映该客户对 BD 激励动作的响应程度

**参考意义：**
- 礼金激活率高 + E90 提升 → BD 动作有效，激励策略可持续
- 礼金激活率高 + E90 未提升 → 客户接受礼金但行为未改变，需调整策略
- 礼金激活率低 → 客户对激励不敏感，或接触时机不对

---

### 低响应标签（系统自动打标）

当高潜客户满足以下条件，系统自动添加「低响应」标签：

```
触发条件（同时满足）：
1. 近 90 天内 BD 跟进记录 ≥ 3 次
2. 礼金发送次数 ≥ 2 次，且礼金激活率 = 0%
3. E90 较首次跟进时提升幅度 < 20%
```

**标签效果：**
- 在高潜榜中**降权排列**（同等机会缺口下，低响应客户排在后面）
- 提示后续 BD：此客户已多次跟进无明显改善，需重新评估投入价值

**两个间接价值：**
1. **保护 BD 时间**：避免在结构性无效客户上重复投入
2. **校准活水模型**：若低响应客户大量积累，说明 T_F 或 T_E 阈值可能需要调整

---

### 比较基准：8周均值

所有事件中涉及"异常程度"的比较（如出金异常、交易量异动），统一以**过去8周同期均值**作为基准：

```
异动幅度 = (当前7日值 - 过去8周同期7日均值) ÷ 过去8周同期7日均值
```

优于环比（前7天）的原因：更稳定，不受单周偶发异常干扰。
