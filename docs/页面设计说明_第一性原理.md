# Sales Hub 页面设计说明 — 从第一性原理出发

## 0. 第一性原理

**资金沉淀就是价值。**

不是"待转化的潜力"，不是"手续费的前置条件"，是价值本身。

原因：资金沉淀带来的收益（平台用资金做市、借贷、资金池等）远大于客户交易产生的手续费。一个客户把 1000 万放在平台上什么都不做，他给平台创造的价值可能比一个每天交易 100 万但不留钱的客户更大。

由此推导出销售工作的本质：

```
销售的工作 = 让更多的钱留在平台上，更久。
```

拆开就是三件事：

1. **守住大盘子** — 已有的大额沉淀不能丢
2. **止住流失** — 正在流出的钱要拦回来
3. **放大增长** — 正在流入的钱要顺势做大

所有页面设计都从这三件事出发。任何不服务于这三件事的元素，不放。

## 1. 首页（home.html）

### 这个页面回答什么

> BD 每天早上打开 App 的第一个问题：**今天有什么事要处理？谁需要我关注？**

### 最终设计：搜索 + 事件流

首页只做两件事：

1. **搜索栏**（主动入口）— BD 想找谁，直接搜
2. **事件流**（被动入口）— 系统告诉 BD 该看谁，按紧急度排序

**首页不放客户列表。** 客户列表移到独立的「客户」Tab。

### 设计决策与理由

#### 为什么首页不再放三 Tab 客户列表

旧设计把"资沉总量 / 流失 / 增长"三个 Tab 放在首页，相当于让 BD 自己翻列表找人。但 BD 早上打开 App 的核心需求是"今天有什么事要处理"，不是"让我翻翻看"。

事件流直接回答这个问题：系统已经帮你挑好了最该关注的事，按紧急程度排好了，你从上往下处理就行。

#### 搜索栏：主动入口

BD 心里有个人想看（出门见客户前、接到电话后），搜索栏让他 2 秒内找到。放在首页顶部，不需要切到客户 Tab。

#### 事件流：被动入口

每条事件卡片展示：
- 事件类型 + 颜色标识（红/金/绿/灰）
- 客户名 + 关键数字（如"资沉 -$150 万"）
- 触发时间
- 点击进入事件线程页

按紧急度排序，不按时间。BD 从上往下处理，处理完标记状态。

**这怎么服务第一性原理：** 首页的唯一职责是"把最需要关注的资沉变化推到 BD 面前"。搜索解决主动找人，事件流解决被动响应，覆盖 BD 打开 App 的所有场景。

## 2. 客户详情页（client-detail.html）

### 这个页面回答什么

> BD 点进一个客户后的四个问题：
> 1. 这是谁，多重要？
> 2. 他的钱最近怎么了？
> 3. 为什么会这样，我该做什么？
> 4. 之前跟他聊过什么？

### 10 个区块，从上到下

| 区块 | 名称 | 核心问题 |
|------|------|---------|
| A | 客户身份 | 我在看谁（头像、名字、UID、状态、系统标签、BD标签） |
| B | BD 备注 | 我之前记了什么（自由文本，localStorage） |
| C | 资沉指标 | 他的钱怎么了（8周均、本周均、趋势、稳定度） |
| D | 资金分布 | 他主要玩什么（各业务线资沉占比，默认展开） |
| E | 综合净贡献 | 给集团赚了多少（净利润贡献，默认折叠） |
| F | 客户近期盈亏 | 他自己赚了还是亏了（7天 + 8周周均，按业务拆分） |
| G | 福利金 | 我能给他什么（⚠️ 待确认） |
| H | 原因 + 动作 | 系统建议我做什么 |
| I | 最近事件 | 有没有紧急的事 |
| J | 跟进记录 | 之前的互动历史 |

### 设计决策与理由

#### 区块 A：客户身份（头像 + 名字 + 标签）

| 元素 | 为什么放 |
|------|---------|
| 头像 + 名字 | 确认"我在看谁"。人类处理面孔和名字比处理 UID 快。 |
| 状态 badge（流失/增长/稳定） | 状态是整个页面的**色调锚点**——流失客户页面偏红，增长客户偏绿。 |
| 系统标签 | 系统自动打的标签，BD 不可编辑。代表"平台认为这个客户是什么"。 |
| BD 标签（可编辑） | BD 自己打的标签，如"偏好短线"、"英文沟通"。这是 BD 的**私人记忆外挂**。 |

#### 区块 B：BD 备注（紧跟身份）

自由文本 textarea，输入即保存（localStorage）。BD 确认"我在看谁"后，立刻看到自己之前的笔记。不需要格式化模板。

#### 区块 C：资沉核心指标（页面视觉中心）

**当前资沉是整个页面最大的数字。** 第一性原理说"资沉就是价值"，那页面上最大的数字就应该是资沉。

| 指标 | 为什么放 |
|------|---------|
| 当前资沉 (8 周均) | 回答"这个客户有多重要" |
| 本周均值 | 回答"最近一周跟 8 周均值有没有偏离" |
| 近期趋势 (%/周) | 回答"在涨还是在跌，多快"（8 周线性回归） |
| 趋势稳定度 | 回答"这个趋势靠不靠谱"（R² ≥ 0.7 = 明确） |

#### 区块 D：资金分布（默认展开）

各业务线资沉占比（理财、合约保证金、现货钱包、金融卡、云算力），金额 + 百分比 + 进度条。BD 一眼看出"这人主要玩什么"。

#### 区块 E：综合净贡献（默认折叠）

集团净利润贡献总额 + 各模块明细（收入绿色、成本红色）。默认只显示总额，点击展开。大多数时候 BD 只看 D 就够，E 是深入分析时才展开。

#### 区块 F：客户近期盈亏

**两个时间窗口：** 近 7 天盈亏（当前情绪）+ 8 周周均盈亏（长期趋势），按业务拆分明细。BD 结合两者判断客户情绪和沟通策略。

#### 区块 G：福利金（⚠️ 待确认）

福利金总额度、已发放、已激活、发券按钮。待业务确认规则后落地。

#### 区块 H：原因 + 建议动作

**合并在一张卡片里。** 原因和动作是因果关系——"因为理财到期了，所以建议你去回访"。只给一个动作，不给选择题。

#### 区块 I：最近事件

只展示 2-3 条最近的事件，点击跳转事件线程。事件颜色跟客户状态联动。

#### 区块 J：跟进记录

时间线样式，最新在最上，区分人工记录和系统记录。

**这怎么服务第一性原理：** 10 个区块严格按 BD 的信息消费顺序排列——先确认身份，再看备注，再看资沉，再看分布和贡献，再看盈亏和福利金，最后看行动建议和历史。每个区块都围绕"资沉"这个唯一北极星。

## 3. 客户列表（客户 Tab）

### 这个页面回答什么

> BD 想找一个特定客户，或者想翻翻自己的客户盘。

### 最终设计：简单通讯录

客户列表从首页移到独立的「客户」Tab，定位是**通讯录**，不是分诊台。

#### 每行展示三项信息

| 内容项 | 说明 |
|--------|------|
| 客户名 + UID | 确认是谁 |
| 资沉金额 | 这个人多重要 |
| 趋势状态标签 | 流失/增长/稳定，一眼看出状态 |

就这三项，不放原因标签、不放建议动作、不放事件 badge。

#### 搜索：按名字或 UID

BD 记得名字的搜名字，从系统跳过来的用 UID。两种找人方式覆盖所有场景。

#### 为什么这么简单

首页已经承担了"今天先看谁"的分诊职责（通过事件流）。客户 Tab 的职责是"让我找到我想看的人"，不需要重复分诊逻辑。

**这怎么服务第一性原理：** 客户列表是 BD 的通讯录，按资沉排序让最重要的客户在最前面。简单 = 快，BD 找人不超过 3 秒。

## 4. 事件中心（events.html）

### 这个页面回答什么

> 最近有什么跟资沉变化相关的事件需要我处理？

### 设计决策与理由

#### 事件类型围绕资沉变化

| 事件类型 | 颜色 | 本质 |
|---------|------|------|
| 资沉流失预警 | 红 | 资沉在减少 |
| 理财到期提醒 | 金 | 资沉即将减少（除非续存） |
| 资沉增长通知 | 绿 | 资沉在增加 |

**旧版事件是围绕"交易行为"设计的**（爆仓、盈利、充值）。新版事件围绕"资沉变化"设计。

**理由：** 交易行为是过程，资沉变化是结果。BD 不需要知道客户每笔交易的细节，BD 需要知道的是"客户的钱是在增加还是在减少，为什么"。

#### 事件列表按紧急程度排序，不按时间

**理由：** 时间排序回答"最近发生了什么"，但 BD 更需要回答"现在最该处理什么"。一个 3 天前的大额流失预警，比 1 小时前的小额入金通知更紧急。

**这怎么服务第一性原理：** 每个事件都是资沉变化的信号。事件中心的职责是**把最重要的资沉变化信号推到 BD 面前**。

## 5. 事件线程（event-thread.html）

### 这个页面回答什么

> 这个事件到底怎么回事，我该怎么处理，进展到哪了？

### 5 个区块结构

| 区块 | 名称 | 内容 |
|------|------|------|
| A | 事件头 | 类型 + 客户名 + 关键数字 + 时间 |
| B | 客户摘要 | 资沉、趋势、BD标签、上次跟进 |
| C | 操作区 | 行动建议 + 话术模板 + 福利金发放 |
| D | 任务状态 | 待处理 / 跟进中 / 已完成 / 已忽略 |
| E | 跟进记录 | 针对该事件的互动历史 |

### 设计决策与理由

#### 区块 A：事件头

```
资沉流失预警 · 张伟
-$150 万 · 2小时前
```

核心指标是**资沉变化金额**，不是交易额，不是盈亏。这个数字直接告诉 BD"我们正在损失多少钱"。

#### 区块 B：客户摘要

BD 不需要跳到详情页就能获得关键上下文：当前资沉、趋势状态、BD 标签（如"偏好短线"）、上次跟进时间。3 秒内建立对这个客户的认知。

#### 区块 C：操作区（行动建议 + 话术模板 + 福利金）

**三合一操作区：**
- **行动建议：** 系统推荐的一个动作（如"安排回访"）
- **话术模板：** 对应场景的沟通话术，BD 可以直接复制发给客户
- **福利金发放：** 如果场景适合，直接在这里给客户发券（⚠️ 待确认）

把"建议做什么"、"怎么说"、"能给什么"放在一起，BD 不需要来回切换。

#### 区块 D：任务状态

四个状态：待处理 / 跟进中 / 已完成 / 已忽略。

"已忽略"很重要——它给了 BD 一个**合法的关闭理由**。不是所有事件都值得跟进。

#### 区块 E：跟进记录

针对该事件的互动历史，时间线样式。BD 可以添加新记录。

**这怎么服务第一性原理：** 事件线程是"发现资沉变化 → 获得客户上下文 → 采取行动（建议+话术+福利金）→ 确认结果"的完整闭环。没有这个闭环，事件就只是通知——看完就忘了。

## 6. 个人页面（profile.html）

### 这个页面回答什么

> 我自己的基本信息和设置。

### 设计决策与理由

**MVP 阶段这个页面是占位。** 因为它不直接服务于"保护和增长资沉"这个核心目标。

未来可能放的内容：
- BD 的客户盘总资沉概览
- 个人 KPI（管理的总资沉、流失率、增长率）
- 设置（通知偏好、语言）

但 MVP 不做——先把前五个页面做好。

## 7. 设计原则总结

### 原则 1：资沉是唯一的北极星

每个页面的核心数字、排序标准、事件触发条件、动作建议，都指向同一个量：资金沉淀。不混入交易量、净贡献、效率等旧世界的指标。

### 原则 2：先看盘，再看变化，再行动

页面的信息层次严格遵循这个顺序。首页先看"盘子多大"（总量 Tab），再看"在变吗"（流失/增长 Tab）。详情页先看资沉数字，再看趋势，再看原因和动作。

### 原则 3：给结论，不给数据

BD 不需要知道"R² = 0.82"，BD 需要知道"趋势明确"。BD 不需要知道"AdjustedTrendRate = -0.032"，BD 需要知道"每周跌 3.2%"。所有后台算法对前台不可见，前台只展示人话。

### 原则 4：一个动作，不给选择

每个客户只给一个建议动作，每个事件只给一套建议方案。减少决策负担，增加执行速度。

### 原则 5：宁可多看一个没事的人，不要漏掉一个在跑的人

趋势可信度（Conf）的下限设为 0.5，意味着即使趋势不确定的客户也不会被完全过滤掉。对销售来说，false negative（漏掉真正在流失的客户）的成本远大于 false positive（多看了一个没事的客户）。

### 原则 6：透明优先于精确

资沉总量榜只按金额排序，不掺任何因子。BD 一看就懂"为什么这个人排第一"。黑箱排序在数学上可能更优，但在销售信任上一定更差。
