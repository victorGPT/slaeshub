# 客户详情页 — 内容设计文档

## 1. 这个页面回答什么问题

BD 点进一个客户详情，脑子里只有四个问题：

1. **这个人是谁，多重要？**
2. **他的钱最近怎么了？**
3. **为什么会这样，我该做什么？**
4. **之前跟他聊过什么，现在什么状态？**

页面的所有内容都围绕这四个问题组织，不回答这四个问题的信息不放。

---

## 2. 区块设计

页面共 10 个区块，按以下顺序从上到下排列：

| 区块 | 名称 | 说明 |
|------|------|------|
| A | 客户身份 | 头像、名字、UID、状态、系统标签、BD标签 |
| B | BD 备注 | 自由文本，localStorage |
| C | 资沉指标 | 8周均、本周均、趋势、稳定度 |
| D | 资金分布 | 各业务线资沉占比 — 默认展开 |
| E | 综合净贡献 | 集团净利润贡献 — 默认折叠 |
| F | 客户近期盈亏 | 7天 + 8周周均，按业务拆分 |
| G | 福利金 | ⚠️ 待确认 |
| H | 原因 + 动作 | 系统建议 |
| I | 最近事件 | 近期触发的事件 |
| J | 跟进记录 | 历史互动 |

---

### 区块 A：客户身份（这个人是谁）

**回答的问题：** 我现在看的是谁，他在我的客户盘里算什么级别。

| 内容项 | 说明 | 必要性 |
|--------|------|--------|
| 客户名 | 中文名 | 必须 |
| UID | 平台唯一 ID | 必须 |
| 头像 | 首字母头像或上传照片 | 有则展示 |
| 状态标签 | 流失 / 增长 / 稳定 | 必须 |
| 系统标签 | 如"流失风险"、"资沉增长中"，系统自动打，BD 不可编辑 | 必须 |
| BD 标签 | BD 自己打的标签，如"偏好短线"、"英文沟通"，可增删 | 必须 |
| BD 归属 | 这个客户归哪个 BD 管，接入时间 | MVP 可缺省 |

**设计要点：**
- 状态标签是最重要的视觉锚点（红/绿/灰），一眼能看出这个客户现在是什么状态
- BD 标签是销售自己的"备忘贴纸"，不要限制数量，但 UI 上折叠到第二行

---

### 区块 B：BD 备注（我之前记了什么）

**回答的问题：** 我自己对这个客户的自由备注。

| 内容项 | 说明 |
|--------|------|
| 自由文本 | BD 自己写的备注，没有格式限制 |

**设计要点：**
- 一个 textarea，输入即保存（localStorage）
- 不需要格式化、不需要 @ 提及、不需要模板
- 这是 BD 最私人的工具，保持极简
- 紧跟身份区块，BD 确认"我在看谁"后立刻看到自己之前的笔记

---

### 区块 C：资沉核心指标（他的钱最近怎么了）

**回答的问题：** 盘子多大，是在涨还是在掉，掉/涨了多少。

| 内容项 | 说明 | 展示方式 |
|--------|------|---------|
| 当前资沉 | 8 周均值（USDT 折算） | **最大字号**，页面视觉中心 |
| 本周均值 | 最近一周日均资沉 | 与 8 周均值对比 |
| 趋势 | AdjustedTrendRate（线性回归斜率，已去市场） | 百分比/周，正绿负红 |
| 趋势稳定度 | R²（线性回归可信度） | R²≥0.7 "趋势明确"，<0.7 "波动较大" |
| 绝对变化 | w8 - w1 的绝对值 | 万/亿格式，带 ↑↓ 箭头 |

**设计要点：**
- 当前资沉是**整个页面最大的数字**，BD 打开页面第一眼看到的就是"这个客户有多少钱"
- 趋势用 8 周线性回归，不是简单两点对比。能区分"持续流失"和"单周波动"
- 趋势稳定度告诉 BD"这个趋势靠不靠谱"：实心箭头 = 明确，虚线箭头 = 不确定
- **不需要额外显示市场基准**（趋势已去掉市场噪音）

**展示示例：**
```
↘ -2.5%/周 · 趋势明确      ← BD 必须关注
↗ +1.8%/周 · 波动较大      ← BD 先观察
→  0.1%/周 · 稳定          ← 常规维护
```

---

### 区块 D：资金分布（这个人主要玩什么）

**回答的问题：** 客户的钱分布在哪些业务里，他是什么类型的玩家。

| 内容项 | 说明 | 展示方式 |
|--------|------|---------|
| 总资沉 | 全部业务账户资沉总额 | 大字号 |
| 各业务线资沉 | 理财、合约保证金、现货钱包、金融卡、云算力 | 金额 + 百分比 + 进度条 |

**设计要点：**
- 每条业务线一行：名称 + 金额 + 占比百分比 + 水平进度条
- 占比最大的业务线进度条最长，BD 一眼看出"这人 80% 的钱在理财"
- 这个区块是**默认展开的**，因为"这个人主要玩什么"是 BD 最基础的认知需求

**BD 能秒懂的结论：**
- 80% 在理财 → 稳健型，别推高风险产品
- 70% 在合约保证金 → 交易型，关注爆仓风险
- 钱分散在 5 个业务 → 深度用户，关系粘性强

---

### 区块 E：综合净贡献（这个人给集团赚了多少）

**回答的问题：** 这个客户在过去一段时间里，让集团净赚了多少钱，哪些业务在赚，哪些在花。

| 内容项 | 说明 | 展示方式 |
|--------|------|---------|
| 净贡献总额 | 5 个收入模块 − 理财利息 − 返佣 | 金色大字 |
| 周环比 | 与上周对比变化 | 百分比，正绿负红 |
| 各模块明细 | 7 个模块各自金额 + 进度条 | 收入绿色，成本红色 |

**设计要点：**
- **默认折叠**，只显示总额。点击展开看明细
- 收入项（合约、杠杆、Mini、金融卡、云算力）用绿色
- 成本项（理财利息、返佣）用红色
- 不需要标注"倒贴"等文字，颜色自解释

**与区块 D 的关系：**
- D 回答"钱放在哪"（客户视角）→ 默认展开
- E 回答"给集团赚了多少"（平台视角）→ 默认折叠
- 大多数时候 BD 只看 D 就够，E 是深入分析时才展开

---

### 区块 F：客户近期盈亏（这个人最近赚了还是亏了）

**回答的问题：** 客户自己的交易赚亏情况，帮 BD 判断客户情绪状态。

**注意：这跟综合净贡献是两回事。** 综合净贡献是"平台从他身上赚了多少"，客户盈亏是"他自己赚了还是亏了"。

**两个时间窗口：**

| 时间窗口 | 含义 | 用途 |
|---------|------|------|
| 近 7 天盈亏 | 当前情绪状态 | BD 判断"这个人现在心情怎么样" |
| 8 周周均盈亏 | 长期趋势 | BD 判断"这个人总体上是赢家还是输家" |

两个数字都展示，BD 结合两者判断策略。

**按业务拆分明细：**

| 内容项 | 说明 | 展示方式 |
|--------|------|---------|
| 总盈亏（近 7 天） | 所有交易业务的盈亏合计 | 大字号，正绿负红 |
| 总盈亏（8 周周均） | 8 周周均盈亏 | 与 7 天对比展示 |
| 双仓合约盈亏 | 非对冲单的用户盈亏 | 金额 + 占比进度条 |
| 超级杠杆盈亏 | 已平仓订单的用户盈亏 | 金额 + 占比进度条 |
| Mini 合约盈亏 | 用户盈亏 | 金额 + 占比进度条 |
| 现货盈亏 | ⚠️ 待确认（需确认口径：买入成本 vs 当前市值？还是已实现盈亏？） | 金额 + 占比进度条 |

**BD 策略矩阵：**

| 近 7 天 | 8 周均值 | BD 判断 |
|---------|---------|--------|
| 亏 | 赚 | 长期赢家，短期回撤，关怀但不慌 |
| 亏 | 亏 | 一直在亏，情绪可能很差，小心沟通 |
| 赚 | 亏 | 刚翻身，情绪正好，适合顺势推产品 |
| 赚 | 赚 | 一直在赢，关系稳，维护就行 |

**设计要点：**
- 放在资沉指标区块下方
- BD 一眼看出"亏在哪"，跟客户沟通时有针对性
- 客户亏钱时注意情绪，不要硬推产品
- 策略矩阵不需要在 UI 上展示，但 BD 心里要有这个框架

**⚠️ 待确认：**
- [ ] 现货盈亏数据是否可用？口径是什么？

---

### 区块 G：福利金（⚠️ 待确认）

**回答的问题：** 我能给这个客户发多少福利金，已经发了多少，还剩多少。

| 内容项 | 说明 | 展示方式 |
|--------|------|---------|
| 福利金总额度 | 该客户可用的福利金预算 | 金额 |
| 已发放 | 已经发出去多少 | 金额 + 占比 |
| 已激活 | 客户已使用多少 | 金额 + 激活率 |
| 发券按钮 | 给客户发福利金 | 操作按钮 |

**待确认事项：**
- [ ] 福利金能发到哪些场景（交易抵扣？理财加息？金融卡返现？）
- [ ] 福利金额度的计算规则（按客户资沉级别？按 BD 权限？固定额度？）
- [ ] 发券的审批流程（BD 自主发？需要主管审批？）
- [ ] 福利金与事件的关系（处理流失事件时能直接发券吗？）

**出现位置：**
- 客户详情页（区块 G）
- 事件线程页（处理事件时可直接发券）

> ⚠️ 以上为占位设计，具体方案待业务确认福利金规则后落地。

---

### 区块 H：原因 + 动作（为什么会这样，我该做什么）

**回答的问题：** 系统认为这个客户为什么值得关注，以及建议我现在做什么。

| 内容项 | 说明 | 展示方式 |
|--------|------|---------|
| 原因标签 | 为什么进入当前视图（如"理财到期/赎回"、"近期持续入金"） | 文本，卡片内 |
| 建议动作 | 当前最值得做的一件事（如"安排销售回访"） | 金色文字，突出 |

**设计要点：**
- 原因和动作放在同一张卡片里，用分隔线隔开
- **只给一个动作**，不给选择题（MVP 原则：减少决策负担）
- 动作来自白名单：销售回访 / 续存提醒+权益说明 / 资沉补贴券 / U卡积分活动 / 场景绑定说明

---

### 区块 I：最近事件（有什么需要我处理的）

**回答的问题：** 系统最近触发了哪些跟这个客户相关的事件。

| 内容项 | 说明 |
|--------|------|
| 事件类型 | 资沉流失预警（红）/ 理财到期提醒（金）/ 资沉增长通知（绿）/ 定期回访（灰） |
| 事件摘要 | 一句话描述 |
| 触发时间 | 相对时间（如"2天前"） |

**设计要点：**
- 最多显示 2-3 条最近的事件
- 每条事件左侧有颜色圆点，一眼区分紧急程度
- 点击可跳转到事件线程页
- 事件内容与客户状态相关：流失客户看到出金预警，增长客户看到入金通知

**MVP 阶段限制：**
- 当前事件数据是静态模板（按客户状态生成），不是真实触发
- 后续接入真实事件系统后替换

---

### 区块 J：跟进记录（之前跟他聊过什么）

**回答的问题：** 这个客户的历史互动记录。

| 内容项 | 说明 |
|--------|------|
| 时间 | 跟进发生的时间 |
| 执行人 | 哪个 BD 做的 / 系统自动触发 |
| 内容 | 做了什么，客户什么反应 |

**设计要点：**
- 时间线样式（左侧竖线 + 节点）
- 最新的在最上面
- 区分"人工记录"和"系统记录"（如"系统：触发资沉流失预警"）
- 顶部有"+ 添加记录"按钮

**MVP 阶段限制：**
- 当前是静态模板数据
- 后续接入真实 CRM 后替换

---

## 3. 区块排列优先级

从上到下，按 BD 的信息消费顺序排列：

```
A. 客户身份         → 先确认"我在看谁"
B. BD 备注          → 我之前记了什么（紧跟身份）
C. 资沉指标         → 他的钱怎么了
D. 资金分布         → 他主要玩什么
E. 综合净贡献       → 给集团赚了多少（折叠）
F. 客户近期盈亏     → 他自己赚了还是亏了
G. 福利金           → 我能给他什么（⚠️ 待确认）
H. 原因 + 动作      → 系统建议我做什么
I. 最近事件         → 有没有紧急的事
J. 跟进记录         → 之前的互动历史
```

**前三块（A/B/C）是核心**，BD 打开页面 3 秒内能看完。
**中间四块（D/E/F/G）是数据补充**，BD 需要深入分析时展开。
**后三块（H/I/J）是行动辅助**，BD 决定要做什么、看历史。

## 4. 明确不放的内容

| 不放的内容 | 原因 |
|-----------|------|
| ~~N90 作为主指标~~ | 改用综合净贡献折叠展示（区块 E） |
| ~~E90（变现效率）~~ | 新框架下资沉就是价值，不需要另算 |
| ~~交易量~~ | 过程量，不是结果量 |
| ~~市场整体趋势~~ | 已在回归中去掉，不需要展示 |
| ~~原始百分比趋势数字~~ | 用融合状态标签替代 |
| 客户画像/AI 标签 | MVP 不做复杂画像 |
| 竞品对比 | 无数据来源 |
| 客户净值分层 | MVP 不做 |
| 推荐多个动作 | 只给一个，减少决策负担 |

## 5. 后续迭代方向

| 优先级 | 迭代项 | 条件 |
|--------|--------|------|
| P1 | 跟进记录接入真实 CRM 数据 | CRM 系统打通 |
| P1 | 事件卡片接入真实触发系统 | 事件引擎上线 |
| P1 | 福利金规则落地 | 业务确认福利金方案 |
| P2 | 业务线具体金额和占比 | 业务级数据可用 |
| P2 | BD 归属 + 接入时间 | 客户分配系统确认 |
| P3 | 资沉趋势折线图（30 天） | 有每日快照数据 |
| P3 | 动作留痕（点击"执行动作"后自动记录） | 动作系统上线 |
