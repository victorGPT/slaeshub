# A 交易所销售工具｜主动观察核心字段计算公式及确认清单

## 1. 文档目的

本文档旨在为“主动观察”榜单中涉及的核心数据字段提供明确的计算公式与定义。由于部分业务的成本结构和利润核算方式存在差异，文档中针对不确定项提供了多个版本的计算方案。

**使用建议**：请将本文档作为与数据团队、财务团队及各业务线负责人对齐的基准材料，通过勾选或确认具体版本，最终固化为可执行的数据开发需求。

---

## 2. 核心底层字段：综合净贡献 (Comprehensive Net Contribution)

**业务意义**：衡量客户在特定周期内（如近 90 天）为平台创造的真实净利润，是“稳住价值”等榜单的最核心排序依据。

**基础公式**：
> 综合净贡献 = 现货净贡献 + 期货净贡献 + 理财净贡献 + U卡净贡献 + 云矿机净贡献

### 2.1 现货与期货净贡献（需确认返佣扣除逻辑）

交易类业务的收入主要来自手续费，但必须扣除返佣成本。

- **版本 A（最严谨：扣除所有返佣）**
  - 现货净贡献 = 现货总手续费收入 - 客户自身手续费返还 - 支付给代理商/邀请人的返佣
  - 期货净贡献 = 期货总手续费收入 + 资金费率平台抽成（如有） - 客户自身手续费返还 - 支付给代理商/邀请人的返佣
  - *适用场景：平台希望销售看到的完全是归属平台的纯利润。*

- **版本 B（销售视角：只扣除客户自身返还）**
  - 现货净贡献 = 现货总手续费收入 - 客户自身手续费返还（VIP 折扣等）
  - 期货净贡献 = 期货总手续费收入 - 客户自身手续费返还
  - *适用场景：代理商返佣被视为平台的营销成本，不应影响销售对客户本身交易价值的判断。*

### 2.2 理财净贡献（需确认盈利模式）

理财业务的利润核算通常较为复杂，取决于平台是赚取利差还是收取管理费。

- **版本 A（利差模式）**
  - 理财净贡献 = ∑ (客户各项理财持仓金额 × (平台实际投资年化收益率 - 支付给客户的年化收益率) × 持有天数 / 365)
  - *适用场景：平台将资金统一运作，赚取底层资产收益与给客户利息之间的差价。*

- **版本 B（管理费模式）**
  - 理财净贡献 = ∑ (客户各项理财持仓金额 × 平台管理费率 × 持有天数 / 365)
  - *适用场景：平台作为通道，按固定比例收取管理费或申购/赎回费。*

- **版本 C（简化替代版：AUM 估算）**
  - 理财净贡献 = 理财日均持仓总额 × 平台综合预估利润率（如 1%）
  - *适用场景：底层资产收益难以精确到单客户核算，采用统一系数进行估算。*

### 2.3 U 卡净贡献（需确认收入构成）

- **版本 A（精细核算）**
  - U 卡净贡献 = 开卡费收入 + ∑ (单笔消费金额 × 消费手续费率) + 汇率点差收入 - 发卡成本 - 支付通道成本
- **版本 B（流水估算版）**
  - U 卡净贡献 = U 卡总消费流水 × 综合利润率（如 1.5%）

### 2.4 云矿机净贡献（需确认成本结构）

- **版本 A（净利润核算）**
  - 云矿机净贡献 = 算力销售总额 - 矿机硬件折旧分摊 - 电费及运维成本
- **版本 B（销售额替代版）**
  - 云矿机净贡献 = 算力销售总额 × 综合毛利率
  - *适用场景：运维成本随市场波动大，难以实时计算，直接用销售额乘以固定毛利率代替。*

## 3. 资产与状态字段

### 3.1 日均资产 (Average Daily Assets)

**业务意义**：衡量客户资金底盘的稳定性和规模，是“释放资产”榜单的核心依据。

**计算公式**：
> 近 90 天日均资产 = ∑ (过去 90 天每日 23:59:59 的全账户总资产折合 USDT) / 90

**需确认细节**：
- 资产折合 USDT 的汇率是采用每日快照汇率，还是当前实时汇率？（建议：采用每日快照汇率更准确反映历史真实情况）。
- 是否包含冻结资产（如挂单冻结、提现冻结）？（建议：包含）。

### 3.2 闲置资产比例 (Idle Asset Ratio)

**业务意义**：用于识别资金未被充分利用的客户，是“释放资产”榜单的重要触发条件。

**计算公式**：
> 闲置资产比例 = (现货账户中的稳定币余额 + 活期理财余额) / 全账户总资产

**需确认细节**：
- 活期理财是否算作“闲置”？（建议：如果活期收益极低，可视为闲置，代表有转化为定期或结构化产品的空间）。
- 哪些币种被定义为稳定币？（USDT, USDC, DAI 等）。

### 3.3 业务覆盖度 (Business Coverage)

**业务意义**：衡量客户与平台关系的广度，用于“扩展关系”榜单。

**计算逻辑**：
定义平台的 5 项核心业务：现货、期货、理财、U 卡、云矿机。
每项业务设定一个“激活阈值”（例如：近 90 天现货交易量 > 1000 U 算作激活现货）。
> 业务覆盖度 = 已激活的业务数量 / 5

## 4. 单业务深度字段（用于“做深强项”榜单）

为了判断客户在某一条业务线上是否足够“强”，需要为每条业务线定义一个深度指标。

| 业务线 | 深度指标建议 | 激活/强项阈值示例（需业务确认） |
| :--- | :--- | :--- |
| **现货** | 近 30 天现货交易量 | > 100,000 USDT |
| **期货** | 近 30 天期货交易量 | > 1,000,000 USDT |
| **理财** | 当前理财总持仓 | > 50,000 USDT |
| **U 卡** | 近 30 天消费总额 | > 5,000 USDT |
| **云矿机** | 当前持有算力总价值 | > 10,000 USDT |

## 5. 榜单入池门槛（“重点人群”定义）

**业务意义**：决定哪些客户有资格进入主动观察的候选池。

**建议方案：相对排名 + 绝对值兜底（双重条件）**

为了适应每个销售管理约 50 个客户的规模，建议采用以下逻辑：

1. **绝对值兜底**：近 90 天日均资产 > [X] USDT，或近 90 天综合净贡献 > [Y] USDT。（过滤掉毫无价值的微小客户）。
2. **相对排名**：在满足绝对值的客户中，按对应指标排名前 [N] 名。

**需确认的具体数值**：
- 资产兜底值 [X] = ? (例如：10,000 USDT)
- 贡献兜底值 [Y] = ? (例如：500 USDT)
- 排名截断值 [N] = ? (如果销售管 50 人，建议 N 设为 15-20)
