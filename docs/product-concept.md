# Sales Hub — 产品概念文档

> 版本：v1.5 | 日期：2026-03-28 | 状态：概念阶段（榜单框架最终收敛）

## 1. 背景

加密货币交易所，非散户模式。Top 150 大客户贡献约 80% 交易额，客户少但单客价值极高。这种模式下，为大客户提供专属化服务的边际收益远高于通用产品。

## 2. 产品定位

Sales Hub 是面向 BD（商务）的核心工作台，核心目标：**让 BD 更好地服务大客户**。

### 2.1 BD 角色重定义

| 传统模式 | Sales Hub 模式 |
|----------|---------------|
| BD 只参与销售环节 | BD 获得处理更多事务的能力 |
| 客户找客服解决问题 | BD 可以直接处理部分客服类需求 |
| BD 需要自己判断该做什么 | 系统提供数据支撑的行动建议 |

**重要澄清**：Sales Hub 是给 BD 增加能力的工具，不是增加负担。传统渠道（客服等）不会被取消，BD 只是多了一个可以处理事务的入口。

### 2.2 覆盖范围

- 仅覆盖已确定的大客户，不涉及线索漏斗和客户转化。

## 3. 产品形态

| 端 | 形态 | 用户 |
|----|------|------|
| 移动端 | H5 + PWA | BD |
| PC 端 | 管理后台 | 管理层、运营 |

> **风险项：PWA 推送**
> PWA 用于原型构建是合适的选择，但推送通知在 iOS 上有限制（需用户主动授权，后台清理后失效）。推送策略（频率、打扰度、渠道备选方案）需在 MVP 验证阶段专门评估。

## 4. 用户角色

| 角色 | 职责 | 备注 |
|------|------|------|
| BD | 客户跟进、事件处理、记录反馈 | 主要用户，移动端 |
| 销售主管/管理层 | 查看团队数据、审批、客户分配 | PC 端 |
| 运营 | 配置事件规则、维护事件模板 | PC 端 |

> 三角色划分为当前假设，待验证。

## 5. 功能模块

### 5.1 首页设计：客群模块（主动观察）

#### 5.1.1 第一性原理

BD 的工作本质只有两件事：
- **守住存量**：维护已有价值，不让高价值客户流失
- **做大增量**：激活潜在价值，把待开发客户转化为更高贡献

因此首页榜单只需要两个分类，由两个核心指标驱动：

| 指标 | 说明 |
|------|------|
| 集团贡献利润 | 客户为平台创造的综合净利润（手续费 + 对赌收益等） |
| 资金沉淀 | 客户在平台的总资产底盘 |

#### 5.1.2 两大榜单与完整分类逻辑

**高价值客户**——谁现在最值钱

**默认视图：综合高价值**
```
入榜条件：近30天综合净贡献 > 门槛B
排序依据：近30天综合净贡献（从高到低）
资沉作用：仅作为卡片辅助信息展示，不参与排序

综合净贡献 =
  双仓合约净贡献
  + 超级杠杆净贡献
  + Mini合约净贡献
  + 金融卡净贡献
  + 云算力净贡献
  - 理财利息支出
  - 反佣支出
```

**可选视图：按业务线拆分（AB 测试）**

原型出两个版本对比：

**版本 A：仅综合视图**
- 只有综合净贡献排行，无业务线拆分
- 界面最简洁

**版本 B：综合 + 现货 + 合约**
- 综合视图（默认）
- 现货高价值：按现货手续费净贡献排序
- 合约高价值：按双仓合约+超级杠杆+Mini合约净贡献之和排序
- BD 可手动切换视图

> 其他业务线（理财、金融卡、云算力）暂不纳入，后续按需扩展。

**待开发客户**——谁的钱还没动起来
```
入榜条件：近90天日均资沉 > 门槛A，且资金利用率低于阈值C
排序依据：近90天日均资沉绝对值（从高到低）
  → 同样利用率低，资沉越大开发价值越大
资金利用率 = 近30天贡献利润 / 近90天日均资沉
```

**不进任何榜单**
```
贡献利润 < 门槛B 且 资沉 < 门槛A → 过滤，不值得 BD 额外投入
```

**特殊情况：高贡献低资沉客户（如合约高频玩家）**
```
通过贡献门槛但资沉极低 → 归入高价值
不计算资金利用率（全仓交易型，利用率指标对其失真）
自动打 Tag「高频交易·低资沉」提示留存风险
```

**客户收益（盈亏）的定位**
```
不纳入榜单排序
归入事件推送触发条件：
  客户本月盈利超过X → 触发事件 → 建议BD送礼/返点
  客户大额亏损 → 触发事件 → 建议BD安抚
```

> 门槛A、门槛B、阈值C 三个具体数值待数据团队根据真实客户分布确定，**后台可配置**，运营团队可随时调整，无需改代码。

#### 5.1.3 主动观察 vs 被动通知

| 模块 | 本质 | 触发方式 |
|------|------|----------|
| 客群模块（首页） | 让 BD 知道谁值得关注 | 销售主动打开（拉模式） |
| 事件推送 | 提醒 BD 有事需要处理 | 系统主动推送（推模式） |

#### 5.1.4 客户卡片结构

每张卡片展示：
- 客户名称 + 榜单标签（高价值 / 待开发）
- 两个核心数值：贡献利润 + 资金沉淀
- 系统 Tag（自动生成，反映客户特征）
- 趋势信号（风险🔴 > 机会🟡 > 稳定⚪）
- AI 行动建议（一句话）

#### 5.1.5 系统 Tag 体系

Tag 是对客户特征的补充描述，帮助 BD 快速判断维护角度。分两类：

**行为 Tag（系统自动生成）**
- `现货强` `合约强` `理财为主` `纯理财` `高频交易` `持续亏损` `持续盈利` `期货亏损` 等

**排名 Tag（每日凌晨更新）**
- `贡献 Top 10` `资沉 Top 20` 等，Top 200 以内自动打标

Tag 只做描述，不决定分类。具体维护策略由 BD 结合 Tag 和客户详情页自行判断。

#### 5.1.6 可扩展性

当前两个分类覆盖核心场景。未来可按需扩展，例如：
- `影响力客户`：拉新贡献高的客户（当前不纳入）

#### 5.1.7 下半屏：今日行动（事件驱动）

从事件推送模块获取数据，展示 BD 需要处理的事件。

**原型出两个内容版本对比：**
- 版本 A：只展示待处理事件
- 版本 B：待处理 + 跟进中，分组展示

### 5.2 模块一：客户数据统计

> 已有研发团队开发的现成版本，可直接复用或迭代。

**目的**：BD 打开客户详情就能全面了解客户状况。

**数据维度**：
- 交易数据：现货、理财、双币、U卡等全业务线
- 福利数据：单客户盈利、已投入福利金额、可投入福利金额（无预设比例上限）
- 数据经过清洗和聚合后展示，不暴露原始交易明细
- 具体数据字段待确认（取决于后端可获取数据）

**展示方式**：
- 客户详情页：数字卡片，核心指标大字展示，一屏可览
- 详情页：点击卡片进入图表详情（趋势、分布等）

**设计原则**：不管数据字段是什么，展示模式统一。

### 5.2 模块二：事件推送（核心模块）

#### 5.2.1 事件来源

| 类型 | 示例 |
|------|------|
| 数据行为触发 | 大额爆仓、大额充值、高频交易、连续提现、U卡重复消费 |
| 人工触发 | 客户/客服向 BD 反馈问题（如"系统卡"） |

#### 5.2.2 事件模型

每个事件 = 一个独立线程（参考 Discord 线程模式）。

线程内包含：
- 事件信息及关联数据
- AI 生成的行动建议和话术模板（BD 可手动修改）
- BD 的跟进记录和反馈
- 时间戳记录（事件产生时间、开始处理时间）

#### 5.2.3 规则引擎架构

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│ 规则配置层   │ ──> │  规则引擎     │ ──> │ 建议生成层   │
│ (人 + AI辅助)│     │  (硬逻辑匹配) │     │ (AI 生成)   │
└─────────────┘     └──────────────┘     └─────────────┘
```

- **规则配置层**：运营/管理层定义规则框架，AI 辅助填写结构化参数
- **规则引擎**：代码硬匹配，确定性 100%，负责事件触发和筛选
- **建议生成层**：AI 基于规则框架 + 客户数据 + 训练语料库，生成话术和行动建议

#### 5.2.4 AI 话术策略

- AI 仅为辅助角色，生成建议供参考
- BD 可手动修改话术后再使用
- 前期需要训练语料库并做质量测试
- 话术方向和逻辑框架由人工界定，AI 在框架内生成具体内容

#### 5.2.5 推送策略

- 精选推送，非全量推送
- MVP 阶段：管理层统一配置阈值
- 事件范围由运营控制，防止推送过载
- 未来：升级为 AI 助手对话式配置（BD 可自然语言调整规则）

#### 5.2.6 事件时间记录

不设强制时效/过期机制。系统仅记录：
- 事件产生时间
- BD 开始处理时间

用于后续数据分析，不作为强制约束。

#### 5.2.7 通知渠道

通过 PWA 推送通知触达 BD 手机端。推送策略（频率控制、是否需要备选渠道）作为待评估项。

#### 5.2.8 任务状态

```
待处理 ──> 跟进中 ──> 已完成
                  └──> 已忽略（可选填写原因，不强制）
```

#### 5.2.9 闭环机制

- BD 可选择跟进或忽略
- 所有跟进动作在线程内完成
- 反馈形式：手动填写 + 系统自动记录
- 提醒机制：防止多客户多事件时遗漏

### 5.3 模块三：客户信息档案

**标签体系（双层）**：
- **系统自动标签**：基于排行榜排名每日更新（如"持仓 Top 10"、"合约活跃 Top 50"），Top 200 以内自动打标
- **BD 手动标签和备注**：如"脾气不好"、"喜欢短线"
- 两类标签共存，视觉上明确区分（如自动标签用不同底色）

**其他内容**：
- 跟进记录（与事件线程关联）

**客户交接**：
- 场景：BD 离职/调岗时，客户整体移交给另一个 BD
- 要求：档案信息足够完整，新 BD 可无缝接手

### 5.4 客户分配

- 初始数据从内部系统同步
- 主管在管理后台手动调整
- BD 与客户绑定关系明确

## 6. 数据权限

MVP 阶段：全透明，所有 BD 可见所有客户数据（与现状一致）。

架构预留：数据模型已有 BD-客户绑定关系，未来可通过查询过滤切换至严格隔离模式。

## 7. MVP 范围

### MVP 核心（必须做）

| 模块 | 范围 |
|------|------|
| 事件推送 | 事件线程 + 四态任务管理 + AI 话术模板（可编辑） |
| 规则配置 | 管理后台统一配置 |

### MVP 包含（已有基础可复用）

| 模块 | 范围 |
|------|------|
| 客户数据统计 | 基于现有版本迭代，核心指标卡片 + 详情图表（字段待定） |

### MVP 包含（必要支撑）

| 模块 | 范围 |
|------|------|
| 客户档案 | 标签 + 备注 + 跟进记录 + 交接支持 |
| 通知 | PWA 推送（待评估可靠性） |

### 不包含（后续迭代）

| 能力 | 说明 |
|------|------|
| 快捷操作入口 | 一键发起返点申请、生成代金券等 |
| 参考资料 | 历史福利记录、同类案例 |
| 跨部门派单 | 技术/研发紧急响应工单 |
| 团队协作 | 任务分派 + 进度追踪 |
| 服务效果量化 | 用户数据前后对比 |
| AI 助手配置 | BD 用自然语言调整规则 |

## 8. 待确认事项

| # | 事项 | 说明 |
|---|------|------|
| 1 | 数据字段 | 需确认后端可提供哪些客户数据 |
| 2 | 用户角色 | 三角色划分待验证 |
| 3 | 事件类型穷举 | 需与业务方共同梳理完整事件列表 |
| 4 | 现有系统 | 需确认从哪个内部系统同步客户-BD 绑定关系 |
| 5 | PWA 推送可靠性 | 需测试目标设备（iOS/Android）的实际推送到达率，评估是否需要备选渠道 |
| 6 | AI 语料库 | 需准备训练语料并做话术质量测试 |

## 9. 关键设计约束

1. **记录体验必须极轻**：BD 在外部 IM 聊完后手动回 Sales Hub 记录，操作步骤越少越好
2. **减少无谓打扰**：精选推送，事件范围由运营控制，行动建议必须有数据支撑
3. **移动端优先**：BD 侧所有功能以手机端体验为第一优先级
4. **增加能力而非负担**：Sales Hub 是 BD 的效率工具，不是监控工具；传统渠道保持不变

## 10. 风险项

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| BD 不记录跟进内容 | 系统数据空转，档案无法支撑交接 | 极简记录体验 + 上线后观察使用率再决定是否需要其他方案 |
| PWA 推送不可靠 | 紧急事件无法及时触达 BD | 原型阶段测试到达率，必要时引入 IM bot 备选 |
| AI 话术质量不达标 | 影响客户沟通效果 | BD 可手动修改 + 前期语料库训练和质量测试 |
| 推送过载 | BD 被事件淹没，忽略重要事件 | 运营控制事件范围 + 推送频率策略待设计 |

## 11. 成功标准

MVP 阶段不设硬性量化指标。以 BD 实际使用体验为准：
- BD 是否愿意使用
- BD 是否觉得事件推送有帮助
- BD 的事件处理数据作为后续量化的基础

上线后根据实际数据再定义正式的成功边界。
