# 事件模块设计文档

> 版本：v5.0 · 2026-04-01
> v5.0 变更：确立"事件维度"工作流；主动沟通跟进 → 直接写入事件队列；事件中心 = BD 统一 inbox；新增跟进事件类型、标记完成 Bottom Sheet、客户详情 Row 6
> v4.0 变更：移除系统触发事件（暂缓），仅保留主动沟通；客户详情"事件 Tab"合并为跟进记录时间轴；首页加入"今日任务"
> v3.0 变更：三态模型（未处理/跟进中/已处理），补充完整操作流程

---

## 一、核心设计理念

### 两种工作方式的区别

| 工作方式 | 入口 | 问题 |
|---------|------|------|
| 人的维度 | 客户列表 → 逐个点开 | BD 不知道"今天要做什么"，靠记忆 |
| **事件维度** | 事件中心 → 清 inbox | BD 看到自己全部待办，逐条处理 |

**v5.0 的核心目标：让 BD 的日常工作流从"翻客户"变成"清 inbox"。**

### 什么是事件

**事件（Event）** = 所有需要 BD 处理的单元，每条事件对应一个客户。

事件有两种来源：

| 来源 | 触发方式 | Phase |
|------|---------|-------|
| BD 手动创建 | 主动沟通时勾选"需要跟进" | **Phase 1（当前）** |
| 系统自动推送 | 意图引擎检测大额充提、风控异常 | Phase 2（待接口） |

**设计原则：不区分来源——BD 看到的就是一个队列，来源是属性，不是独立系统。**

---

## 二、事件类型

### Phase 1（当前版本）

| 类型 | 触发方 | BD 需要做什么 |
|------|--------|------------|
| **主动跟进** | BD 手动创建（来自主动沟通） | 执行 next_action，记录结果，决定是否继续跟进 |

### Phase 2（意图引擎上线后）

| 类型 | 触发方 | BD 需要做什么 |
|------|--------|------------|
| 大额充值 | 系统 | 主动联系客户，了解意图，推荐产品 |
| 大额提现 | 系统 | 挽留客户，了解提现原因 |
| 账户问题 | 系统 | 查询原因，协调内部处理 |

## 三、状态流转

```
  主动沟通 + 勾选"需要跟进"
  （Phase 2：意图引擎推送）
              │
              ▼
           待处理
          /       \
    点"跟进"     点"忽略"
        │              │
        ▼              ▼
     跟进中          已忽略
        │
        ▼
      已完成  ←  点"标记完成"（可从待处理直接到）
```

### 状态定义

| 状态 | 含义 | 在哪里可见 |
|------|------|----------|
| 待处理 | 新事件，未认领 | 事件中心"待处理"筛选、首页今日任务（到期项） |
| 跟进中 | BD 已认领，正在进行 | 事件中心"跟进中"筛选 |
| 已完成 | 已结案 | 事件中心"已完成"、客户详情跟进记录时间轴 |
| 已忽略 | BD 判断不需要处理 | 事件中心"全部"可查 |

### 状态规则

- "跟进"的作用：**防止事件被新任务淹没，BD 能找回来**；操作区在待处理和跟进中都可见
- BD 可以不点跟进，直接标记已完成（快速路径）
- 主动跟进类事件**不自动过期**（BD 自己创的，不会因为 48h 没处理就消失）
- Phase 2 系统事件：48h 无操作自动过期（"待处理"状态移出首页，事件中心"全部"仍可查）

## 四、主动沟通 → 创建跟进事件

### 4.1 字段定义

BD 在客户详情页记录主动沟通，字段如下：

| 字段 | 类型 | 填写方式 | 说明 |
|------|------|---------|------|
| `channel` | enum | 手动 | Telegram / WhatsApp / 微信 / 电话 / 线下 / 邮件 |
| `result` | enum | 手动 | 已联系上 / 有回复 / 已见面 / 未接通 / 未回复 / 仅发送消息 |
| `occurred_at` | datetime | 手动（默认当前时间） | 实际联系时间 |
| `summary` | text | 选填 | 简要说明聊了什么（建议 20–120 字） |
| `follow_up` | bool | 手动勾选 | 是否需要后续跟进 |
| `follow_up_at` | date | 手动，勾选后出现 | 预计跟进日期 |
| `follow_up_note` | text | 手动，选填 | 跟进要点，例："确认合约意向" |

**约束：**
- `follow_up_at` 和 `follow_up_note` 仅在 `follow_up = true` 时展示
- `follow_up_at` 不能早于今天
- 同一客户只允许 1 条"活跃跟进"（新建时自动关闭旧的）

### 4.2 提交逻辑

```
BD 填写主动沟通表单
        │
   不勾选"需要跟进"     勾选"需要跟进"
        │                     │
        ▼                     ▼
   写入跟进记录           写入跟进记录
   时间轴归档             +
   流程结束              向事件队列写入一条"主动跟进"事件
                         （status = 待处理，due = follow_up_at）
```

### 4.3 典型场景

**场景 1：一次性沟通（不需要跟进）**

```
04-01  channel: Telegram · result: 未接通
       follow_up: false
→ 记录写入跟进时间轴，不产生事件，流程结束
```

**场景 2：持续跟进**

```
04-01  channel: 电话 · result: 已联系上
       summary: 客户对云算力感兴趣，让发资料
       follow_up: true
       follow_up_at: 04-04
       follow_up_note: 发云算力产品介绍 + 确认是否看过

→ 事件队列写入一条"主动跟进"，due = 04-04

04-04  BD 打开事件中心，看到该条到期
       → 打开事件详情，看到 follow_up_note
       → 发完资料，在事件详情页"添加活动记录"
       → 再次勾选跟进：follow_up_at: 04-07，note: "确认是否看完资料"
       → 或直接标记已完成

（循环，直到客户成交或 BD 主动结案）
```

│  │ 确认合约意向                  │ │

│  （无任务时：今日无待跟进 ✓）     │
├─────────────────────────────────┤
│  ★ 我的收藏                      │
├─────────────────────────────────┤
│  🏆 资沉前五                     │

- 逾期（`follow_up_at` < 今天）标红提醒
- 点击跳转事件详情页

## 六、事件中心（底部导航"事件"页）

**功能定位：BD 的统一 inbox，跨所有客户的待办队列。**

```
┌─────────────────────────────────┐
│  事件中心                        │
├─────────────────────────────────┤
│ [全部] [待处理] [跟进中] [已完成] [已忽略] │
├─────────────────────────────────┤
│  共 8 个事件 · 待处理 5 · 跟进中 2 │
├─────────────────────────────────┤
│  🔵 主动跟进 · 胡军              │  ← BD 创建
│     确认云算力意向  · 待处理 · 明天到期 │
│  ─────────────────────────────  │
│  🔵 主动跟进 · 李强              │
│     回复产品问题  · 跟进中 · 逾期2天 🔴 │
│  ─────────────────────────────  │
│  （Phase 2 系统事件混排于此）     │
│  🔴 大额提现 · 王芳              │
│     $280,000 · 待处理 · 2小时前   │
└─────────────────────────────────┘
```

**排序规则：**
1. 逾期事件置顶（按逾期天数降序）
2. 今日到期其次
3. 其余按 `follow_up_at` 升序（越快要处理的越靠前）

## 八、标记已完成交互

点击"标记已完成"后弹出 Bottom Sheet：

```
┌─────────────────────────────────┐
│  标记完成                   ✕   │
├─────────────────────────────────┤
│                                 │
│  备注（选填）                    │
│  ┌─────────────────────────┐   │
│  │ 例：已发资料，客户表示    │   │
│  │ 需要考虑几天。           │   │
│  └─────────────────────────┘   │
│                                 │
│  [  确认完成  ]  [  推迟 3 天  ] │
└─────────────────────────────────┘
```

**按钮行为：**

| 按钮 | 行为 |
|------|------|
| 确认完成 | `follow_up_done = true`，备注写入 `done_note`，事件进入"已完成" |
| 推迟 3 天 | `follow_up_at += 3天`，状态保持"跟进中"，关闭 sheet |
| ✕ / 外部点击 | 关闭 sheet，不变更状态 |

**"跳过，直接标记"快捷路径：** 在"确认完成"按钮下方可加一行小字链接"跳过备注，直接完成"，一触即关闭+标记，无需输入。

## 九、客户详情集成

### 9.1 Header Card Row 6：活跃跟进提示

当该客户存在 `follow_up_done = false` 的跟进事件时，在 Row 5（上次联系）下方显示 Row 6：

```
待跟进   [明天]  确认云算力意向          [完成 ✓]
```

**颜色规则：**

| 截止距今 | 颜色 |
|---------|------|
| ≥ 3 天 | 蓝色 `#4DA6FF` |
| 1–2 天 | 金色 `#F0B90B` |
| 今天 | 金色 `#F0B90B` + 加粗 |
| 已逾期 | 红色 `#F6465D` |

**`[完成 ✓]` 按钮：** 点击直接弹出标记已完成 Bottom Sheet（同第八节）。

### 9.2 跟进记录时间轴

客户详情"跟进"Tab 的时间轴显示该客户所有活动记录，来源包括：
- 主动沟通记录（每次沟通）
- 跟进事件的活动记录
- （Phase 2）系统事件的处理结果

## 十、数据结构

### 跟进事件（follow_up event）

存于 `sessionStorage`，key = `follow_up_events_{uid}`（每客户独立，支持多条历史）。

```javascript
{
  id: "fu_12345678_1680000000",   // 唯一 ID
  client_uid: "12345678",
  status: "待处理",               // 待处理 / 跟进中 / 已完成 / 已忽略
  created_at: "2026-04-01T10:00:00",
  follow_up_at: "2026-04-04",
  follow_up_note: "发云算力产品介绍 + 确认是否看过",
  activities: [                   // 每次沟通追加
    {
      occurred_at: "2026-04-01T10:00:00",
      channel: "电话",
      result: "已联系上",
      summary: "客户对云算力感兴趣，让发资料"
    }
  ],
  done_note: "",                  // 完成时备注
  done_at: ""                     // 完成时间
}
```

### 主动沟通记录（comm_record）

存于 `sessionStorage`，key = `comm_records_{uid}`，不关联跟进事件（两者独立）。

```javascript
{
  occurred_at: "2026-04-01T10:00:00",
  channel: "电话",
  result: "已联系上",
  summary: "...",
  bd: "BD",
  follow_up: true,           // 是否产生了跟进事件
  follow_up_event_id: "fu_12345678_1680000000"  // 关联事件 ID（如有）
}
```

## 十一、操作路径汇总

### 路径 1：事件中心批量处理（主路径）

```
BD 打开底部导航"事件"
  → 事件中心，默认展示全部未完成
  → 统计行：待处理 5 · 跟进中 2 · 逾期 1
  → 逐条点开 → 处理 → 标记完成
  → 清空 inbox，完成今日工作
```

### 路径 2：首页今日任务直达

```
BD 打开首页
  → "今日任务"模块：今日到期 2 条
  → 点击某条 → 直接进事件详情页
  → 处理完成
```

### 路径 3：客户维度顺带处理

```
BD 查看某客户详情
  → Header Row 6 显示"待跟进 · 明天 · 确认意向"
  → 点"完成 ✓" → Bottom Sheet → 标记完成
  → 或点"查看完整资料" → 事件详情页处理
```

### 路径 4：主动沟通 → 新建跟进

```
BD 在客户详情页
  → 点"+ 记录主动沟通"
  → 填写渠道/结果/摘要
  → 勾选"需要跟进" → 填写截止日期 + 跟进要点
  → 提交 → 写入跟进记录 + 事件队列
```

## 十三、Phase 2 Roadmap

| 功能 | 前置条件 | 验证目标 |
|------|---------|--------|
| 系统触发事件（大额充提、账户异常） | 意图引擎接口上线 | 系统事件是否有增量价值 |
| 逾期任务 Push 通知 | 后端支持 push | 提醒是否有效提升跟进率 |
| 客户列表"跟进中"角标 | Phase 1 稳定 | 列表维度是否有辅助价值 |
| 任务优先级排序 | 任务量 > 5 时 BD 反馈混乱 | 优先级是否提升处理效率 |
