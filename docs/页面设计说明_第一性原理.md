# Sales Hub 页面设计说明 — 从第一性原理出发

## 0. 第一性原理

**资金沉淀就是价值。**

不是"待转化的潜力"，不是"手续费的前置条件"，是价值本身。

原因：资金沉淀带来的收益（平台用资金做市、借贷、资金池等）远大于客户交易产生的手续费。一个客户把 1000 万放在平台上什么都不做，他给平台创造的价值可能比一个每天交易 100 万但不留钱的客户更大。

由此推导出销售工作的本质：

```
销售的工作 = 让更多的钱留在平台上，更久。
```

拆开就是三件事：

1. **守住大盘子** — 已有的大额沉淀不能丢
2. **止住流失** — 正在流出的钱要拦回来
3. **放大增长** — 正在流入的钱要顺势做大

所有页面设计都从这三件事出发。任何不服务于这三件事的元素，不放。

## 1. 首页（home.html）

### 这个页面回答什么

> BD 每天早上打开 App 的第一个问题：**我现在该做什么？**

首页定位：**Next Best Action** — 不是信息展示板，是行动指令台。

### 核心概念：事件 vs 动态

首页只推**事件（Event）**，不推动态（Activity）。区分标准：

> **老板不干预，结果会变坏吗？**

- **事件**：需要老板采取行动的，推到首页
  - 风控触发（账户冻了，必须找研发）
  - 超大额出金（占客户资沉比例很高，可能跑路）
  - 合约爆仓（客户情绪崩溃需安抚）
- **动态**：帮助了解客户的，留在客户详情页时间线
  - 正常出入金（即使金额不小）
  - 理财到期（大部分客户自动续）
  - 交易盈亏（除非特别极端）

### 关注名单机制

**只有关注名单内的客户才会生成首页事件。**

- BD/老板维护前 20-30 名客户（按资沉排序的 TOP 客户）
- 其余客户数据照常追踪，通过搜索或客户 Tab 可达
- 这确保首页不会被低优先级信息淹没，每天大约 2-3 条事件，老板 5 分钟可处理完

### 最终设计：头条模式（1+4+进度条）

首页恒定展示 **6 个元素**：

| 位置 | 元素 | 内容 |
|------|------|------|
| 顶部 | 1 张大卡片 | 最紧急事件：事件类型 + 客户名 + 关键数字 + **客户资沉** + **建议动作** |
| 中部 | 4 行小行 | 次紧急事件：事件类型 + 客户名 + 关键数字 |
| 底部 | 进度条 | 显示"还有 N 条未处理" |

**大卡片比小行多两个信息：** 客户资沉 + 建议动作。这让 BD 对最紧急的事件无需点进去就能行动。

#### 空态

所有事件已处理时，显示：
- "所有事件已处理"
- 推荐跟进：最久未联系的关注名单客户（引导 BD 做主动维护）

#### 满态

仍然 1+4 截断，进度条显示"还有 N 条"。点击进度条进入事件 Tab 查看全部。

### 事件生命周期

事件有过期窗口，不会无限积压：

| 事件类型 | 过期窗口 | 理由 |
|---------|---------|------|
| 风控触发 | 不过期（直到解除） | 账户冻结不会自动好转 |
| 超大额出金 | 48 小时 | 钱已经走了，超过 48h 挽留意义大减 |
| 合约爆仓 | 72 小时 | 客户情绪冷却周期，超过 72h 安抚效果递减 |

过期事件自动沉底，变为"已过期"状态，归档到事件 Tab。

### 设计决策与理由

#### 为什么用头条模式而非事件流

旧设计是无限滚动的事件流。问题：BD 不知道"到底有多少事"，也不知道"处理到哪了"。头条模式用恒定 6 元素 + 进度条，给 BD 明确的"工作量感知"和"完成感"。

#### 为什么 MVP 不用颜色分区

颜色分区需要算法准确率验证（Phase 2）。MVP 用大小梯度（大卡片 vs 小行）区分优先级，视觉层次足够清晰。

#### 搜索栏

保留在首页顶部。BD 心里有个人想看（出门见客户前、接到电话后），搜索栏让他 2 秒内找到，不需要切到客户 Tab。

**这怎么服务第一性原理：** 首页的唯一职责是"把关注名单中最需要行动的事件推到 BD 面前"。头条模式确保 BD 5 分钟处理完当天所有紧急事项，剩下的时间用来做客户 Tab 里的主动维护。

## 2. 客户详情页（client-detail.html）

### 这个页面回答什么

> BD 点进一个客户后的四个问题：
> 1. 这是谁，多重要？
> 2. 他的钱最近怎么了？
> 3. 为什么会这样，我该做什么？
> 4. 之前跟他聊过什么？

### 10 个区块，从上到下

| 区块 | 名称 | 核心问题 |
|------|------|---------|
| A | 客户身份 | 我在看谁（头像、名字、UID、状态、系统标签、BD标签） |
| B | BD 备注 | 我之前记了什么（自由文本，localStorage） |
| C | 资沉指标 | 他的钱怎么了（8周均、本周均、趋势、稳定度） |
| D | 资金分布 | 他主要玩什么（各业务线资沉占比，默认展开） |
| E | 综合净贡献 | 给集团赚了多少（净利润贡献，默认折叠） |
| F | 客户近期盈亏 | 他自己赚了还是亏了（7天 + 8周周均，按业务拆分） |
| G | 福利金 | 我能给他什么（⚠️ 待确认） |
| H | 原因 + 动作 | 系统建议我做什么 |
| I | 最近事件 | 有没有紧急的事 |
| J | 跟进记录 | 之前的互动历史 |

### 设计决策与理由

#### 区块 A：客户身份（头像 + 名字 + 标签）

| 元素 | 为什么放 |
|------|---------|
| 头像 + 名字 | 确认"我在看谁"。人类处理面孔和名字比处理 UID 快。 |
| 状态 badge（流失/增长/稳定） | 状态是整个页面的**色调锚点**——流失客户页面偏红，增长客户偏绿。 |
| 系统标签 | 系统自动打的标签，BD 不可编辑。代表"平台认为这个客户是什么"。 |
| BD 标签（可编辑） | BD 自己打的标签，如"偏好短线"、"英文沟通"。这是 BD 的**私人记忆外挂**。 |

#### 区块 B：BD 备注（紧跟身份）

自由文本 textarea，输入即保存（localStorage）。BD 确认"我在看谁"后，立刻看到自己之前的笔记。不需要格式化模板。

#### 区块 C：资沉核心指标（页面视觉中心）

**当前资沉是整个页面最大的数字。** 第一性原理说"资沉就是价值"，那页面上最大的数字就应该是资沉。

| 指标 | 为什么放 |
|------|---------|
| 当前资沉 (8 周均) | 回答"这个客户有多重要" |
| 本周均值 | 回答"最近一周跟 8 周均值有没有偏离" |
| 近期趋势 (%/周) | 回答"在涨还是在跌，多快"（8 周线性回归） |
| 趋势稳定度 | 回答"这个趋势靠不靠谱"（R² ≥ 0.7 = 明确） |

#### 区块 D：资金分布（默认展开）

各业务线资沉占比（理财、合约保证金、现货钱包、金融卡、云算力），金额 + 百分比 + 进度条。BD 一眼看出"这人主要玩什么"。

#### 区块 E：综合净贡献（默认折叠）

集团净利润贡献总额 + 各模块明细（收入绿色、成本红色）。默认只显示总额，点击展开。大多数时候 BD 只看 D 就够，E 是深入分析时才展开。

#### 区块 F：客户近期盈亏

**两个时间窗口：** 近 7 天盈亏（当前情绪）+ 8 周周均盈亏（长期趋势），按业务拆分明细。BD 结合两者判断客户情绪和沟通策略。

#### 区块 G：福利金（⚠️ 待确认）

福利金总额度、已发放、已激活、发券按钮。待业务确认规则后落地。

#### 区块 H：原因 + 建议动作

**合并在一张卡片里。** 原因和动作是因果关系——"因为理财到期了，所以建议你去回访"。只给一个动作，不给选择题。

#### 区块 I：最近事件

只展示 2-3 条最近的事件，点击跳转事件线程。事件颜色跟客户状态联动。

#### 区块 J：跟进记录

时间线样式，最新在最上，区分人工记录和系统记录。

**这怎么服务第一性原理：** 10 个区块严格按 BD 的信息消费顺序排列——先确认身份，再看备注，再看资沉，再看分布和贡献，再看盈亏和福利金，最后看行动建议和历史。每个区块都围绕"资沉"这个唯一北极星。

## 3. 客户列表（客户 Tab）= 望远镜

### 这个页面回答什么

> 我应该主动维护谁的关系？谁最久没联系了？

客户 Tab 的定位是**望远镜** — 看重要的、关系驱动的事，不是看紧急的。

### 最终设计：按"最久未联系"排序的客户列表

默认排序：**最久未联系** 排在最前面。这回答"我该主动找谁聊聊"。

#### 每行展示三项信息

| 内容项 | 说明 |
|--------|------|
| 客户名 + UID | 确认是谁 |
| 资沉金额 | 这个人多重要 |
| 最近联系时间 | 多久没联系了 |

#### 搜索：按名字或 UID

BD 记得名字的搜名字，从系统跳过来的用 UID。两种找人方式覆盖所有场景。

#### 与首页的分工

- 首页 = 显微镜：看紧急的、事件驱动的（关注名单的紧急事件）
- 客户 Tab = 望远镜：看重要的、关系驱动的（按最久未联系排序）

BD 处理完首页事件后，来客户 Tab 做主动维护。

**这怎么服务第一性原理：** 资沉流失往往从"失联"开始。按最久未联系排序，把即将失联的客户推到 BD 面前，预防性维护比事后挽留成本低得多。

## 4. 事件中心（events.html）= 档案馆

### 这个页面回答什么

> 历史上发生了什么事件？我需要回顾或筛选某类事件。

事件 Tab 的定位是**档案馆** — 历史归档 + 筛选检索。

### 与首页的关系

首页只展示当前有效的、关注名单内的紧急事件（头条模式 1+4）。事件 Tab 则保留**所有事件的完整历史**，包括：
- 已处理的事件
- 已过期的事件
- 非关注名单客户的事件
- 动态类信息（降级为动态后的记录）

### 功能

- **筛选**：按事件类型、客户、时间范围、状态（待处理/已处理/已过期/已忽略）
- **搜索**：按客户名或事件关键词
- **排序**：按时间倒序（档案馆逻辑，不是紧急度）

### 设计决策

#### 为什么事件 Tab 不再是主要工作台

旧设计中事件中心承担了"今天该做什么"的职责。新设计中这个职责完全移交给首页（头条模式）。事件 Tab 退化为纯归档，BD 日常工作中很少需要主动打开它。

**这怎么服务第一性原理：** 档案馆确保没有信息丢失。BD 需要回顾某个客户的事件历史时（如准备见面前做功课），这里是唯一的完整记录。

## 5. 事件线程（event-thread.html）

### 这个页面回答什么

> 这个事件到底怎么回事，我该怎么处理，进展到哪了？

### 5 个区块结构

| 区块 | 名称 | 内容 |
|------|------|------|
| A | 事件头 | 类型 + 客户名 + 关键数字 + 时间 |
| B | 客户摘要 | 资沉、趋势、BD标签、上次跟进 |
| C | 操作区 | 行动建议 + 话术模板 + 福利金发放 |
| D | 任务状态 | 待处理 / 跟进中 / 已完成 / 已忽略 |
| E | 跟进记录 | 针对该事件的互动历史 |

### 设计决策与理由

#### 区块 A：事件头

```
资沉流失预警 · 张伟
-$150 万 · 2小时前
```

核心指标是**资沉变化金额**，不是交易额，不是盈亏。这个数字直接告诉 BD"我们正在损失多少钱"。

#### 区块 B：客户摘要

BD 不需要跳到详情页就能获得关键上下文：当前资沉、趋势状态、BD 标签（如"偏好短线"）、上次跟进时间。3 秒内建立对这个客户的认知。

#### 区块 C：操作区（行动建议 + 话术模板 + 福利金）

**三合一操作区：**
- **行动建议：** 系统推荐的一个动作（如"安排回访"）
- **话术模板：** 对应场景的沟通话术，BD 可以直接复制发给客户
- **福利金发放：** 如果场景适合，直接在这里给客户发券（⚠️ 待确认）

把"建议做什么"、"怎么说"、"能给什么"放在一起，BD 不需要来回切换。

#### 区块 D：任务状态

四个状态：待处理 / 跟进中 / 已完成 / 已忽略。

"已忽略"很重要——它给了 BD 一个**合法的关闭理由**。不是所有事件都值得跟进。

#### 区块 E：跟进记录

针对该事件的互动历史，时间线样式。BD 可以添加新记录。

**这怎么服务第一性原理：** 事件线程是"发现资沉变化 → 获得客户上下文 → 采取行动（建议+话术+福利金）→ 确认结果"的完整闭环。没有这个闭环，事件就只是通知——看完就忘了。

## 6. 个人页面（profile.html）= 设置

### 这个页面回答什么

> 我的设置和基本信息。

### Tab 架构总览

| Tab | 隐喻 | 定位 | 核心逻辑 |
|-----|------|------|---------|
| 首页 | 显微镜 | 紧急，事件驱动 | 关注名单的紧急事件，Next Best Action |
| 客户 | 望远镜 | 重要，关系驱动 | 按最久未联系排序，主动维护 |
| 事件 | 档案馆 | 历史归档 + 筛选检索 | 所有事件完整记录 |
| 我的 | 设置 | 个人信息和偏好 | 关注名单管理、通知设置 |

### 设计决策与理由

MVP 阶段重点内容：
- **关注名单管理**：维护 TOP 20-30 客户列表（按资沉排序推荐，BD 可手动调整）
- 基本个人信息

未来可能放的内容（Phase 2+）：
- BD 的客户盘总资沉概览
- 个人 KPI（管理的总资沉、流失率、增长率）
- 通知偏好、语言设置

## 7. 设计原则总结

### 原则 1：资沉是唯一的北极星

每个页面的核心数字、排序标准、事件触发条件、动作建议，都指向同一个量：资金沉淀。不混入交易量、净贡献、效率等旧世界的指标。

### 原则 2：事件推行动，动态供了解

推到首页的必须是"不干预会变坏"的事件，其余都是动态。这条原则防止首页被信息噪音淹没。

### 原则 3：先看盘，再看变化，再行动

页面的信息层次严格遵循这个顺序。首页看"现在最该做什么"（头条模式），客户 Tab 看"谁最久没联系"（望远镜），详情页看资沉数字、趋势、原因和动作。

### 原则 4：给结论，不给数据

BD 不需要知道"R² = 0.82"，BD 需要知道"趋势明确"。BD 不需要知道"AdjustedTrendRate = -0.032"，BD 需要知道"每周跌 3.2%"。所有后台算法对前台不可见，前台只展示人话。

### 原则 5：一个动作，不给选择

每个客户只给一个建议动作，每个事件只给一套建议方案。减少决策负担，增加执行速度。

### 原则 6：宁可多看一个没事的人，不要漏掉一个在跑的人

趋势可信度（Conf）的下限设为 0.5，意味着即使趋势不确定的客户也不会被完全过滤掉。对销售来说，false negative（漏掉真正在流失的客户）的成本远大于 false positive（多看了一个没事的客户）。

### 原则 7：透明优先于精确

资沉总量榜只按金额排序，不掺任何因子。BD 一看就懂"为什么这个人排第一"。黑箱排序在数学上可能更优，但在销售信任上一定更差。
