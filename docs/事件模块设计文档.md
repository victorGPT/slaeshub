# 事件模块设计文档

> 版本：v4.0 · 2026-04-01
> v4.0 变更：移除系统触发事件，仅保留主动沟通；客户详情页"事件Tab"合并为"跟进记录Tab"单条时间轴；首页加入"今日任务"模块
> v3.0 变更：加入三态模型（未处理/跟进中/已处理），补充完整操作流程
> v2.0 变更：CEO Review 后收敛 MVP 范围，简化状态模型

---

## 一、概念定义

**事件（Event）** 是系统或 BD 主动发起的、需要跟进处理的客户相关任务。每条事件对应一个客户。

**设计原则：** 不区分"系统事件"和"BD 任务"——从 BD 视角看，后续动作一样（打开 → 操作 → 关闭）。"谁触发的"是属性，不是独立系统。

---

## 二、事件类型

### MVP（Phase 1）

| 事件类型 | 触发方 | BD 需要做什么 | 工具包（操作区） |
|----------|--------|-------------|-----------------|
| 大额充值 | 系统（意图引擎） | 主动联系客户，了解意图，推荐产品 | 话术模板 + 复制话术 |
| 大额提现 | 系统（意图引擎） | 挽留客户，了解提现原因 | 话术模板 + 复制话术 |
| 账户问题 | 系统 | 查询原因，协调内部处理 | 复制 UID |

**备注：**
- 触发条件由意图引擎定义，我们只接收信号并展示给 BD。
- 账户问题类话术模板后续补充。
- 礼金功能本版本不做，后续版本再加。
- 合约/杠杆爆仓不单独做事件类型，盈亏数据已在资产 Tab 展示。

### Phase 2

| 类型 | 触发方 | 说明 |
|------|--------|------|
| 主动维护 | BD 手动 | BD 主动发起，如送礼金维护关系 |
| 其他 | BD 手动 | 自定义，如客户拜访计划 |

手动创建入口：客户详情页（BD 必须先看完客户档案才能创建）。

## 三、状态流转

```
           系统触发（意图引擎 / 风控）
                     │
                     ▼
                  未处理
               /    |    \
         点"跟进"   |   点"忽略"
             │      |      │
             ▼      |      ▼
          跟进中    |    已忽略
             │      │
             ▼      ▼
           已处理 ← 直接标记

         48h 无操作（未处理状态）→ 已过期
```

### 状态定义

| 状态 | 含义 | 在哪里看得到 |
|------|------|-------------|
| 未处理 | 新事件，还没人碰 | 首页英雄卡/小卡片、事件中心"未处理"筛选 |
| 跟进中 | BD 已认领，正在处理 | 首页摘要栏"跟进中 N"、事件中心"跟进中"筛选 |
| 已处理 | 已结案 | 事件中心"已处理"筛选、客户详情时间线 |
| 已忽略 | BD 主动判断不需要处理 | 事件中心"全部"可查 |
| 已过期 | 48h 无操作，系统自动清理 | 事件中心"全部"可查 |

### 规则

- "跟进"不是门禁——操作区（话术、礼金、复制 UID）在未处理和跟进中**都可见**
- "跟进"只做一件事：**防止事件被新事件顶掉，确保 BD 能找回来**
- BD 也可以不点跟进，直接标记已处理（快速路径）
- 已处理时可选填一句结束说明（非必填）

### 过期机制

**所有事件类型统一 48h 过期。** 保证 BD 打开页面看到的都是"今天需要操作的事"，过了时效的不占位置。

| 规则 | 说明 |
|------|------|
| 过期窗口 | 触发后 48h 无操作（未处理状态） |
| 跟进中是否过期 | **不过期**——BD 已认领，说明在处理 |
| 过期后去哪 | 从首页、"未处理"列表消失，事件中心"全部"仍可查 |
| 能否恢复 | 不需要——过了时效，行动价值已大减 |

## 四、入口总览与操作路径

### 4.1 入口总览

| 入口 | 位置 | 场景 |
|------|------|------|
| A. 推送入口 | 首页英雄卡 + 小卡片 | 系统把最紧急事件推到面前 |
| B. 摘要入口 | 首页摘要栏"跟进中 N" | 找回正在处理的事件 |
| C. 任务清单 | 底部导航"事件" → 事件中心 | 主动批量消化待办队列 |
| D. 客户维度 | 客户详情页 → 事件 Tab | 查看某客户时顺带处理 |

### 4.2 操作流程

#### 流程 1：快速处理（不需要跟进的简单事件）

```
BD 打开首页
  → 看到英雄卡片：大额充值 · 张伟 · $50,000
  → 点击进入事件详情页
  → 复制话术，发给客户
  → 点"标记已处理"（可选填备注）
  → 返回首页，进度条 +1
```

**全程 1 分钟，不需要点"跟进"。**

#### 流程 2：需要时间处理（中间会被打断）

```
BD 早上 9:00 打开首页
  → 看到英雄卡片：大额提现 · 李强 · $280,000
  → 点击进入事件详情页
  → 点"跟进"（事件变为跟进中，不会被新事件顶掉）
  → 复制话术，打电话给客户
  → 客户说晚点回复，BD 先去忙别的

BD 上午 11:00 回来
  → 首页摘要栏显示：跟进中 1
  → 点击 → 跳转事件中心，自动筛选"跟进中"
  → 找到李强的事件，点进去
  → 客户已回复确认，点"标记已处理"
  → 填写备注："客户确认是购房用途，正常提现。"
  → 完成
```

**"跟进"解决的是"找回来"的问题。**

#### 流程 3：从事件中心批量处理

```
BD 打开底部导航"事件"
  → 事件中心列表
  → 筛选 pills：[ 未处理 ] [ 跟进中 ] [ 已处理 ]（默认不选 = 显示全部）
  → 统计行：共 12 个事件 · 未处理 5 · 跟进中 2
  → 逐条点开处理
```

#### 流程 4：从客户详情页处理

```
BD 在客户列表点击"李强"
  → 客户详情页（事件 Tab 默认展开）
  → 当前事件区块：显示未处理 + 跟进中的事件
  → 点击某条事件 → 进入事件详情页处理
  → 处理完返回客户详情页，事件区块刷新
```

```
│  │ 发产品介绍文档               │ │
│  └────────────────────────────┘ │
│  （无任务时：今日无待跟进 ✓）    │
├─────────────────────────────────┤
│  ★ 我的收藏                     │
├─────────────────────────────────┤
│  🏆 资沉前五                    │
├─────────────────────────────────┤
│  ↓ 本周入金前五                 │
├─────────────────────────────────┤

├─────────────────────────────────┤
```

**今日任务逻辑：**
- 数据来源：所有 `follow_up_at` = 今天的主动沟通记录（从 localStorage 读取）

- 逾期未处理的任务（`follow_up_at` < 今天）也显示，标红提醒

### 6.3 主动沟通

BD 手动创建的联系记录，不由意图引擎推送，用于记录主动外呼/发消息等行为。

**字段定义：**

| 字段 | 类型 | 填写方式 | 说明 |
|------|------|----------|------|
| event_type | enum | 自动 | 固定值：`主动联系` |
| client_id | relation | 自动 | 当前客户 |
| occurred_at | datetime | 手动（默认当前时间） | 实际联系时间，不是创建时间 |
| channel | enum | 手动 | `Telegram` / `WhatsApp` / `微信` / `电话` / `线下` / `邮件` |
| result | enum | 手动 | `已联系上` / `有回复` / `已见面` / `未接通` / `未回复` / `仅发送消息` |
| summary | text | 手动（选填） | 简要说明聊了什么，建议 20–120 字 |
| next_action | text | 手动（选填） | 下一步要做什么，例："发云算力产品介绍 + 3天后回访确认" |
| follow_up_at | date | 手动（选填） | 跟进提醒日期；填写后系统在当天将该客户推送至首页 |
| owner_id | relation | 自动 | 创建人（当前登录 BD） |
| created_at | datetime | 自动 | 记录创建时间 |

**展示文本（用于时间线）：** `{channel} · {result}`，若 summary 非空则追加 `：{summary}`。

**示例：** `Telegram · 已联系上：询问了近期操作计划，客户表示暂无大额动作。`

---

### 6.4 主动沟通闭环流程

**核心设计原则：** 每次沟通必须记录两件事——**本次结果** + **下一步动作**。没有下一步，沟通记录就是黑洞。

#### 完整闭环

```
BD 主动联系客户
      │
      ▼
填写本次沟通记录
  · channel（渠道）
  · result（结果）
  · summary（聊了什么，选填）
      │
      ▼
需要跟进吗？
  │           │
  是           否
  │           │
  ▼           ▼
填写 next_action    保存，结束
填写 follow_up_at
      │
      ▼
系统在 follow_up_at 当天
将该客户推送至首页
（标记为"待跟进"）
      │
      ▼
BD 看到提醒，打开客户详情
  · 查看上次记录的 next_action
  · 执行，再次填写新的沟通记录
      │
      ▼
（循环，直到不再需要跟进）
```

#### 典型场景：客户表示对产品感兴趣

```
第一次联系（04-01）
  result    ：已联系上
  summary   ：客户对云算力产品感兴趣，让我发资料给他
  next_action：发云算力产品介绍文档
  follow_up_at：04-04（3天后）

→ 04-04 系统推送提醒至首页
→ BD 打开，看到 next_action："发云算力产品介绍文档"
→ BD 发完资料，记录第二次沟通：

第二次联系（04-04）
  result    ：仅发送消息
  summary   ：已发产品资料，等待客户反馈
  next_action：确认客户是否看过资料，了解意向
  follow_up_at：04-07

→ 04-07 系统再次推送提醒
→ 循环直到客户成交或明确放弃
```

#### 无需跟进的场景（一次性处理）

```
result    ：未接通
summary   ：（空）
next_action：（空）
follow_up_at：（空）

→ 记录保存，不产生提醒，流程结束
```

**关键约束：**
- `next_action` 和 `follow_up_at` 必须同时填或同时不填
- `follow_up_at` 不能早于今天
- 同一客户同一天只产生一条"待跟进"提醒（避免重复推送）

## 七、事件详情页结构

### 6.1 页面布局（三段式）

```
┌─────────────────────────────────┐
│  ← 返回    事件详情             │
│                                 │
│ ┄┄┄┄┄┄┄ 第一段：客户信息 ┄┄┄┄┄┄ │
│                                 │
│  ★ 12345678              📋    │  ← 收藏 + UID + 复制
│                                 │
│  7日资沉                        │
│  $320万         ▼ -12.3%       │  ← 资沉 + 变化率
│                                 │
│  备忘：老客户，做合约为主，      │  ← 备忘（可编辑）
│  上次聊过想买房                  │
│                                 │
│  [高净值] [合约偏好] [稳定]      │  ← 标签（Phase 2: Tag 系统）
│                                 │
│  查看完整资料 →                  │
│                                 │
│ ┄┄┄┄┄┄ 第二段：事件卡片 ┄┄┄┄┄┄ │
│                                 │
│  🔴 大额提现  ·  2小时前        │  ← 事件类型 + 时间
│  $280,000                       │  ← 触发金额
│                                 │
│ ┄┄┄┄┄ 第三段：应对工具包 ┄┄┄┄┄ │
│                                 │
│  ── 大额充提类 ──               │
│  建议行动                        │
│  建议联系客户确认提现意图...     │
│                                 │
│  话术参考（可折叠）              │
│  "您好，注意到您近期有一笔大额   │
│   操作，想了解一下是否一切顺利..." │
│  [复制话术]                     │
│                                 │
│  ── 账户问题类 ──               │
│  建议行动                        │
│  查询封禁原因，协调内部处理      │
│  [复制 UID]                     │
│                                 │
├─────────────────────────────────┤
│  [跟进]   [忽略]   [✓ 标记已处理] │  ← 底部固定操作栏
└─────────────────────────────────┘
```

**设计说明：**
- 第一段 = 客户详情页固定头部的完整复用（UID、资沉、变化率、备忘、标签）
- 第二段 = 本次事件的核心信息（类型、金额、时间）
- 第三段 = BD 应对工具（建议行动 + 话术参考 + 操作按钮）
- BD 看完客户信息后，结合备忘和标签，自行判断怎么沟通
- 话术模板降级为"参考"，可折叠，不是主要操作

### 6.2 第一段字段补充：上次联系

**字段来源：** 取该客户最近一条 `主动沟通` 记录（按 `occurred_at` 倒序第一条）。

**展示位置：** 标签行之后、"查看完整资料"链接之前。

**展示格式：**

```
上次联系        3天前   Telegram · 已联系上
               ↑时间   ↑channel  ↑result
```

**时间显示规则（`occurred_at` 距当前时刻）：**

| 距离现在 | 显示 | 示例 |
|---------|------|------|
| < 1小时 | 刚刚 | 刚刚 |
| 1–23小时 | X小时前 | 3小时前 |
| 1–6天 | X天前 | 3天前 |
| 7–29天 | X周前 | 2周前 |
| ≥ 30天 | M月D日 | 3月15日 |

> **实现说明：** `occurred_at` 存储 ISO 格式时间戳，前端在渲染时实时计算差值，不存储"刚刚"等文字字符串。存文字会导致时间永远不更新。

**无记录时：** 显示"暂无记录"（灰色 `#474D57`）。

### 6.2 跟进中状态

```
┌─────────────────────────────────┐
│  ← 返回    事件详情    跟进中   │
├─────────────────────────────────┤
│  （同上三段式布局）              │
├─────────────────────────────────┤
│            [✓ 标记已处理]        │
└─────────────────────────────────┘
```

与未处理的区别：
- 顶部显示"跟进中"标签
- 底部按钮只剩"标记已处理"（跟进和忽略消失）
- 三段内容完全一致

### 6.3 已处理状态（只读）

## 七、事件中心结构

```
┌─────────────────────────────────┐
│  ← 返回    事件中心             │
├─────────────────────────────────┤
```

## 八、主动事件结构

### 8.1 设计决策

主动事件 = BD 主动发起并持续跟进的一件事，有明确的开始和结束。

**核心概念：**
- 一个事件对应一件正在跟进的事（有标题）
- 每次拜访/沟通 → 在该事件下追加一条活动记录
- 多次沟通归属同一个事件，不是分散的独立记录
- 状态"跟进中"= 这件事未完结，"已完成"= 彻底结案

### 8.2 事件数据结构

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | 事件唯一 ID |
| `title` | text | 事件标题，例："拜访胡军 - 云算力意向跟进" |
| `client_id` | relation | 关联客户 |
| `status` | enum | `跟进中` / `已完成` |
| `created_at` | datetime | 创建时间 |
| `next_action` | text | 下一步动作（选填） |
| `follow_up_at` | date | 跟进提醒日期（选填） |
| `activities` | array | 活动记录列表（见下） |

**活动记录（activity）字段：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `occurred_at` | datetime | 本次沟通时间 |
| `channel` | enum | Telegram / WhatsApp / 微信 / 电话 / 线下 / 邮件 |
| `result` | enum | 已联系上 / 有回复 / 已见面 / 未接通 / 未回复 / 仅发送消息 |
| `summary` | text | 本次沟通内容（选填） |

### 8.3 事件卡片视图（事件列表/详情页）

只展示**当前事件**自己的活动记录：

```
┌─────────────────────────────────┐
│  拜访胡军 - 云算力意向跟进  [跟进中] │
│  下一步：04-10 发正式报价         │
├─────────────────────────────────┤
│  活动记录                        │
│  ● 04-07  电话 · 已联系上        │
│           客户确认有意向，需报价  │
│  ● 04-04  线下 · 已见面          │
│           首次拜访，客户感兴趣    │
├─────────────────────────────────┤
│  [+ 添加活动记录]  [标记已完成]   │
└─────────────────────────────────┘
```

### 8.4 客户详情页 · 跟进记录 Tab

展示该客户**所有事件**的活动记录，全量时间轴，用 Tag 标注归属：

```
[档案]  [跟进记录]  [资沉]

● 04-07  [拜访 - 云算力]  电话 · 已联系上
         客户确认有意向，需报价
● 04-05  [产品推荐]  Telegram · 有回复
         客户对理财产品感兴趣
● 04-04  [拜访 - 云算力]  线下 · 已见面
         首次拜访，客户感兴趣
● 03-28  [账户问题]  电话 · 已联系上
         协助解封，已处理
```

**Tag** = 事件标题的缩写（取前 6 字），点击可跳转到对应事件详情。

### 8.5 数据关系

```
事件（Event）
  └── 活动记录（Activity）× N

事件卡片  →  按事件 ID 过滤活动记录
客户详情  →  该客户所有事件的活动记录合并，按时间排序
```

底层同一份数据，两个视图用不同维度展示。
