# 产品架构：事件与关注名单

> 2026-03 圆桌讨论结论。本文档是产品设计的权威参考，覆盖事件系统、关注名单机制、首页设计和 Tab 架构。

---

## 一、事件 vs 动态

### 定义

| 概念 | 定义 | 去向 |
|------|------|------|
| **事件（Event）** | 需要老板采取行动的信息 | 推到首页 |
| **动态（Activity）** | 帮助了解客户的信息 | 留在客户详情页时间线 |

### 判断标准

**核心问题：老板不干预，结果会变坏吗？**

- 会 → 事件
- 不会 → 动态

### 具体举例

**事件（推首页）：**

| 事件 | 为什么是事件 |
|------|-------------|
| 风控触发（账户冻结） | 账户冻了不会自动解冻，必须找研发处理 |
| 超大额出金（占资沉比例高） | 可能是跑路信号，48 小时内挽留有效 |
| 合约/杠杆爆仓 | 客户情绪崩溃，72 小时内安抚有效 |

**动态（留详情页）：**

| 动态 | 为什么是动态 |
|------|-------------|
| 正常出入金（即使金额不小） | 客户自由操作，不需要干预 |
| 理财到期 | 大部分客户自动续，不干预也不会变坏 |
| 交易盈亏（除非极端） | 日常波动，不需要每次都行动 |
| 资沉趋势变化 | 慢变量，客户 Tab 按最久未联系排序已覆盖 |
| 大额入金/盈利 | 好事，不需要紧急行动 |

## 二、关注名单机制

### 核心规则

**只有关注名单内的客户才会生成首页事件。**

### 机制设计

| 项 | 说明 |
|----|------|
| 名单容量 | 前 20-30 名客户 |
| 排序依据 | 按资沉排序的 TOP 客户 |
| 维护者 | BD/老板手动维护（系统按资沉推荐，可手动调整） |
| 管理入口 | 「我的」Tab 中的关注名单管理 |

### 非关注名单客户

- 数据照常追踪，不丢失
- 通过搜索可达
- 通过客户 Tab 可达
- 事件仍然生成，但只出现在事件 Tab（档案馆）和客户详情页，不推首页

### 设计理由

一个 BD 管 200+ 客户，但真正需要贴身服务的是前 20-30 个大客户。关注名单机制确保：
1. 首页不被低优先级信息淹没
2. 每天大约 2-3 条事件，老板 5 分钟可处理完
3. BD 精力集中在最有价值的客户上

## 三、事件生命周期

### 状态流转

```
触发 → 待处理 → 跟进中 → 已完成
                        ↘ 已忽略
        ↓（超过过期窗口）
      已过期（自动沉底）
```

### 各类型过期窗口

| 事件类型 | 过期窗口 | 理由 |
|---------|---------|------|
| 风控触发 | **不过期**（直到解除） | 账户冻结是持续性问题，不会自动好转 |
| 超大额出金 | **48 小时** | 钱已经走了，超过 48h 挽留意义大减 |
| 合约/杠杆爆仓 | **72 小时** | 客户情绪冷却周期，超过 72h 安抚效果递减 |

### 过期处理

- 过期事件自动沉底，变为"已过期"状态
- 不从系统中删除，归档到事件 Tab（档案馆）
- BD 仍可在事件 Tab 中查看和筛选

## 四、首页设计方案：头条模式

### 布局：6 元素恒定

| 位置 | 元素 | 展示内容 |
|------|------|---------|
| 顶部 | 搜索栏 | 按客户名或 UID 搜索 |
| 主体上 | **1 张大卡片** | 最紧急事件：事件类型 + 客户名 + 关键数字 + 客户资沉 + 建议动作 |
| 主体中 | **4 行小行** | 次紧急事件：事件类型 + 客户名 + 关键数字 |
| 主体下 | **进度条** | "还有 N 条未处理"，点击进入事件 Tab |

### 大卡片 vs 小行

大卡片比小行多两个信息：

| 信息 | 大卡片 | 小行 |
|------|--------|------|
| 事件类型 | 有 | 有 |
| 客户名 | 有 | 有 |
| 关键数字 | 有 | 有 |
| 客户资沉 | **有** | 无 |
| 建议动作 | **有** | 无 |

### 空态

当所有事件已处理时：

- 主文案："所有事件已处理"
- 推荐跟进：最久未联系的关注名单客户
- 引导 BD 从"被动响应"切换到"主动维护"

### 满态

- 仍然 1+4 截断，不做无限滚动
- 进度条显示"还有 N 条"
- 点击进度条进入事件 Tab 查看全部

### 视觉层次

- MVP 用**大小梯度**（大卡片 vs 小行）区分优先级
- **不用颜色分区**（Phase 2，需要算法准确率验证后再引入）

## 五、Tab 架构

### 四 Tab 结构

| Tab | 隐喻 | 定位 | 核心逻辑 | 使用频率 |
|-----|------|------|---------|---------|
| **首页** | 显微镜 | 紧急，事件驱动 | 关注名单的紧急事件，Next Best Action | 每天必看 |
| **客户** | 望远镜 | 重要，关系驱动 | 按"最久未联系"排序，主动维护 | 每天看 |
| **事件** | 档案馆 | 历史归档 + 筛选检索 | 所有事件完整记录 | 偶尔查 |
| **我的** | 设置 | 个人信息和偏好 | 关注名单管理、通知设置 | 很少用 |

### 各 Tab 详细说明

#### 首页 = 显微镜

- 回答："我现在该做什么？"
- 头条模式 1+4+进度条
- 只展示关注名单内、未过期的事件
- 空态推荐最久未联系的关注名单客户

#### 客户 Tab = 望远镜

- 回答："我应该主动维护谁的关系？"
- 默认按"最久未联系"排序
- 所有客户可达（不限关注名单）
- 搜索按名字或 UID

#### 事件 Tab = 档案馆

- 回答："历史上发生了什么？"
- 保留所有事件完整记录（包括已处理、已过期、非关注名单）
- 支持筛选：按事件类型、客户、时间范围、状态
- 按时间倒序排列

#### 我的 = 设置

- 关注名单管理（MVP 核心功能）
- 个人基本信息
- 通知偏好（Phase 2）

## 六、MVP 范围与 Phase 2 规划

### MVP（当前阶段）

**做：**
- 3 类首页事件：风控触发、超大额出金、合约/杠杆爆仓
- 关注名单机制（TOP 20-30 客户）
- 头条模式首页（1+4+进度条）
- 大小梯度视觉层次
- 事件生命周期（过期窗口）
- 四 Tab 架构

**不做：**
- 算法推荐（哪些事件该推、推给谁）
- 颜色分区（红/金/绿/灰优先级标识）
- 智能排序（超过紧急度的多因子排序）
- 动态的自动生成和详情页时间线展示

### Phase 2（需要数据验证后）

| 功能 | 前提条件 |
|------|---------|
| 颜色分区 | 算法准确率验证，事件优先级分类准确率 ≥ 85% |
| 算法推荐 | 积累足够的 BD 行为数据（哪些事件被处理了、忽略了） |
| 更多事件类型 | 验证 MVP 3 类事件的处理率和有效性 |
| 智能排序 | 多因子排序模型训练完成 |
| 个性化通知 | 理解不同 BD 的工作习惯差异 |

### 关键指标

MVP 成功的衡量标准：
- 每天推送 2-3 条事件到首页
- BD 5 分钟内可处理完所有首页事件
- 事件处理率 ≥ 80%（非忽略）
- BD 每天打开首页的频率 ≥ 1 次
