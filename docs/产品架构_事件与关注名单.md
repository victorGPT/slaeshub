# 产品架构：事件与关注名单

> 2026-03 圆桌讨论结论。本文档是产品设计的权威参考，覆盖事件系统、关注名单机制、首页设计、Tab 架构和客户详情页结构。

---

## 一、事件 vs 动态

### 定义

| 概念 | 定义 | 去向 |
|------|------|------|
| **事件（Event）** | 需要老板采取行动的信息 | 推到首页 |
| **动态（Activity）** | 帮助了解客户的信息 | 留在客户详情页时间线 |

### 判断标准

**核心问题：老板不干预，结果会变坏吗？**

- 会 → 事件
- 不会 → 动态

### 具体举例

**事件（推首页，仅 3 类）：**

| 事件 | 为什么是事件 |
|------|-------------|
| 风控触发（账户冻结） | 账户冻了不会自动解冻，必须找研发处理 |
| 超大额出金（占资沉比例高） | 可能是跑路信号，48 小时内挽留有效 |
| 合约/杠杆爆仓 | 客户情绪崩溃，72 小时内安抚有效 |

**动态（留详情页）：**

| 动态 | 为什么是动态 |
|------|-------------|
| 正常出入金（即使金额不小） | 客户自由操作，不需要干预 |
| 理财到期 | 大部分客户自动续，不干预也不会变坏 |
| 交易盈亏（除非极端） | 日常波动，不需要每次都行动 |
| 资沉趋势变化 | 慢变量，客户 Tab 按最久未联系排序已覆盖 |
| 大额入金/盈利 | 好事，不需要紧急行动 |

## 二、关注名单机制

### 核心规则

**只有关注名单内的客户才会生成首页事件。**

### 机制设计

| 项 | 说明 |
|----|------|
| 名单容量 | TOP 30 客户 |
| 排序依据 | 按资沉（fMean）排序 |
| 维护者 | BD/老板手动维护（系统按资沉推荐，可手动调整） |
| 管理入口 | 「我的」Tab 中的关注名单管理 |

### 非关注名单客户

- 数据照常追踪，不丢失
- 通过搜索可达（首页搜索栏 + 客户 Tab 搜索栏）
- 通过客户 Tab 可达
- 事件仍然生成，但只出现在事件 Tab（档案馆）和客户详情页，不推首页

### 设计理由

一个 BD 管 200+ 客户，但真正需要贴身服务的是前 30 个大客户。关注名单机制确保：
1. 首页不被低优先级信息淹没
2. 每天大约 2-3 条事件，老板 5 分钟可处理完
3. BD 精力集中在最有价值的客户上

## 三、事件生命周期

### 各类型过期窗口

| 事件类型 | 过期窗口 | 理由 |
|---------|---------|------|
| 风控触发 | **不过期**（直到解除） | 账户冻结是持续性问题，不会自动好转 |
| 超大额出金 | **48 小时** | 钱已经走了，超过 48h 挽留意义大减 |
| 合约/杠杆爆仓 | **72 小时** | 客户情绪冷却周期，超过 72h 安抚效果递减 |

### 处理闭环

当前实现使用 `sessionStorage` 管理处理状态：
1. BD 在事件线程页点击"标记已处理"
2. 事件 ID 写入 `sessionStorage('processed_events')`
3. 首页和客户详情页读取该列表，排除已处理事件
4. 标记后自动返回上一页

## 四、首页设计方案：头条模式

### 布局：6 元素恒定

| 位置 | 元素 | 展示内容 |
|------|------|---------|
| 顶部 | 搜索栏 | 按客户名或 UID 搜索，下拉浮层展示结果 |
| 摘要 | 今日统计 | 待处理 N + 跟进中 N + 全部链接 |
| 主体上 | **1 张大卡片** | 最紧急事件：事件类型 + 客户名 + 关键数字 + 客户资沉 + 建议动作 |
| 主体中 | **4 行小行** | 次紧急事件：事件类型 + 客户名 + 关键数字 |
| 主体下 | **进度条** | "还有 N 条未处理"或"已处理 X/Y"，点击进入事件 Tab |

### 大卡片 vs 小行

大卡片比小行多两个信息：

| 信息 | 大卡片 | 小行 |
|------|--------|------|
| 事件类型 | 有 | 有 |
| 客户名 | 有 | 有 |
| 关键数字 | 有 | 有 |
| 客户资沉 | **有** | 无 |
| 建议动作 | **有** | 无 |

### 空态

当所有事件已处理时：

- 大卡片区域：绿色勾 + "所有事件已处理"
- 小行区域：推荐跟进客户（关注名单中最久未联系的，带"推荐"标签）
- 进度条：绿色满进度 + "今日已处理 N 条"

### 满态

- 仍然 1+4 截断，不做无限滚动
- 进度条显示"还有 N 条待处理 · 查看全部"
- 点击进入事件 Tab 查看全部

## 五、Tab 架构

### 四 Tab 结构

| Tab | 隐喻 | 定位 | 核心逻辑 | 使用频率 |
|-----|------|------|---------|---------|
| **首页** | 显微镜 | 紧急，事件驱动 | 关注名单的紧急事件，Next Best Action | 每天必看 |
| **客户** | 望远镜 | 谁被遗忘了 | 按"最久未联系"排序，主动维护 | 每天看 |
| **事件** | 档案馆 | 历史归档 + 筛选检索 | 所有事件完整记录，按时间倒序 | 偶尔查 |
| **我的** | 设置 | 个人信息和偏好 | 关注名单管理、通知设置 | 很少用 |

### 各 Tab 详细说明

#### 首页 = 显微镜

- 回答："我现在该做什么？"
- 头条模式 1+4+进度条
- 只展示关注名单内、未过期、未处理的事件
- 空态推荐最久未联系的关注名单客户

#### 客户 Tab = 望远镜

- 回答："谁被遗忘了？"
- 默认按"最久未联系"排序
- 所有客户可达（不限关注名单）
- 搜索按名字或 UID
- 每行：客户名 + UID + 资沉 + 状态标签 + 最近联系时间（7 天+ 标红）

#### 事件 Tab = 档案馆

- 回答："历史上发生了什么？"
- 保留所有事件完整记录
- 筛选 pills：全部 / 待处理 / 跟进中 / 已完成 / 已忽略
- 统计摘要：共 N 个事件 · X 待处理 · Y 跟进中
- 按时间倒序排列

#### 我的 = 设置

- 关注名单管理（MVP 核心功能）
- 个人基本信息
- 通知偏好（Phase 2）

## 六、客户详情页结构

### 固定头部 + 3 Tab 架构

客户详情页采用固定头部 + 3 Tab 架构，共 10 个模块：

**固定头部（始终可见）：**
1. 客户身份：头像 + 名字 + 状态标签 + UID + 来源标签
2. 资沉核心指标：本周资沉 + vs 8 周均 + 流入/流出 + 异常告警
3. 概览判断：状态标签 + 一句话备忘

**Tab 1 事件：**
4. 当前事件（与 sessionStorage 联动）
5. 跟进记录（时间线样式）

**Tab 2 资产：**
6. 资金分布（5 条业务线占比）
7. 业务参与（6 条业务线状态）
8. 资沉流动（双向柱状图，默认展开）
9. 综合净贡献（默认折叠）+ 客户近期盈亏

**Tab 3 档案：**
10. BD 标签 + 备注

### 资沉流动指标

资沉变化不再使用趋势稳定度（R²）和线性回归趋势百分比，改为直观的流入/流出展示：

| 指标 | 位置 | 说明 |
|------|------|------|
| 本周流入 | 固定头部 | 绿色，本周资金流入总额 |
| 本周流出 | 固定头部 | 红色，本周资金流出总额 |
| 异常倍数 | 固定头部 | 流出超均值 2 倍时红色告警 |
| 流动柱状图 | Tab 2 资产 | 双向柱状图，上绿（流入）下红（流出），7 周数据 |

## 七、事件线程页

### 简化结构

事件线程页（event-thread.html）已简化为 3 区块 + 1 按钮：

| 区块 | 内容 |
|------|------|
| 事件头 | 紧急度圆点 + 事件类型 + 客户名 + 时间 + 金额 |
| 客户摘要 | 资沉 + 趋势状态 + 查看完整资料链接 |
| 操作区 | 风控：复制信息给研发。非风控：行动建议 + 话术模板 + 福利金 |
| 标记按钮 | 单按钮"标记已处理"，标记后自动返回 |

**已砍掉：** 4 状态按钮（待处理/跟进中/已完成/已忽略）、页内跟进记录。

## 八、MVP 范围

### 做

- 3 类首页事件：风控触发、超大额出金、合约/杠杆爆仓
- 关注名单机制（TOP 30 客户，按资沉排序）
- 头条模式首页（1+4+进度条 + sessionStorage 联动）
- 客户详情页（固定头部 + 3 Tab，10 模块）
- 资沉流动指标（流入/流出 + 异常倍数 + 双向柱状图）
- 事件线程简化（3 区块 + 单按钮标记已处理）
- 四 Tab 架构（显微镜/望远镜/档案馆/设置）

### 不做

- 算法推荐（哪些事件该推、推给谁）
- 颜色分区（红/金/绿/灰优先级标识）
- 智能排序（超过紧急度的多因子排序）
- 趋势稳定度（R²）等抽象统计指标
- 多状态事件流转（简化为已处理/未处理）
