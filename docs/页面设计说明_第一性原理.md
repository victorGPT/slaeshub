# Sales Hub 页面设计说明 — 从第一性原理出发

## 0. 第一性原理

**资金沉淀就是价值。**

不是"待转化的潜力"，不是"手续费的前置条件"，是价值本身。

原因：资金沉淀带来的收益（平台用资金做市、借贷、资金池等）远大于客户交易产生的手续费。一个客户把 1000 万放在平台上什么都不做，他给平台创造的价值可能比一个每天交易 100 万但不留钱的客户更大。

由此推导出销售工作的本质：

```
销售的工作 = 让更多的钱留在平台上，更久。
```

拆开就是三件事：

1. **守住大盘子** — 已有的大额沉淀不能丢
2. **止住流失** — 正在流出的钱要拦回来
3. **放大增长** — 正在流入的钱要顺势做大

所有页面设计都从这三件事出发。任何不服务于这三件事的元素，不放。

## 1. 首页（home.html）

### 这个页面回答什么

> BD 每天早上打开 App 的第一个问题：**我现在该做什么？**

首页定位：**Next Best Action** — 不是信息展示板，是行动指令台。

### 核心概念：事件 vs 动态

首页只推**事件（Event）**，不推动态（Activity）。区分标准：

> **老板不干预，结果会变坏吗？**

- **事件**：需要老板采取行动的，推到首页（仅 3 类）
  - 风控触发（账户冻了，必须找研发）
  - 超大额出金（占客户资沉比例很高，可能跑路）
  - 合约爆仓（客户情绪崩溃需安抚）
- **动态**：帮助了解客户的，留在客户详情页时间线
  - 正常出入金（即使金额不小）
  - 理财到期（大部分客户自动续）
  - 交易盈亏（除非特别极端）

### 关注名单机制

**只有关注名单内的客户才会生成首页事件。**

- 关注名单 = TOP 30 客户（按资沉 fMean 排序）
- BD/老板可手动调整
- 其余客户数据照常追踪，通过搜索或客户 Tab 可达
- 这确保首页不会被低优先级信息淹没，每天大约 2-3 条事件，老板 5 分钟可处理完

### 最终设计：头条模式（1+4+进度条）

首页恒定展示 **6 个元素**：

| 位置 | 元素 | 内容 |
|------|------|------|
| 顶部 | 搜索栏 | 按客户名或 UID 搜索，下拉展示匹配结果 |
| 摘要行 | 今日统计 | 待处理 N + 跟进中 N + 全部链接 |
| 主体上 | 1 张大卡片 | 最紧急事件：事件类型 + 客户名 + 关键数字 + **客户资沉** + **建议动作** |
| 主体中 | 4 行小行 | 次紧急事件：事件类型 + 客户名 + 关键数字 |
| 底部 | 进度条 | 显示"还有 N 条未处理"或"已处理 X/Y" |

**大卡片比小行多两个信息：** 客户资沉 + 建议动作。这让 BD 对最紧急的事件无需点进去就能行动。

#### 空态

所有事件已处理时，显示：
- 大卡片区域：绿色勾 + "所有事件已处理"
- 小行区域：推荐跟进客户（关注名单中最久未联系的客户，带"推荐"标签）
- 进度条：显示"今日已处理 N 条"（绿色满进度）

#### 满态

仍然 1+4 截断，进度条显示"还有 N 条待处理 · 查看全部"。点击进入事件 Tab 查看全部。

### 事件生命周期

事件有过期窗口，不会无限积压：

| 事件类型 | 过期窗口 | 理由 |
|---------|---------|------|
| 风控触发 | 不过期（直到解除） | 账户冻结不会自动好转 |
| 超大额出金 | 48 小时 | 钱已经走了，超过 48h 挽留意义大减 |
| 合约爆仓 | 72 小时 | 客户情绪冷却周期，超过 72h 安抚效果递减 |

过期事件自动沉底，变为"已过期"状态，归档到事件 Tab。

### 事件处理联动

首页与事件线程页通过 `sessionStorage` 联动：
- 事件线程页点击"标记已处理"后，事件 ID 写入 `sessionStorage('processed_events')`
- 首页读取该列表，排除已处理事件
- 标记后自动返回首页（600ms 延迟）

### 设计决策与理由

#### 为什么用头条模式而非事件流

旧设计是无限滚动的事件流。问题：BD 不知道"到底有多少事"，也不知道"处理到哪了"。头条模式用恒定 6 元素 + 进度条，给 BD 明确的"工作量感知"和"完成感"。

#### 为什么 MVP 不用颜色分区

颜色分区需要算法准确率验证（Phase 2）。MVP 用大小梯度（大卡片 vs 小行）区分优先级，视觉层次足够清晰。

#### 搜索栏

保留在首页顶部。BD 心里有个人想看（出门见客户前、接到电话后），搜索栏让他 2 秒内找到，不需要切到客户 Tab。搜索结果以下拉浮层展示，显示客户名、UID、资沉和状态。

**这怎么服务第一性原理：** 首页的唯一职责是"把关注名单中最需要行动的事件推到 BD 面前"。头条模式确保 BD 5 分钟处理完当天所有紧急事项，剩下的时间用来做客户 Tab 里的主动维护。

## 2. 客户详情页（client-detail.html）

### 这个页面回答什么

> BD 点进一个客户后的核心问题：
> 1. 这是谁，多重要？
> 2. 他的钱最近怎么了，资金是在流入还是流出？
> 3. 有没有紧急事件要处理？
> 4. 他的资产怎么分布，参与了哪些业务？

### 页面架构：固定头部 + 3 Tab

#### 固定头部（3 个模块，始终可见）

| 模块 | 名称 | 核心内容 |
|------|------|---------|
| 1 | 客户身份 | 头像 + 名字 + 状态标签（流失/增长/稳定）+ UID + 来源标签（来自事件/客户列表/搜索） |
| 2 | 资沉核心指标 | 本周资沉（页面最大数字）+ vs 8 周均值 + 本周流入/流出 + 异常告警（流出超均值倍数） |
| 3 | 概览判断 | 状态标签 + 一句话备忘（输入即保存到 localStorage） |

#### Tab 1：事件（默认激活）

| 区块 | 名称 | 内容 |
|------|------|------|
| 4 | 当前事件 | 该客户未处理的事件列表（与 sessionStorage 联动），无事件时显示"暂无待处理事件" |
| 5 | 跟进记录 | 时间线样式，区分人工记录（蓝点）和系统记录（灰点），支持"添加记录" |

#### Tab 2：资产

| 区块 | 名称 | 内容 |
|------|------|------|
| 6 | 资沉分布 | 7 条业务线（理财/双仓合约/现货/金融卡/云算力），金额 + 百分比 + 进度条 |
| 7 | 业务参与 | 7 条业务线（双仓合约/超级杠杆/Mini合约/理财/金融卡/云算力）参与状态 |
| 8 | 资沉流动 | 双向柱状图（流入绿/流出红），按周展示，直接展开 |
| 9a | 综合净贡献 | 默认折叠，点击展开。总额 + 周环比 + 各模块条形图（收入绿/成本红） |
| 9b | 客户近期盈亏 | 近 7 天总盈亏（大数字）+ 8 周周均盈亏 + 按业务拆分条形图 |

#### Tab 3：档案

| 区块 | 名称 | 内容 |
|------|------|------|
| 10a | BD 标签 | 可编辑标签（如"偏好短线"、"英文沟通"），支持添加/删除，localStorage 持久化 |
| 10b | 备注 | 自由文本 textarea，输入即保存到 localStorage |

### 核心指标说明

#### 资沉核心指标（模块 2）

| 指标 | 说明 |
|------|------|
| 本周资沉 | 页面最大数字（24px），回答"盘子多大" |
| vs 8 周均 | 与 8 周均值对比，回答"变化了多少" |
| 本周流入/流出 | 资金双向流动，绿色流入 + 红色流出 |
| 异常告警 | 当流出超过均值 2 倍时，显示红色提示"流出是均值的 X.X 倍" |

**注意：已移除"趋势稳定度"（R²）和"近期趋势百分比"指标。** 资沉变化通过流入/流出双向数据直观展示，无需抽象统计指标。

#### 资沉流动（模块 8）

双向柱状图直接展开（非折叠），每周一组双柱（流入绿 + 流出红），展示 7 周数据（W2-W8）。配有图例说明。

### 设计决策

- **固定头部**：身份 + 资沉 + 备忘始终可见，切换 Tab 不丢失上下文
- **3 Tab 而非长滚动**：事件/资产/档案是三个不同的信息消费场景，Tab 切换比长页面滚动更高效
- **来源标签**：通过 URL 参数 `from` 传入（event/clients/search），帮助 BD 理解当前进入上下文
- **综合净贡献默认折叠**：大多数时候 BD 只看资沉和流动，净贡献是深入分析时才需要
- **资沉流动直接展开**：资金的流入流出是 BD 判断客户状态的核心依据，不应藏在折叠里

**这怎么服务第一性原理：** 固定头部确保"这个客户有多少钱、最近钱怎么流动"始终可见。Tab 架构让不同场景（处理事件 / 分析资产 / 管理档案）各取所需，不互相干扰。

## 3. 事件线程（event-thread.html）

### 这个页面回答什么

> 这个事件到底怎么回事，我该怎么处理？

### 简化结构（3 区块 + 1 按钮）

| 区块 | 名称 | 内容 |
|------|------|------|
| A | 事件头 | 紧急度圆点 + 事件类型（含子类型）+ 客户名 + 时间 + 关键金额（大数字） |
| B | 客户摘要 | 资沉 + 趋势状态 + "查看完整资料"链接 |
| C | 操作区 | 根据事件类型分流（见下文） |
| - | 标记按钮 | 单按钮"标记已处理"（绿色），处理后显示"已处理"状态 |

#### 操作区分流逻辑

**风控触发事件：**
- 行动建议
- 一键复制信息发给研发（客户 UID + 风控类型 + 触发时间 + 客户资沉）

**非风控事件（超大额出金/爆仓）：**
- 行动建议
- 话术模板 + 复制话术按钮
- 福利金剩余额度 + 发放按钮

### 标记已处理逻辑

- 点击"标记已处理"后，事件 ID 写入 `sessionStorage('processed_events')`
- 按钮变为"已处理"状态（绿色描边）
- 600ms 后自动 `history.back()` 返回上一页（通常是首页）

### 与旧设计的差异

已砍掉的内容：
- ~~4 状态按钮（待处理/跟进中/已完成/已忽略）~~→ 简化为单按钮"标记已处理"
- ~~跟进记录区块~~→ 跟进记录移至客户详情页 Tab 1
- ~~复杂状态流转~~→ 处理 = 标记 + 返回，极简闭环

**这怎么服务第一性原理：** 事件线程是"看清事件 → 获得上下文 → 一键操作 → 标记完成 → 返回首页"的极简闭环。减少页面停留时间，让 BD 快速回到首页继续处理下一条。

## 4. 客户列表（客户 Tab）= 望远镜

### 这个页面回答什么

> 谁被遗忘了？我应该主动维护谁的关系？

客户 Tab 的定位是**望远镜** — 看重要的、关系驱动的事，不是看紧急的。

### 最终设计：按"最久未联系"排序的客户列表

默认排序：**最久未联系** 排在最前面（按最近事件时间计算，无事件 = 7 天+）。

#### 每行展示内容

| 内容项 | 说明 |
|--------|------|
| 客户名 + UID | 确认是谁 |
| 资沉金额 | 这个人多重要 |
| 状态标签 | 流失/增长/稳定（带颜色） |
| 最近联系时间 | 多久没联系了（7 天+ 标红） |

#### 搜索：按名字或 UID

顶部搜索栏，实时过滤。

#### 与首页的分工

- 首页 = 显微镜：看紧急的、事件驱动的（关注名单的紧急事件）
- 客户 Tab = 望远镜：看被遗忘的、关系驱动的（按最久未联系排序）

BD 处理完首页事件后，来客户 Tab 做主动维护。

**这怎么服务第一性原理：** 资沉流失往往从"失联"开始。按最久未联系排序，把即将失联的客户推到 BD 面前，预防性维护比事后挽留成本低得多。

## 5. 事件中心（events.html）= 档案馆

### 这个页面回答什么

> 历史上发生了什么事件？我需要回顾或筛选某类事件。

事件 Tab 的定位是**档案馆** — 历史归档 + 筛选检索。按时间倒序排列。

### 与首页的关系

首页只展示当前有效的、关注名单内的紧急事件（头条模式 1+4）。事件 Tab 则保留**所有事件的完整历史**，包括：
- 已处理的事件
- 已过期的事件
- 非关注名单客户的事件

### 筛选 pills

顶部横向滚动筛选栏：
- 全部（默认）
- 待处理
- 跟进中
- 已完成
- 已忽略

选中态为金色底色（#F0B90B），未选中为深色卡片底色。

### 统计摘要

筛选结果上方显示统计行：`共 N 个事件 · X 待处理 · Y 跟进中`

### 设计决策

#### 为什么事件 Tab 不再是主要工作台

旧设计中事件中心承担了"今天该做什么"的职责。新设计中这个职责完全移交给首页（头条模式）。事件 Tab 退化为纯归档，BD 日常工作中很少需要主动打开它。

**这怎么服务第一性原理：** 档案馆确保没有信息丢失。BD 需要回顾某个客户的事件历史时（如准备见面前做功课），这里是唯一的完整记录。

## 6. 个人页面（profile.html）= 设置

### 这个页面回答什么

> 我的设置和基本信息。

### Tab 架构总览

| Tab | 隐喻 | 定位 | 核心逻辑 |
|-----|------|------|---------|
| 首页 | 显微镜 | 紧急，事件驱动 | 关注名单的紧急事件，Next Best Action |
| 客户 | 望远镜 | 谁被遗忘了 | 按最久未联系排序，主动维护 |
| 事件 | 档案馆 | 历史归档 + 筛选检索 | 所有事件完整记录，按时间倒序 |
| 我的 | 设置 | 个人信息和偏好 | 关注名单管理、通知设置 |

### 设计决策与理由

MVP 阶段重点内容：
- **关注名单管理**：维护 TOP 30 客户列表（按资沉排序推荐，BD 可手动调整）
- 基本个人信息

## 7. 设计原则总结

### 原则 1：资沉是唯一的北极星

每个页面的核心数字、排序标准、事件触发条件、动作建议，都指向同一个量：资金沉淀。

### 原则 2：事件推行动，动态供了解

推到首页的必须是"不干预会变坏"的事件（仅 3 类），其余都是动态。这条原则防止首页被信息噪音淹没。

### 原则 3：资沉用流入/流出展示变化

不使用抽象统计指标（如趋势稳定度 R²、线性回归斜率），而用直观的资金流入/流出双向数据展示资沉变化。异常时标注"流出是均值的 X 倍"。

### 原则 4：给结论，不给数据

BD 不需要知道复杂的统计公式，BD 需要知道"钱在流出，而且比平时多很多"。所有后台算法对前台不可见，前台只展示人话。

### 原则 5：一个动作，不给选择

每个事件只给一个建议动作 + 一套话术。减少决策负担，增加执行速度。事件线程页只有一个"标记已处理"按钮，不给多状态选择。

### 原则 6：极简闭环

事件处理闭环：首页看到 → 点进事件线程 → 操作（复制/发放）→ 标记已处理 → 自动返回首页。整个流程不超过 30 秒。
