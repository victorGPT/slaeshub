# 用户动线与场景设计 — 从销售视角出发

## 0. 设计原则

**不要从"我有什么功能"出发，要从"销售在什么情境下打开 App"出发。**

销售打开 App 只有两种情况：

1. **我主动想看** — 有目的地打开（主动事件）
2. **系统叫我看** — 收到推送后打开（被动事件）

每种情况下，销售的心理状态、时间预算、期望看到的内容完全不同。如果动线设计不对，他会在 3 秒内关掉 App。

---

## 1. 主动事件场景

### 场景 A：早上到公司，打开 App 看看今天要做什么

**销售心态：** "今天有什么事要处理？"

**时间预算：** 30 秒扫一眼

**期望动线：**
```
打开 App
  → 首页
    → 一眼看到：有几件待处理的事（红点/数字）
    → 最紧急的排最前面
    → 点进去看详情 → 知道该做什么 → 标记"跟进中"
    → 回到列表继续扫下一个
```

**当前设计能走通吗？**

| 步骤 | 能否走通 | 问题 |
|------|---------|------|
| 打开 App → 首页 | ✅ | index.html 跳转 home.html |
| 一眼看到待处理数量 | ⚠️ | 有流失/增长 Tab 的 badge 数字，但没有"待处理任务"的数字 |
| 最紧急的排最前面 | ❌ | 默认 Tab 是"资沉总量"（按金额排），不是"待处理"。销售要自己切到流失 Tab |
| 点进去 → 标记 → 回来 | ⚠️ | 可以点进详情页，但任务状态只在事件线程里，详情页没有直接标记按钮 |

**🔴 断点：销售早上打开 App，第一眼看不到"今天有几件事要做"。**

**修复建议：** 首页顶部增加一个"待处理摘要栏"：

```
┌─────────────────────────────────────┐
│  📋 今日待处理：3 件事              │
│  🔴 2 个流失预警  🟡 1 个理财到期    │
└─────────────────────────────────────┘
```

点击这个栏直接进入待处理列表（或自动切到流失 Tab 并只显示有待处理事件的客户）。

### 场景 B：出门前，搜一下今天要拜访的客户

**销售心态：** "我要去见张伟，先看看他的情况。"

**时间预算：** 10 秒找到人，1 分钟看完情况

**期望动线：**
```
打开 App
  → 搜索（输入名字或 UID）
    → 找到张伟
    → 点进详情页
    → 看资沉、趋势、最近事件、上次跟进记录
    → 心里有数了，出门
```

**当前设计能走通吗？**

| 步骤 | 能否走通 | 问题 |
|------|---------|------|
| 打开 App → 搜索 | ❌ | 首页没有搜索框。搜索在 clients.html 页面里。销售要先点底部 Tab"客户"，再搜索。多了一步 |
| 找到人 → 点进详情 | ✅ | clients.html 搜索正常 |
| 看完情况 | ✅ | 详情页信息完整 |

**🟡 摩擦点：首页没有搜索入口，销售要多点一步。**

**修复建议：** 首页顶部的搜索图标做成可点击的，点击后展开搜索框（或跳转到 clients.html 并自动聚焦搜索框）。

---

### 场景 C：闲着没事，翻翻客户看看谁值得维护

**销售心态：** "没什么急事，看看有没有可以主动做的。"

**时间预算：** 5-10 分钟慢慢翻

**期望动线：**
```
打开 App
  → 首页 → 资沉总量 Tab
    → 从上到下翻，看看大客户的状态
    → 看到某个稳定客户，想"好久没联系了"
    → 点进去 → 看跟进记录 → 发现上次联系是 20 天前
    → 决定今天打个电话
    → 添加一条跟进记录
    → 回到列表
```

**当前设计能走通吗？**

| 步骤 | 能否走通 | 问题 |
|------|---------|------|
| 翻列表看大客户 | ✅ | 资沉总量 Tab 正好是按金额排 |
| 看到某人想点进去 | ✅ | 卡片/列表行可点击 |
| 看跟进记录 | ⚠️ | 有跟进记录区块，但是静态占位数据 |
| 添加跟进记录 | ❌ | 详情页有"+ 添加记录"按钮文字，但功能未实现 |
| 回到列表 | ✅ | 返回按钮正常 |

**🟡 摩擦点：跟进记录是静态的，不能添加。** MVP 阶段可以接受（占位），但需要在 UI 上明确标注"即将上线"。

---

### 场景 D：开完周会，经理让看一下流失情况

**销售心态：** "老板问我最近流失了多少，我得看看数据。"

**时间预算：** 2-3 分钟

**期望动线：**
```
打开 App
  → 首页 → 切到"资沉流失"Tab
    → 看到流失客户列表 + 数量
    → 快速扫一遍，了解谁在掉
    → 可能截图发给经理
```

**当前设计能走通吗？**

| 步骤 | 能否走通 | 问题 |
|------|---------|------|
| 切到流失 Tab | ✅ | Tab 切换正常 |
| 看到数量 | ✅ | Tab 上有 badge 数字 |
| 扫列表 | ✅ | 按 ChurnScore 排序 |

**✅ 这个场景完全走通。**

## 2. 被动事件场景

### 场景 E：收到推送"张伟大额提现 $180 万"

**销售心态：** "卧槽，赶紧看看怎么回事！"

**时间预算：** 5 秒内定位到信息

**期望动线：**
```
收到推送通知
  → 点击推送
    → 直接跳到相关页面（事件线程 or 客户详情）
    → 看到：什么事、多严重、建议做什么
    → 决定：立刻打电话 or 先标记待跟进
    → 标记任务状态
    → 回到首页（或关掉 App）
```

**当前设计能走通吗？**

| 步骤 | 能否走通 | 问题 |
|------|---------|------|
| 收到推送 | ❌ | **没有推送系统。** 目前是纯静态原型，无推送能力 |
| 点击推送 → 跳转 | ❌ | 没有 deeplink 机制 |
| 看到事件详情 | ⚠️ | event-thread.html 是静态占位 |
| 标记任务状态 | ⚠️ | 有 4 个状态按钮但是静态的 |

**🔴 完全断裂。被动事件流程在 MVP 原型阶段是空的。**

**MVP 可以做的最小化方案：**

不做真推送，但在首页模拟"未读事件"的体验：

```
┌─────────────────────────────────────┐
│ 🔔 新事件                           │
│ 🔴 张伟 · 大额提现 $180万 · 2小时前  │
│ 🟡 李芳 · 理财到期 $500万 · 今天     │
└─────────────────────────────────────┘
```

放在首页最顶部（Tab 上方），点击直接跳到事件线程。这样即使没有推送，销售打开 App 也能第一眼看到紧急事件。

---

### 场景 F：收到推送"客户被风控"

**销售心态：** "这个客户出什么事了？会不会影响关系？"

**期望动线：** 与场景 E 类似，但需要额外信息：风控原因、是否影响客户操作、BD 能做什么。

**当前设计：** 完全不支持。风控事件不在当前事件类型中。

**建议：** Phase 2 补充风控类事件。MVP 不做。

## 3. 动线断点汇总

| 断点 | 严重程度 | 影响场景 | 建议修复 |
|------|---------|---------|---------|
| 首页没有"待处理"摘要 | 🔴 高 | A（早上看任务） | 首页顶部加待处理摘要栏 |
| 首页没有搜索入口 | 🟡 中 | B（出门前搜人） | 顶部搜索图标点击可用 |
| 被动事件完全缺失 | 🔴 高 | E/F（收到推送） | 首页加"新事件"横幅模拟 |
| 跟进记录不能添加 | 🟡 中 | C（维护客户） | MVP 标注"即将上线" |
| 默认 Tab 不是最紧急的 | 🟡 中 | A（早上看任务） | 如果有待处理事件，默认切到流失 Tab |
| 任务状态列表里看不到 | 🟡 中 | A（早上看任务） | 列表行加任务状态点 |

## 4. 修复后的首页结构

```
┌─────────────────────────────────────┐
│ 状态栏                               │
├─────────────────────────────────────┤
│ 🔍 搜索客户...        🔔(红点)       │  ← 搜索可点击，铃铛显示未读数
├─────────────────────────────────────┤
│ 🔔 新事件 (2)                        │  ← 紧急事件横幅（被动事件入口）
│ 🔴 张伟 · 大额提现 $180万 · 2h前     │
│ 🟡 李芳 · 理财到期 $500万 · 今天      │
├─────────────────────────────────────┤
│ 📋 今日待跟进：5 人                   │  ← 待处理摘要（主动事件入口）
├─────────────────────────────────────┤
│ [ 资沉总量 ] [ 资沉流失(3) ] [ 增长(5) ] │  ← 三 Tab（不变）
├─────────────────────────────────────┤
│ 客户列表...                          │
│ #1 张伟  $1,200万 🔴快速流失 ● → 回访 │
│ #2 李芳  $850万  🟠缓慢流失 ◐ → 续存 │
│ #3 王强  $2,100万 ⚪稳定    — → U卡  │
├─────────────────────────────────────┤
│ [ 首页 ] [ 客户 ] [ 事件 ] [ 我的 ]   │
└─────────────────────────────────────┘
```

**从上到下的信息层次：**

1. **搜索**（主动找人——场景 B）
2. **新事件横幅**（被动事件——场景 E/F）
3. **待跟进摘要**（今天要做什么——场景 A）
4. **三 Tab 列表**（看盘/止损/放大——场景 C/D）

**销售无论是哪种心态打开 App，前 3 秒内都能找到入口。**

## 5. 主动 vs 被动的完整定义

经过场景分析，明确定义：

### 主动事件（销售发起）

**定义：销售自己决定打开 App 去做的事。**

| 场景 | 触发者 | 入口 |
|------|--------|------|
| 搜索特定客户 | 销售 | 搜索框 |
| 浏览客户列表 | 销售 | 三 Tab 列表 |
| 查看客户详情 | 销售 | 点击列表行 |
| 添加跟进记录 | 销售 | 详情页 |
| 打标签/写备注 | 销售 | 详情页 |

**特点：** 销售有时间、有目的、愿意花时间探索。UI 需要提供足够的信息深度。

### 被动事件（系统发起）

**定义：系统检测到资沉变化后，主动推给销售的提醒。**

| 事件类型 | 触发条件 | 紧急程度 |
|---------|---------|---------|
| 大额提现预警 | 单笔/累计出金超过阈值 | 🔴 最高 |
| 理财到期提醒 | 理财产品 T-3 天即将到期 | 🟡 高 |
| 资沉持续流失 | 连续 N 天 AdjTrendRate < -δ | 🟠 中 |
| 大额入金通知 | 单笔/累计入金超过阈值 | 🟢 信息性 |
| 资沉持续增长 | 连续 N 天 AdjTrendRate > δ | 🟢 信息性 |
| 风控触发 | 客户被风控系统标记 | 🔴 最高 |

**特点：** 销售可能在忙别的事，被打断。UI 需要**3 秒内**让他知道"什么事、多急、我做什么"。不需要深度，需要速度。

### 大额提现属于哪类？

**大额提现是被动事件。**

- 销售不是自己发现的，是系统检测到的
- 它需要系统主动推送（即使 MVP 没有真推送，也要在首页顶部模拟）
- 它的紧急程度是 🔴 最高——客户在主动搬钱走，每多等一小时，钱可能就转到竞品了

## 6. "知识的诅咒"检查清单

用来自检每个页面是否对"第一次用的销售"足够清晰：

| 检查项 | 通过标准 |
|--------|---------|
| 打开 App 后，3 秒内能看到"今天有什么要做的" | 首页有待处理摘要或紧急事件横幅 |
| 不需要教程就知道三个 Tab 是什么意思 | Tab 名字自解释（资沉总量/流失/增长） |
| 每个列表行，不需要点进去就能判断"要不要管" | 列表行有状态标签 + 任务状态点 |
| 每个详情页，不需要滚动就能看到最重要的信息 | 资沉金额 + 趋势在首屏 |
| 从任何页面都能用 1 步回到首页 | 底部 Tab 栏始终可见 |
| 搜索客户不超过 2 步 | 首页有搜索入口 |
| 看到紧急事件不超过 1 步 | 首页有事件横幅（被动事件入口） |
| 标记"已处理"不需要离开当前页面 | 详情页或事件线程里有状态按钮 |
| 所有按钮的作用不需要猜 | 按钮文案自解释（不用图标代替文字） |
| 没有死胡同（点进去出不来） | 每个页面有返回按钮或底部导航 |
