# 资沉变化归因框架 — 不是所有涨跌都一样

## 1. 问题

现在我们知道一个客户的资沉在涨或在跌，但不知道**为什么**。

同样是增加 100 万：
- 客户主动充了 100 万进来 → 关系在加深，非常好
- 客户炒合约赚了 100 万 → 市场好而已，明天可能就亏回去

同样是减少 100 万：
- 客户主动提了 100 万走 → 关系在流失，需要紧急干预
- 客户理财到期自动赎回了 100 万 → 正常操作，可能马上续存
- 客户炒合约亏了 100 万 → 市场风险，不是关系问题

如果不区分原因，BD 会把时间浪费在不需要干预的客户身上，或者漏掉真正在流失的客户。

## 2. 核心思路：把资沉变化拆成"来源"

资沉在某一段时间内的变化，可以拆成几个来源：

```
ΔF = ΔF_deposit + ΔF_withdraw + ΔF_trading + ΔF_yield + ΔF_other
```

| 来源 | 含义 | 数据来源 |
|------|------|---------|
| ΔF_deposit | 充值入金（法币/加密货币转入） | 充提记录 |
| ΔF_withdraw | 提现出金（法币/加密货币转出） | 充提记录 |
| ΔF_trading | 交易盈亏（合约、现货、杠杆等） | 交易系统 |
| ΔF_yield | 理财/借贷收益或到期赎回 | 理财系统 |
| ΔF_other | 其他（手续费、空投、返佣等） | 各系统 |

验证：ΔF_deposit + ΔF_withdraw + ΔF_trading + ΔF_yield + ΔF_other ≈ 实际资沉变化

## 3. 变化质量分级

### 3.1 资沉增加的质量

不是所有增加都值得高兴。

| 增加来源 | 质量评级 | 理由 | BD 应该做什么 |
|---------|---------|------|-------------|
| 主动入金（ΔF_deposit > 0） | 🟢 **高质量** | 客户主动选择把钱放进来，代表信任加深 | 顺势推荐更多业务线，加深关系 |
| 理财收益（ΔF_yield > 0，非赎回） | 🟢 **高质量** | 利息/收益自然增长，资沉在滚雪球 | 维护即可，确保到期续存 |
| 交易盈利（ΔF_trading > 0） | 🟡 **中等质量** | 市场好的时候赚了钱，但可能随时亏回去 | 建议客户将部分盈利转入理财锁定 |
| 空投/返佣（ΔF_other > 0） | ⚪ **低质量** | 不是客户主动行为，不代表关系变化 | 不需要特别动作 |

**核心判断标准：这个增加是客户主动选择的，还是市场/系统被动造成的？**

主动选择 = 高质量（关系在加深）
被动造成 = 低质量（可能随时逆转）

### 3.2 资沉减少的质量

不是所有减少都需要紧张。

| 减少来源 | 紧急程度 | 理由 | BD 应该做什么 |
|---------|---------|------|-------------|
| 主动出金（ΔF_withdraw < 0） | 🔴 **高度紧急** | 客户主动把钱转走，关系可能在断裂 | 立即联系，了解原因，挽留 |
| 理财到期赎回（ΔF_yield < 0） | 🟡 **需关注** | 理财到期是正常操作，但如果不续存就变成流失 | 提前提醒续存，推送权益 |
| 交易亏损（ΔF_trading < 0） | 🟡 **需关注** | 市场风险，不是关系问题，但持续亏损会导致客户离开 | 关注客户情绪，适时沟通风控 |
| 手续费扣除（ΔF_other < 0） | ⚪ **正常** | 交易成本，金额小，完全正常 | 不需要动作 |

**核心判断标准：这个减少是客户主动选择离开，还是业务正常消耗？**

主动离开 = 高紧急（关系在断裂）
业务消耗 = 低紧急（正常运营）

## 4. 归因后的客户状态矩阵

把"变化方向"和"变化质量"交叉，得到 6 种客户状态：

### 4.1 资沉增加的客户

| 状态 | 条件 | 含义 | 前台标签 | 前台颜色 |
|------|------|------|---------|---------|
| **健康增长** | 增加主要来自入金或理财收益 | 客户在主动加深关系 | "主动加仓" | 🟢 绿色 |
| **虚胖增长** | 增加主要来自交易盈利 | 看起来在涨，但随时可能回吐 | "盈利浮增" | 🟡 黄色 |

**BD 看到"主动加仓"→ 放心做大。看到"盈利浮增"→ 别高兴太早，建议客户锁定部分盈利。**

### 4.2 资沉减少的客户

| 状态 | 条件 | 含义 | 前台标签 | 前台颜色 |
|------|------|------|---------|---------|
| **主动流失** | 减少主要来自出金 | 客户在主动搬钱走 | "主动出金" | 🔴 红色 |
| **到期风险** | 减少主要来自理财到期赎回 | 不是在走，但钱已经"解锁"了 | "理财待续" | 🟡 黄色 |
| **市场亏损** | 减少主要来自交易亏损 | 不是关系问题，是市场问题 | "交易回撤" | 🟠 橙色 |

**BD 看到"主动出金"→ 立刻救。看到"理财待续"→ 赶紧推续存。看到"交易回撤"→ 先别慌，关注情绪。**

### 4.3 资沉稳定的客户

| 状态 | 条件 | 含义 | 前台标签 |
|------|------|------|---------|
| **稳定沉淀** | 各项变化互相抵消或都很小 | 正常状态 | "稳定" |

## 5. 前台展示设计

### 5.1 首页卡片增加"变化性质"标签

现在的卡片：
```
张伟                    $ 1,200 万
● 流失  -3.2%/周 · 趋势明确
理财到期/赎回
→ 安排销售回访
```

增加归因后的卡片：
```
张伟                    $ 1,200 万
● 流失  -3.2%/周 · 趋势明确
🔴 主动出金 · 近7天出金 $180万
→ 立即联系，了解出金原因
```

或者：
```
李芳                    $ 850 万
● 流失  -1.8%/周 · 波动较大
🟡 理财待续 · 500万理财即将到期
→ 推送续存权益说明
```

**变化：**
- "原因标签"从模糊文案升级为**带来源的精确归因**
- 标签颜色直接反映紧急程度
- 建议动作与归因原因强关联

### 5.2 客户详情页增加"变化归因"区块

> **设计原则：给结论，不给数据。**
>
> 交叉验证指出：直接展示 5 行 delta 金额违反了"给结论不给数据"原则。
> BD 不需要看到每一笔 delta，BD 需要看到**一句结论**。

在核心指标区块下方，增加一个结论式归因卡片：

**流失客户示例：**
```
本周资沉变化
净变化：-$154 万
────────────────────────
🔴 主要原因：主动出金（占 72%）
→ 客户在主动搬钱走，需要立即了解原因
```

**高质量增长示例：**
```
本周资沉变化
净变化：+$254 万
────────────────────────
🟢 主要原因：主动入金（占 79%）
→ 真实增长，值得顺势做大
```

**虚胖增长示例：**
```
本周资沉变化
净变化：+$387 万
────────────────────────
🟡 主要原因：交易盈利（占 98%）
→ 增长不稳定，建议引导部分盈利转入理财
```

**设计要点：**
- 只展示净变化金额 + 主因结论 + 一句话建议
- 不逐项列出 5 个 delta 流的具体金额
- 如果 BD 需要细节，可以点击"查看明细"展开（折叠隐藏）
- 结论颜色与主因紧急度/质量一致

## 6. 归因算法

### 6.1 数据需求

要实现归因，需要按来源分类的资沉变化数据：

| 字段 | 说明 | 数据来源 |
|------|------|---------|
| `delta_deposit` | 本周净入金（充值 - 不含提现） | 充提系统 |
| `delta_withdraw` | 本周净出金（提现金额，负值） | 充提系统 |
| `delta_trading` | 本周交易盈亏 | 交易系统（合约+现货+杠杆） |
| `delta_yield` | 本周理财/借贷净变化（收益 - 到期赎回） | 理财系统 |
| `delta_other` | 其他（手续费、空投、返佣等） | 各系统汇总 |

### 6.2 主因判定

```
components = [delta_deposit, delta_withdraw, delta_trading, delta_yield, delta_other]

# 对于资沉减少的客户，找最大的负向贡献
if 净变化 < 0:
    主因 = 绝对值最大的负向 component

    if 主因 == delta_withdraw:
        标签 = "主动出金"，紧急度 = 🔴
    elif 主因 == delta_yield (且为到期赎回):
        标签 = "理财待续"，紧急度 = 🟡
    elif 主因 == delta_trading:
        标签 = "交易回撤"，紧急度 = 🟠
    else:
        标签 = "其他减少"，紧急度 = ⚪

# 对于资沉增加的客户，找最大的正向贡献
if 净变化 > 0:
    主因 = 绝对值最大的正向 component

    if 主因 == delta_deposit:
        标签 = "主动加仓"，质量 = 🟢
    elif 主因 == delta_yield (且为收益):
        标签 = "理财滚存"，质量 = 🟢
    elif 主因 == delta_trading:
        标签 = "盈利浮增"，质量 = 🟡
    else:
        标签 = "其他增加"，质量 = ⚪
```

### 6.3 主因占比

```
主因占比 = |主因| / Σ|所有同方向 components|
```

当主因占比 > 70% 时，归因结论可信度高，前台可以直接展示。
当主因占比 < 50% 时，归因不明确，前台显示"综合因素"。

## 7. 对排序公式的影响

### 7.1 ChurnScore 可以引入"紧急度加权"

```
ChurnScore = log(1 + F/T_F) × max(0, -AdjTrendRate - δ) × Conf × UrgencyWeight
```

| 主因 | UrgencyWeight | 理由 |
|------|--------------|------|
| 主动出金 | 1.5 | 关系在断裂，最紧急 |
| 理财待续 | 1.2 | 有窗口期但需要抓紧 |
| 交易回撤 | 0.8 | 市场问题，不是最紧急的关系问题 |
| 其他/不明 | 1.0 | 默认 |

**效果：** 同样是跌 3%/周的客户，如果是主动出金导致的，ChurnScore 会乘 1.5，排名提前。如果只是交易亏损，ChurnScore 乘 0.8，排名靠后。

### 7.2 GrowthScore 可以引入"质量加权"

```
GrowthScore = log(1 + F/T_F) × max(0, AdjTrendRate - δ) × Conf × QualityWeight
```

| 主因 | QualityWeight | 理由 |
|------|--------------|------|
| 主动入金 | 1.5 | 最可靠的增长 |
| 理财滚存 | 1.2 | 自然增长，可持续 |
| 交易盈利 | 0.7 | 不可靠，可能回吐 |
| 其他/不明 | 1.0 | 默认 |

**效果：** 同样是涨 5%/周的客户，主动入金的排在前面，纯靠炒币赚钱的排后面。

## 8. 实施路径

> **交叉验证结论：分阶段实施，避免一次性做太多违背 MVP 精神。**

### Phase 1（当前 MVP）

不做归因。原因标签用预定义池随机分配。

### Phase 2a（充提数据就绪后）— 最小归因

**只需要 2 个数据流**（最容易获取）：

```
delta_deposit   = 本周净入金
delta_withdraw  = 本周净出金
delta_residual  = 总变化 - delta_deposit - delta_withdraw（包含交易盈亏+理财+其他）
```

**三分法状态**：
- 入金驱动（主因 = delta_deposit）
- 出金驱动（主因 = delta_withdraw）
- 其他变化（主因 = delta_residual）

**这已经覆盖了 80% 的归因价值**——BD 最需要知道的就是"客户在主动搬钱"还是"市场波动"。

**排序：不引入权重**。ChurnScore / GrowthScore 保持原公式不变，归因只用于标签展示。

### Phase 2b（理财+交易数据就绪后）— 完整归因

将 delta_residual 拆分为：
- delta_trading（交易盈亏）
- delta_yield（理财收益/到期）
- delta_other（手续费等）

状态升级为完整 6 类。

### Phase 3（数据积累后）— 权重验证

**前提：Phase 2 运行 3 个月以上，有足够的动作效果数据。**

验证：
- 哪种类型的流失，BD 干预后挽回率最高？
- "主动出金"是否真的比"交易回撤"更值得投入？

验证通过后，再引入 UrgencyWeight / QualityWeight 到排序公式。
**不基于假设引入权重，基于数据。**

## 9. 已知局限（交叉验证标记）

| 局限 | 说明 | 计划解决时间 |
|------|------|------------|
| 交易盈利一律视为"不可靠" | 专业做市商/量化团队的交易盈利可能是可持续的 | Phase 3 引入交易稳定度子指标 |
| 详情页展示可能泄露过多原始数据 | 已修正为结论式展示 + 折叠明细 | Phase 2a 按新规范实施 |
| UrgencyWeight/QualityWeight 未经数据验证 | 当前仅为理论推导值 | Phase 3 用真实数据回测后再启用 |

## 9. 一句话总结

```
现在我们知道资沉在涨还是在跌。
归因告诉我们：为什么在涨，这个涨靠不靠谱；为什么在跌，这个跌要不要救。
```
