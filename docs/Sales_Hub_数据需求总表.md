# Sales Hub 客户详情页 -- 数据需求说明

> 给研发看的数据对接文档。每条数据讲清楚：是什么、怎么算、为什么要、从哪来。
>
> 最后更新：2026-03-31

---

## 一、页面目标

一句话：**让 BD（销售）打开一个客户的详情页，30 秒内搞清楚"这个客户现在什么情况、要不要紧、我该做什么"**。

页面分为四个区域：

- **头部**：身份 + 资沉概览，一眼看出"客户是谁、钱多不多、最近在涨还是跌"
- **事件 Tab**：这个客户有没有需要我处理的事（出金、爆仓、风控）
- **资产 Tab**：钱分布在哪些业务线、最近资金进出情况、平台从他身上赚了多少
- **档案 Tab**：BD 自己的笔记和标签

## 三、各模块详细说明

### 头部 -- 身份区

#### 1. 客户 UID

**是什么**：客户在平台的唯一编号，比如 `12300840`。一个客户只有一个 UID，永远不会变。

**为什么需要**：BD 在和其他部门（风控、客服、研发）沟通时，用 UID 来定位客户。页面支持点击复制，方便直接粘贴到工单或聊天里。

**数据来源**：主站用户系统。

---

#### 2. 收藏标记

**是什么**：BD 手动把客户标记为"收藏"，类似于微信置顶。

**为什么需要**：BD 管理几百个客户，收藏功能让他们把重点客户挑出来，方便快速访问。

**数据来源**：本地存储（当前原型），正式版需要后端存储。

---

### 头部 -- 资沉指标

这是整个页面最核心的数据区，用 4 个数字讲清楚一件事：**客户的钱是在增加还是减少，变化有多大**。

---

#### 3. 本周资沉

**是什么**：客户当前在平台上的资沉（所有业务线资产总额）。这是页面上最大的数字，放在最显眼的位置。

**怎么算**：过去 7 天（T-6 到 T-0）每日资沉快照的算术平均值。用滚动 7 天均值而非某一天的快照，是为了平滑加密市场的日间波动——某天币价闪崩不会导致这个数字剧烈跳动。

**"本周"不是自然周**：不是"周一到周日"，而是从今天往回数 7 天。比如今天是周三，那就是上周四到本周三这 7 天的均值。每天都在滚动。

**举例**：何秀英过去 7 天的每日资沉分别是 155万、162万、171万、168万、175万、170万、182万，算术平均 = 169 万。

**为什么需要**：这是衡量客户价值最直接的指标。资沉越高，客户越重要。

**数据来源**：资产系统。需要每日记录一次客户的资沉。

---

#### 4. 8 周均值

**是什么**：本周之前连续 8 个"滚动 7 天均值"的算术平均。代表这个客户的"正常水平"。

**怎么算**：
1. 先算出过去每一周的"滚动 7 天均值"（跟本周资沉的算法一样，只是时间窗口往前推）
2. 取本周之前的 8 个这样的均值
3. 8 个数加起来除以 8
4. **不包含本周数据**——本周是被比较的对象，不能参与基线计算

**举例**：何秀英前 8 周的滚动 7 天均值分别是 357万、392万、338万、300万、265万、333万、272万、313万，算术平均 = 304 万。

**为什么需要**：单看"本周 169 万"你不知道多还是少。但如果你知道她过去 8 周平均 304 万，那 169 万就明显偏低了。这个均值就是用来做比较的"正常基线"。

**数据来源**：资产系统。基于每日资沉快照聚合计算。

---

#### 6. 资沉变化率

**是什么**：跟过去 8 周的正常水平比，资沉变了多少，用百分比表示。正数 = 比以前多了，负数 = 比以前少了。

**怎么算**：

```
资沉变化率 = (本周资沉 - 8 周均值) / 8 周均值 x 100%
```

**举例**：何秀英本周 169 万，8 周均值 304 万：

```
(169 - 304) / 304 = -44.5%
```

说明她的资沉比过去 8 周少了 44.5%，快少了一半，非常严重。

**为什么需要**：让 BD 一眼看出资沉变化幅度。-5% 可能不用太担心，-44.5% 就必须立即关注了。页面上会用红色/绿色箭头显示，红色向下 = 变少了，绿色向上 = 变多了。

**数据来源**：前端根据本周资沉和 8 周均值计算，不需要后端额外接口。

---

#### 7. 资沉变化额

**是什么**：跟过去 8 周的正常水平比，具体多了或少了多少万，用金额表示。

**怎么算**：

```
资沉变化额 = 本周资沉 - 8 周均值
```

**举例**：何秀英：169 万 - 304 万 = -135 万。她比过去 8 周的正常水平少了 135 万。

**为什么需要**：变化率告诉你"严不严重"，变化额告诉你"少了多少钱"。一个 -44.5% 可能是少了 1 万（小客户），也可能少了 135 万（大客户），BD 需要两个维度来判断优先级。

**数据来源**：前端计算。

---

#### 8. 备忘

**是什么**：一行文字输入框，BD 可以快速写一句话备注，比如"下周二回访"、"客户在出差"。

**为什么需要**：比起正式的备注（在档案 Tab），备忘更轻量，适合记录临时信息，打开页面直接就能看到。

**数据来源**：本地存储（当前原型），正式版需要后端存储。

---

### 事件 Tab

这个 Tab 回答一个问题：**这个客户有没有需要我马上处理的事？**

---

#### 9. 事件列表

**是什么**：该客户名下所有未处理、且未过期的事件集合。事件是系统根据客户行为自动触发的预警信号。

**目前只有 3 类事件**：

| 事件类型 | 什么时候触发 | 多久过期 | 为什么要关注 |
|---------|-------------|---------|------------|
| 超大额出金 | 客户单笔提现金额 >= 资沉的 30% | 48 小时 | 大量资金外流，可能要流失了 |
| 爆仓 | 合约或杠杆仓位被强制平仓 | 72 小时 | 客户亏了大钱，情绪可能不好，需要安抚 |
| 风控触发 | 风控系统标记账户异常（冻结/限制/KYC 问题） | 不过期 | 客户账户受限，需要协助解决 |

**举例**：何秀英有 2 条待处理事件：
- 超大额出金 -120 万（30 分钟前），状态"跟进中"
- 爆仓 -58 万（2 小时前，合约爆仓），状态"待处理"

**为什么需要**：事件是 BD 工作的核心驱动力。没有事件 = 客户暂时没事；有事件 = BD 需要行动。

**数据来源**：事件系统（根据充提系统、交易系统、风控系统的数据自动生成）。

---

#### 10. 事件明细

**是什么**：每条事件的详细信息。

**具体包含**：

| 字段 | 说明 | 举例 |
|------|------|------|
| 事件类型 | 超大额出金 / 爆仓 / 风控触发 | 超大额出金 |
| 子类型 | 更具体的分类（如风控的冻结/限制/KYC，爆仓的合约爆仓/杠杆爆仓） | 合约爆仓 |
| 关联金额 | 触发事件的金额（出金金额、爆仓损失等） | -120 万 |
| 事件状态 | 待处理 / 跟进中 / 已完成 / 已忽略 | 跟进中 |
| 发生时间 | 事件触发的相对时间 | 30 分钟前 |
| 过期时间 | 距离事件自动过期还有多久 | 48 小时 |

**为什么需要**：BD 根据这些信息判断轻重缓急。金额大 + 刚发生 = 优先处理。

**数据来源**：事件系统。

---

#### 11. 跟进记录

**是什么**：一条时间线，按时间倒序显示该客户所有的跟进历史，包括 BD 手动添加的记录和系统自动生成的记录。

**每条记录包含**：日期时间、执行人（BD 姓名或"系统"）、内容描述。

**举例**（何秀英的跟进记录）：
- 03-28 14:30 | BD 李明远 | 电话回访，客户表示近期有大额出金计划
- 03-25 10:00 | BD 李明远 | 发送续存权益说明，客户已读
- 03-20 16:45 | 系统 | 触发资沉流失预警

**为什么需要**：BD 需要知道之前和这个客户做过什么沟通，避免重复联系或遗漏。如果有别的 BD 接手，也能快速了解历史。

**数据来源**：CRM 系统（正式版）。当前原型根据客户状态生成模拟数据。

---

### 资产 Tab

这个 Tab 回答三个问题：**客户的钱分布在哪？钱是在流入还是流出？平台从他身上赚了多少？**

---

#### 12. 业务线余额

**是什么**：客户在平台 6 条业务线各自有多少余额。

**具体包含**：

| 业务线 | 是什么 | 举例（何秀英） |
|--------|--------|-------------|
| 双仓合约 | 客户做期货交易（做多/做空）的账户余额 | 101 万 |
| 超级杠杆 | 客户做杠杆现货交易的账户余额 | 204 万 |
| 理财 | 客户买理财产品（如 USDT 活期/定期）的金额 | 0 |
| 金融卡 | 客户的加密货币借记卡账户余额 | 0 |
| 云算力 | 客户购买云算力（挖矿）产品的投入金额 | 0 |
| Mini 合约 | 简化版合约交易，门槛更低，账户余额 | 0 |

**为什么需要**：让 BD 了解客户在平台上都在玩什么。何秀英只用合约和杠杆，说明她是"高风险偏好型"客户。如果她只用理财，说明她偏保守。这直接影响 BD 的沟通策略和产品推荐方向。

**数据来源**：资产系统，各业务线的余额快照。

---

#### 13. 业务线占比

**是什么**：各业务线余额占总余额的百分比。

**怎么算**：

```
某业务线占比 = 该业务线余额 / 6 条业务线余额之和 x 100%
```

**举例**：何秀英双仓合约 101 万，超级杠杆 204 万，总计 305 万：
- 双仓合约占比 = 101 / 305 = 33%
- 超级杠杆占比 = 204 / 305 = 67%

**为什么需要**：比起看绝对金额，占比能更直观地看出客户的资金集中在哪。如果 90% 都在一条线上，说明客户依赖度很高，这条线出问题对客户影响很大。

**数据来源**：前端根据各业务线余额计算。

---

#### 14. 周度资金流入

**是什么**：客户从外部往平台充值的金额，按周汇总，展示近 8 周。

**注意区分**：这不是"资沉变化"。资沉变化受很多因素影响（充值、提现、交易赚了亏了、币价涨了跌了），而"资金流入"只看一件事——**客户从外面转了多少真金白银进来**。

**怎么算**：统计该客户在某一周内所有充值交易的金额总和。

**为什么需要**：让 BD 了解客户的资金流动模式。如果流入在持续减少，可能客户在逐步撤离。如果突然出现一笔大额流入，可能客户有新的投资计划，是做续存引导的好时机。

**数据来源**：充提系统。

---

#### 15. 周度资金流出

**是什么**：客户从平台往外部提现的金额，按周汇总，展示近 8 周。

**怎么算**：统计该客户在某一周内所有提现交易的金额总和。

**为什么需要**：和流入配合看。如果流出远大于流入，客户在"搬家"；如果流入和流出差不多，客户在正常使用；如果流入远大于流出，客户在加仓。

**数据来源**：充提系统。

---

#### 16. 流入流出周均

**是什么**：前 8 周（不含本周）的平均每周流入、平均每周流出、平均每周净值（流入减流出）。

**怎么算**：

```
周均流入 = 前 8 周每周流入之和 / 8
周均流出 = 前 8 周每周流出之和 / 8
周均净值 = 周均流入 - 周均流出
```

**为什么需要**：和本周数据对比，看客户这周的资金行为是否异常。如果正常周均净流入 5 万，但本周净流出 50 万，说明出了问题。

**数据来源**：前端根据周度流入/流出数据计算。

---

#### 17. 综合净贡献

**是什么**：平台从这个客户身上赚了多少净利润。简单说就是"这个客户值多少钱"。

**怎么算**：

```
净利润 = 5 个收入模块 - 2 个成本模块
       = (双仓合约 + 超级杠杆 + Mini 合约 + 金融卡 + 云算力)
         - (理财利息 + 返佣)
```

**7 个模块具体解释**：

**收入模块（平台从客户交易中赚的钱）**：

| 模块 | 平台怎么赚钱 | 举例（何秀英） |
|------|------------|-------------|
| 双仓合约 | 用户做合约交易（如做多/做空 BTC）。对冲订单平台只赚手续费；非对冲订单平台当庄家，赚手续费 + 用户亏损 | +8,414 |
| 超级杠杆 | 用户做杠杆现货交易。逻辑和合约一样——对冲赚手续费，非对冲赚手续费 + 用户亏损 | +1,673 |
| Mini 合约 | 简化版合约，不区分对冲/非对冲，统一算。平台赚 = 手续费 + 资金费率 - 用户盈亏 | +1,652 |
| 金融卡 | 用户开卡和刷卡消费时平台收的手续费，没有对赌风险，纯稳定收入 | +819 |
| 云算力 | 用户购买算力和电费，平台收多少赚多少 | +875 |

**成本模块（平台要付出去的钱）**：

| 模块 | 平台怎么花钱 | 举例（何秀英） |
|------|------------|-------------|
| 理财利息 | 用户买了理财产品，平台要付利息给用户。用户存得越多、利率越高，平台成本越大 | -602 |
| 返佣 | 如果这个用户是代理，他推荐的下级用户交易时，平台要按比例给他返佣。不是代理则为 0 | -335 |

**何秀英的净贡献**：(8,414 + 1,673 + 1,652 + 819 + 875) - (602 + 335) = 12,496，约 1.2 万。

**为什么需要**：BD 需要知道哪些客户"值得花力气"。资沉高不代表贡献高——如果客户只买理财不做交易，反而是在消耗平台成本。净贡献才是真正衡量客户价值的指标。

**数据来源**：贡献收益查询系统，并行查询 7 个模块后汇总。

---

#### 18. 净贡献周环比

**是什么**：本周净贡献相对上周的变化率。

**怎么算**：

```
周环比 = (本周净贡献 - 上周净贡献) / |上周净贡献| x 100%
```

注意分母取绝对值，防止上周为负数时结果方向反了。

**举例**：何秀英的周环比是 +3.8%，说明本周贡献比上周稍微增加了一点。

**为什么需要**：净贡献本身是一个累计值，周环比告诉 BD 趋势——客户的价值在增长还是衰减。

**数据来源**：贡献收益查询系统。

---

#### 19. 净贡献子模块明细

**是什么**：第 17 条"综合净贡献"里 7 个模块各自的金额明细。页面上做成可展开/收起的形式，默认收起，点击展开看详情。

**具体包含**：双仓合约、超级杠杆、Mini 合约、金融卡、云算力这 5 个收入项，以及理财利息、返佣这 2 个成本项。每项显示金额和占比条形图。

**为什么需要**：当 BD 发现净贡献下降时，需要知道是哪个模块在拖后腿。比如合约收入暴跌可能因为客户减少了交易频率，返佣突然增加可能是代理结构变了。

**数据来源**：贡献收益查询系统。

---

#### 20. 近 7 天总盈亏

**是什么**：客户最近 7 天在交易上赚了多少或亏了多少。注意这是**客户的盈亏**（客户赚了还是亏了），不是**平台的收入**（两者经常方向相反）。

**怎么算**：把 4 条业务线的近 7 天盈亏加在一起：

```
总盈亏 = 双仓合约盈亏 + 超级杠杆盈亏 + Mini 合约盈亏 + 现货盈亏
```

**举例**：何秀英近 7 天总盈亏 = -6.2 万（合约 -1.2 万 + 杠杆 -4.9 万 + Mini -692 + 现货 0）。她这周亏了 6.2 万。

**为什么需要**：客户亏钱时情绪容易波动，BD 需要关注并主动安抚。如果客户连续亏损 + 大额出金，流失风险极高。

**数据来源**：交易系统。

---

#### 21. 业务线盈亏明细

**是什么**：4 条交易类业务线各自的近 7 天盈亏。

| 业务线 | 说明 | 举例（何秀英） |
|--------|------|-------------|
| 双仓合约 | 期货交易的盈亏 | -1.2 万 |
| 超级杠杆 | 杠杆现货交易的盈亏 | -4.9 万 |
| Mini 合约 | 小额合约的盈亏 | -692 |
| 现货 | 现货交易（非杠杆）的盈亏 | 0 |

**为什么需要**：让 BD 知道客户在哪条线上亏了。何秀英主要在杠杆上亏损，BD 可以针对性地提供风控教育或产品建议。

**数据来源**：交易系统。

#### 22. 8 周均盈亏

**是什么**：过去 8 周客户的平均每周盈亏。

**怎么算**：

```
8 周均盈亏 = 过去 8 周盈亏总和 / 8
```

**举例**：何秀英的 8 周均盈亏是 -2.9 万/周，说明她最近两个月平均每周亏将近 3 万。

**为什么需要**：和近 7 天盈亏对比。如果近 7 天亏了 6.2 万、但周均才亏 2.9 万，说明这周亏得比平时严重，BD 应该额外关注。

**数据来源**：交易系统。

## 四、关键概念

### 资沉 vs 资金流入/流出

这两个概念经常被混淆，但它们完全不同：

**资沉**（资产沉淀）= 客户在平台上总共有多少钱。它是一个**时间点的快照**，像银行存款余额。

**资金流入/流出** = 客户从外部充了多少钱进来、提了多少钱出去。它是一段**时间内的流水**。

**为什么不一样**？因为资沉变化不仅仅来自充值和提现。

```
资沉变化 = 充值（流入） - 提现（流出） + 交易盈亏 + 币价波动
```

举个例子：何秀英本周没充钱也没提钱（流入 = 流出 = 0），但如果她做合约亏了 50 万，她的资沉也会减少 50 万。所以看"流入/流出"只能知道客户搬了多少钱进出，看"资沉"才能知道客户实际还剩多少。两个要结合看。

---

### 事件 vs 跟进记录

**事件**是系统自动生成的预警信号，只有 3 种类型（超大额出金、爆仓、风控触发），有触发条件和过期时间。事件是"系统告诉 BD 发生了什么"。

**跟进记录**是 BD 和系统的所有历史操作记录，包括 BD 手动写的（"电话回访，客户表示..."）和系统自动记的（"触发资沉流失预警"）。跟进记录是"BD 做了什么"。

**为什么只有 3 类事件？** 目前系统只监控 3 种高风险行为（大额出金、爆仓、风控异常），因为这 3 种最紧急、BD 最需要立即行动。后续可能会扩展更多事件类型（如长期不登录、资沉持续下降等），但 V1 先聚焦最核心的场景。

---

### 对冲 vs 非对冲

这是理解"综合净贡献"中合约和杠杆模块收入的关键概念。

**对冲订单**：平台把客户的订单转发到外部交易所执行。不管客户赚还是亏，平台只收手续费。风险在外部交易所，平台旱涝保收。

**非对冲订单**：平台自己做客户的对手方。客户亏 = 平台赚，客户赚 = 平台亏。除了对赌盈亏，平台还收手续费。

> 举例：客户做多 BTC，亏了 100 USDT，手续费 2 USDT。
> - 对冲订单：平台赚 2 USDT（只有手续费）
> - 非对冲订单：平台赚 102 USDT（手续费 + 客户亏损）

所以非对冲订单高收益但高风险，客户赚钱时平台反而亏钱。
